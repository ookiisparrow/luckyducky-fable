# 验收走查

> 本 skill = 元模式的 **acceptance 环**（`docs/元模式.md` A5）：非技术所有者看现象验收，专盯机器够不到的（真支付/真机/库权限，对应靠人病根 #8）。

所有者**非技术**——验收靠**看现象**，不读代码。任何「做好了吗 / 怎么确认 / 上线前核」按此走，并用业务语言带着走。完整单见 `docs/验收手册.md`。

## 三个观察窗口（都不用懂代码）
- **小程序**（手机或开发者工具预览）：买家侧现象。
- **控制台**（admin）：运营侧现象。
- **真机**：涉微信能力（支付 / 扫码 / 视频）**必须真机**，模拟器不算数。

## 走哪张单（按场景）
- **某批做完**：`docs/验收手册.md` §三 对应验收单，逐条看现象。
- **上线 / 切换前**：**验收单 X**（16 集合权限档位，控制台逐表核，绝无「所有人可读写」）+ **Y**（支付黄金路径：真机下一笔小额真单 → 拉起收银台 → 付 → 订单转已付）+ **Z**（退款黄金路径：申请 → 控制台审批 → 原路退到账）。切换专项另见 `docs/切换runbook.md`。
- **加了功能**：走它的黄金路径（`docs/验收手册.md` §二）。
- **付费视频**：用**未购课**账号验——目录看不到视频地址、付费段取不到、换不出 URL。

## 判质量：四个代理信号（不懂代码也能判）
1. **验收单全过**——行为对 > 代码「看起来对」。
2. **测试只增不减**——每批日志写「test N」，数字变小要问为什么（当前 234）。
3. **文档与现实一致**——`项目现状` / `接口正册` 说有的，现象能验到。
4. **问题有账本**——出过的问题在 `docs/调试日志.md` 有现象 / 根因 / 教训。

## 验收方法论（根因#8：过了 ≠ 真能用）
- 用**真实样本**，不用玩具（图走真实大小、列表走真实条数、金额走真实分）。
- 凭证 / 签名走完整请求形状到最后一步（「拿到」≠「用通」）。
- 兜底覆盖主路径全部失败点、且被测试逼出。

## 何时打回 / 追问
- 验收单有一条不过 → **打回**，不接受「代码层面是对的」。
- 涉真机的只在模拟器跑过 → 追问。
- 说不清「用户会看到什么变化」→ 追问。
- 没更新文档就说完成 → 打回。

> 一句话：看现象不看代码、真机验真支付、四个代理信号判质量、一条不过就打回。
