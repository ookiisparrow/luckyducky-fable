# 文档体检

> 本 skill = 元模式的 **health 环**（`docs/元模式.md` A5）的文档版：周期体检活文档四维 + 「准确」这维强制对现实核。循环定义见元模式。

审文档 = **看现象，不看声称**（A7）。曾因信了 `现状与路线` 一段旧快照、没核 git，把早被提交解决的 `🔴` 当成"没做"、**报低了进度**——本 skill 立身之本就是不再犯：判「准确」**必先 `git log`/代码、再看 prose；文档与 commit 打架，以 commit 为准**。

## 四维尺子（每维：机器层已有什么 / 判断层 LLM 补什么）
1. **准确（doc = 现实·最重·先做）**
   - 计数/接口/集合 → 机器层兜：`npm run check` 绿即 `docs-budget`/`*-count-synced`/`interface-catalog-sync`/`requirement-trace` 已同步。
   - 状态/进度类（`现状与路线`/`待办与债`/`重构日志`）→ 判断层：对 `git log` 最近提交 + `重构日志` 顶部条 + 「部署记账」commit **交叉核**，揪「`🔴`/占位 已被后续 commit 解决却没回写」「描述滞后于已部署代码」。
   - 法：**派生真值（git/check 输出/`系统事实`）→ 比对声称 → 标矛盾**。
2. **简洁**：机器层 `docs-budget`（行数/份数）；判断层补 活文档条目 >8 行（铁律①）、重复、过期未删（铁律③）、同一状态三处漂移（铁律④）、**该退役未退役的过程账**（已完成批次/已关状态/边角料还赖在活文档·核心留、过程账退 `archive/` + 登记 `archive/README.md`·见 CLAUDE 退役制）。
3. **精准**：引用不悬空（「见 §X」「见 docs/X」能解析；`requirement-trace`/`guard-coverage` 已覆盖需求链与 `[机器守:id]`）；一词一义（`系统事实 §13`）；术语口径一致。
4. **清晰**：能一眼定位、标题层级合理；客观/主观分治不混（铁律⑥）；一需求一家（铁律⑦）。

## 流程
0. **先跑 `npm run check`**：机器层（docs-budget / count-synced / interface-catalog-sync / requirement-trace / guard-coverage）先过——绿了说明可机器判的简洁/计数/引用已守，审计**聚焦机器够不到的判断层**（别重造机器已做的）。
1. **取真值基线**：`git log -20 --oneline` + `重构日志` 顶部最新条 +（状态类文档）相关「部署记账」commit。
2. **逐活文档（15 份）按四维体检**，重点「准确」对 git 核；doc 与 commit 打架以 commit 为准。
3. **分级 findings**（CLAUDE §7 尺子）：P0 误导性失真（读了会误判·如进度坑）｜P1 stale 过期｜P2 冗余/越界/漂移｜P3 风格；每条带 **「文档 : 证据 : 真值来源」**。
4. **业务语言汇报**；够不到的（如"该删但需所有者确认"）老实标。

## 红线
- 判「准确」**必对机器真值（git / check 输出 / 系统事实）核**，不靠读 prose 下结论——本 skill 全部价值在此。
- doc 与 commit 打架，**以 commit 为准**。
- **只审计不大改**：findings → `refactor-batch` 逐条修（先守卫/先核 → 改到绿 → 反向自检 → 记账），本 skill 不夹带修改。
- 守 `CLAUDE.md` §4 七铁律 + 元模式 A2「客观/主观分治」；**不破坏单源**——别把内容复制到多处去"补全"，缺了就指针引单源。

## 产出
- 分级 findings 表（每条 文档:证据:真值来源 + 已修/建议修/够不到）。
- 真问题 → 按 `refactor-batch` 修；客观计数类漂移优先看是不是该补/改机器守卫（而非手改文档）。

> 一句话：对文档也看现象不看声称——先跑 check 过机器层，再逐活文档按四维体检、「准确」死磕对 git 核，分级标证据与真值源，真问题交 refactor-batch 修，别破单源。
