# 防过度工程（guardrail 环）

> 本 skill = 元模式的 **guardrail 环**（`docs/元模式.md` A5 之外的护栏 skill，与 health 环并列）：在「加抽象 / 重构」这条最容易长投机复杂度的路上设闸。判据全在 `CLAUDE.md` §7「抽象简约」+「业务预留分流」，本 skill 是它的执行流程。

防过度工程 = **先证明痛存在，再加解药**。最贵的不是重复，是「为不存在的需求建的机制」——它永远在、永远要维护、还挡路。`CLAUDE.md` §7 判例：S11 不引 xCounts、cs 域不立空册——别为不存在的需求建机制（呼应元模式「别摆设守卫」）。

## 先喂三个事实（不喂不动手）
1. **规模**：现在几处用？几个调用方？数据量级？——决定值不值得抽。
2. **寿命**：这段是 demo/占位/临时，还是长期主路径？——短命的别镀金。
3. **用途**：解决的是**当前真痛**，还是「万一以后 / 方便将来」？——后者即投机，删或延后。

## 流程
1. **先具体、先最笨**：先写「内联具体、能跑通主路径」的最笨版本（`CLAUDE.md` §7：内联具体优先）。跑通了再单独问「要不要抽」，不在第一版就抽。
2. **重构前先钉行为**：要改/抽一段已有代码，先用**真实脏样本**写 characterization test（差分测试）锁住「现在到底是什么行为」——抽象不能偷偷改行为（根因#8：构建过≠行为没变）。样本走真实尺寸/真实条数/真实分，不用玩具数据。
3. **抽象前过三问**（任一为否＝别抽）：
   - **当前痛点**：现在就有具体的痛（改一处要改 N 处 / bug 反复同源），还是只是「看着像能复用」？
   - **第三次重复**（Rule of Three）：是不是已经第三次重复了？前两次宁可先重复——「本质相同」还是「碰巧长得像」？碰巧像就别合。
   - **业务预留来源**：「以后/预留/将来」**先查** `需求清单`/`关键决策记录`/`现状与路线`/用户明确意图——对得上＝业务预留（保留并标清来源）；对不上且只为「方便将来」＝投机抽象（删或延后）；**不确定先问、不直接改**。
4. **新增接口/层/泛型/配置项旁注一句为什么**：理由含「万一以后 / 方便将来 / 更优雅」即删。这条本身是 `CLAUDE.md` §7 red flag。
5. **小批次改 + `npm run check`**：真要改走 `/refactor-batch`（先守卫后实现 → 反向自检 → 记账），一次一个意图，别借重构夹带抽象。

## 红线
- **「过度抽象」是判断题、机器证不出**（`CLAUDE.md` §7 明示·同根因#8/#10）——**别为这条硬上守卫**，那本身就是过度工程。本 skill 是靠人流程，不产出新守卫（除非抽象落地后有了可判真假的不变量，那才按 `/refactor-batch` 配守卫）。
- **不确定 → 问，不猜**：业务预留 vs 投机抽象分不清时，先查文档再问用户，别直接删也别直接抽。
- 删抽象同样要先钉行为（characterization test）——拆解药也可能咬人。

## 产出
- 一个结论：**抽 / 不抽（先重复）/ 延后 / 删**，每条带「规模·寿命·用途 + 三问结果 + 业务预留来源核查」。
- 若决定动代码 → 转 `/refactor-batch` 执行（带 characterization test）。

> 一句话：先喂规模/寿命/用途、先写最笨能跑通的版本、改前用真实脏样本钉行为，抽象前过「当前痛点/第三次重复/业务预留来源」三问，任一为否就先重复或先问——别为不存在的需求建机制。
