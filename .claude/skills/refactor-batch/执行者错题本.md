# 执行者错题本（refactor-batch 附册）

> 历任执行者/子代理**真实踩过**的坑。每个批次执行者开工前必读；收尾时把本批新踩的坑记回来（一条一行、带首踩批次）。同类坑满三条 → 按元模式收敛成机器守卫，并在此标「已机器化」。已机器化的条目仍保留——机器拦得住行为，拦不住浪费一轮返工。

- **E1 方法体正则守卫必须走 `methodBody()`+`stripComments()` 共享 helper**（scripts/check-structure.mjs）：裸写正则会咬到注释假绿——2026-07-08 播放页批/客服批连栽两次。已机器化：`guard-strip-single-source`。
- **E2 抄既有模式要抄全两半**：wxml 事件绑定（如 `catchtouchmove="noop"`）与 ts 方法定义是一对，抄一半=真机报错。首踩：2026-07-08 播放页批（noop 漏定义）。部分机器化：catchtouchmove⇒同页方法存在断言。
- **E3 改动文件 ⊆ 批次白名单**：误碰立即 `git checkout` 撤回并在返回值自报。首踩：2026-07-08 四拍板批（误 touch 现状与路线.md，自行撤回）。已机器化：batch-pipeline 边界断言。
- **E4 金额单位跨界必须显式换算**：mp 前端 cart/展示层是「元 number」，云端与订单链路是「分整数 Fen」——别把分塞进 cart.add、别把元写进库。首踩：无（四拍板批评审预防项），列此防患。
- **E5 规格矛盾/歧义 → 记 openQuestions 停手**：不擅自设计、不脑补规格。正例：2026-07-08 客服批（openCustomerService 参数两处描述冲突，执行者停问后按更安全字面执行）。
- **E6 账本双源要同批同步**：改 docs/待办与债.md 条目时，它指名的「明细源」（如 rewrite/mp/README.md §已知未搬面）必须同批改齐，否则两账互相矛盾。首踩：2026-07-08 客服批（README 漏同步）。
- **E7 债目勾销前重新 grep、不凭上批记忆**：账本写「现场核实确系修后版、无残留」前必须当场对目标文件跑一遍 grep/复读，不能沿用上一批的核实结论——上一批可能只改了部分行、遗漏点位下一批未必会自动补上。首踩：2026-07-09 批E round2（待办与债151 行债目声称 SKILL.md「无 next 时代旧文残留」，但第33行「样板房」实际仍在，上一次核实未真 grep 全文）。
- **E8 元守卫自指边界要抗非目标关键字绕过**：用「下一个 XX 关键字出现处」划 span 终点时，只认一种声明关键字（如 `function`）会被同语义但不同写法（`const`/`let`/`var` 箭头函数）的裸写插入绕过——插入点落在「前一个 span 终点之前」即被误判在界内放行。反向自检必须覆盖「非目标关键字写法插在两个 span 之间」这一类，不能只测「文件末尾追加」（那种远离 span、必被抓到，测不出边界漏洞）。首踩：2026-07-09 批E round2（`guard-strip-single-source` 的 `findSpan()` 只认 `function` 关键字，const 箭头函数裸写绕过；修复：边界关键字集扩到 `function|const|let|var`）。
- **E9 治理文档别手抄守卫计数**：guard-count-synced 扫 govDocs 四份（现状与路线/CLAUDE/元模式/根因账本·原自动化验证系统册已退役），文档里出现「N repoCheck/fileRule」错数即红——写「以 check-structure.mjs 为准」不写数字。首踩：2026-07-10 战役 II 批A。
- **E10 取真源断言必须对「剥注释后的函数体」匹配**：花括号配平时跳过注释但返回原始切片，注释文本会让 toMatch/indexOf 假命中——把保护代码注释掉断言仍绿（E1「注释假绿」病根在测试层的翻版）。helper 返回值必须剥注释。首踩：2026-07-10 战役 II 批I（admin g/h/commerce-c 与 agent desk 四份复制 helper 同修）。
- **E11 worktree 开工先 `npm install`**：新 worktree 无 node_modules，命令沿祖先目录借主仓的包**碰巧半绿**（npx vitest 能跑、npm run check 到 vue-tsc 才炸）——半绿比全红更误导。顺带：反向自检篡改**未跟踪新文件**后 `git checkout` 还原不了（不在 git 里），得手工还原。首踩：2026-07-11 flip-demo 批。
- **E12 只读评审员/验证者在共享工作区禁用 git checkout/restore/stash 类还原命令**：验证性篡改必须「自己备份→自己精确恢复（cp 回写）」，绝不可用 `git checkout -- <file>` 还原——它清掉的是**全部未提交改动**（含本批执行者成果），且并行评审员会在事故窗口读到瞬态工作区、产出「改动缺失」类假 P0（连锁污染两个评审）。首踩：2026-07-11 播放器重设计批D（评审员 checkout 清空 player.ts 后靠早前捕获的 diff `git apply` 侥幸救回；另一评审员在窗口内报假 P0）。部分机器化：batch-pipeline 评审员提示词已加禁令（2026-07-11）。
- **E13 执行者自己做反向自检也适用 E12**：篡改→验红→还原的「还原」必须用与篡改对称的精确编辑（Edit 加哪行就 Edit 删哪行），不可 `git checkout -- <file>`——文件里还躺着本批未提交改动，checkout 连它们一起冲掉。首踩：2026-07-12 投屏取缔批（验 rw-mp-no-casting 咬红后 checkout player.ts，冲掉本批四处编辑，靠上下文重放救回）。
- **E14 反向自检的篡改必须先确认「真的改到了」再信红绿**：`perl -0pi -e "s/…/…/"`（含转义括号/引号）就地替换可能静默没命中→文件没变→守卫仍绿，会被误读成「篡改后还绿＝守卫是摆设」的假结论（其实是没篡改成功）。规矩：篡改后先 `grep -c` 确认目标模式已消失/变化，再跑守卫看红。首踩：2026-07-16 加载页批（验 rw-mp-splash-auto-dismiss，perl 替换 triggerEvent('done') 没命中给假绿，改 `grep -v` 删行并 `grep -c` 确认为 0 后才真红）。
- **E15 编号型引用（决策§N/病根#N/R-N）落笔前先查目标文件现值**：凭记忆/上文口头编号直接写进代码注释与文档会撞已占用的号——2026-07-18 VOD 批把「决策§29」写进 20+ 文件，实际关键决策已到 §30、§29 被 SEO 拍板占用，收尾全批改号返工；且批量改号的语境正则漏了行内关键词大小写（isVodFileId 的 Vod 不匹配 /VOD|vod/）致 3 处残留、还差点误伤既有 §29 合法引用。规矩：落笔前 `tail`/grep 目标账本拿现值；跨文件大量引用时编号细节写一处、他处短指引用。首踩：2026-07-18 VOD 转码管线批。
- **E16 改「共用函数」的行为前必须 grep 全调用面——注释写了隔离 ≠ 代码实现了隔离**：2026-07-18 VOD 批2 把 admin `uploadVideo` 改成「VOD 配了就走 VOD」，注释白纸黑字写着「帮助视频线恒走老路不迁」，但该函数被课程线与帮助线**共用**、代码没有任何分支实现这句话——配置 VOD 后帮助视频会被一并传进 VOD，而其播放侧只认云存储 fileID → 求助面板视频静默全哑（无告警无兜底）。`npm run check` 全绿、2021 例测试全过：跨线影响静态测不出（根因#8），是写详细操作手册时人工复盘才逮到。规矩：① 改共享函数先 `grep -rn '<函数名>'` 列全调用点，逐个问「这条线要的是新行为还是旧行为」；② 新行为一律**显式 opt-in、默认旧行为**（fail-safe：漏改的调用方自动安全，而非自动踩雷）；③ 隔离承诺必须落成守卫，不能只落成注释。首踩：2026-07-18 VOD 转码管线批2。
- **E18 路径豁免类判定必须以「ROOT 相对路径」为准，绝不可拿绝对路径匹配 dot 目录**：本仓工作 worktree 本身就住在 `<主仓>/.claude/worktrees/` 下——`checkFile()` 拿绝对路径 test `/[\\/]\.claude[\\/]/` 会把 worktree 里**全部活文件**一并豁免（fileRules 在所有 worktree 静默全灭、只在 main/CI 上活着＝病根#16「空样本=绿」在守卫自己身上复发）。规矩：豁免/排除判定先 `relative(ROOT, file)` 再匹配；同类守卫落地后必须**在 worktree 里**做一次「真违例咬得红」的正向验证（不能只在 main 验）。首踩：2026-07-18 盲区体检批1 埋、批2 反向自检①当场逮住。
- **E19 反向自检的「还原」与「篡改」同权：还原后也必须 grep 验证落地**（E14 的镜像）：批1 反①还原 perl 因行尾残留空格 `$` 锚不匹配→静默没还原→下一步守卫红被误读一轮；规矩：还原后 `grep -c` 目标模式回到篡改前计数、再跑守卫看绿，perl 正则还原失败一次就改用精确 Edit/python replace（带 assert t!=s）。首踩：2026-07-18 盲区体检批1。
- **E17 反向自检的篡改必须覆盖「全部调用点」，否则会把守卫的正确沉默误读成漏洞**：删一个函数的**一处**调用去验守卫，若该函数在同文件还有第二处调用（如 `releaseExitGuard` 在 onHide + onUnload 各一处），守卫**理应仍绿**——这是守卫对的、用例错的，别急着改守卫。规矩：篡改前先 `grep -c '<函数名>('` 数清调用点，删就删光（`replace(old,new,-1)`）。姊妹坑同批逮到：**接线类断言必须先剥 import 语句再匹配**——只测「名字出现过」的话，删掉调用点、光留 `import { releaseExitGuard }` 照样绿（同 E8「断言太宽松被绕过」在 import 面的翻版）；断言形式用 `名(`（调用/方法定义）且在剥 import 后的正文上跑。首踩：2026-07-18 tab 退出提醒批（守卫 `rw-mp-tabbar-exit-guard` 断言③初版漏过，反向自检当场暴露）。
- **E20 非编译面语言（.wxs/.wxs 类受限运行时）写代码前先查它的语法子集——「桩在 Node 跑绿」不算证**：`seek.wxs` 用了 `Number(value) || 0`，微信编译器语法层直接拒（`Unexpected identifier '('` → 连带 `__route__ is not defined` 整页白），而 `npm run check` 全绿——因为 ① `.wxs` 不进 tsc/eslint 编译面；② 唯一覆盖它的对拍测试用 `new Function` 把函数体抽到 **Node** 里跑，Node 有 `Number`（桩比真运行时宽松＝根因#8/日志 P 同型）。WXS 是自带受限 builtin 的 ES5 子集：`Number/String/Boolean/Array` 是不可调用的常量对象、无 `Object`/`new`/箭头/`let-const`/模板串/`try-catch`/ES6 方法（正则要 `getRegExp()`、日期要 `getDate()`）。规矩：① 往非编译面语言写代码前先核它的子集清单，别凭 JS 直觉；② 这类面的「测试」若跑在宿主语言里，必须另配静态守卫咬语法（本批 `rw-mp-wxs-runtime-subset`）；③ 双源同式的另一半若在受限运行时，**两侧都写受限侧的合法式**，别让 TS 侧的合法写法成为下次抄过去就炸的陷阱。首踩：2026-07-20 批N（seek.wxs 真机编译失败）。
- **E21 篡改/还原的锚点必须在文件内唯一——同一文件存在字面相同的行时，`replace(old,new,1)` 会命中错的那处，且「还原」会把它插到错的位置（一处丢失＋一处重复，双向破坏）**：2026-07-20 批P 验 `measureSeekRect` 重试路径时，用整行 `if (!this._seekRect) this.measureSeekRect()` 做锚点删除——该行在 `onPlaying` 里本就存在一份字面完全相同的，`replace(...,1)` 删掉的是 onPlaying 那处（守卫因此**正确地保持绿**，被误读成「守卫是摆设」一轮），随后的「还原」又把它插到 onTimeUpdate 锚点前，结果 onPlaying 少一行、onTimeUpdate 多一行，靠 `git diff` 逐行核对才发现（还捎带一处缩进被改）。规矩：① 篡改前先 `grep -c '<整行>'` 数清同款行数，>1 就必须换带上下文的唯一锚点（如「上一行注释＋目标行」两行一起替换）；② 还原后不只看计数，**必须 `git diff` 核对本批改动逐字符与篡改前一致**（计数相同也可能位置错了）；③ 守卫没变红时先怀疑「篡改没落到目标处」，别先怀疑守卫（E14 的姊妹面：E14 是没改成，本条是改错了地方）。首踩：2026-07-20 批P。
- **E23 共享工作区的子代理提示词必须显式禁用 git 还原类命令（checkout/restore/stash 全家），不能默认它懂 E12**：2026-07-24 变异分诊批二，contentsec 分诊代理排查瞬态失败时自作主张 `git stash -u` 做「干净基线对照」，把并行在途的源码修复+兄弟代理测试产物整树暂存——虽当场 `stash pop` 零损失，但制造了两个次生事故：① 主脑取证 grep 恰撞 stash 窗口，读到「修复消失」假象、烧一轮排查；② 另一代理全量跑撞窗口报 5 个瞬态红、又各自烧一轮解释。规矩：① 给子代理的提示词凡涉共享工作区就带一句「禁 git checkout/restore/stash，对照基线用 scratchpad 副本」；② 主脑看到「文件被还原」怪象先怀疑并行方的瞬态窗口（stash/checkout 在途），用 `git log`+`git status`+目标 grep 三点取证再下结论，别在单一快照上定案（E14「先确认篡改真落地」的镜像：先确认丢失真发生）。首踩：2026-07-24 变异分诊批二。
- **E22 平台回调的「载荷形状」必须实测，不能按同类 API 想当然——形状错了不会报错，只会静默取到 undefined 再落进兜底值**：`ownerInstance.callMethod(name, args)`（WXS 回桥）把 args **直接**作为方法第一个实参传入，**没有 `.detail`**；而同页面其它方法都是 wxml 事件绑定（有 `.detail`），照着写就错。后果不是崩溃而是**沉默降级**：`e.detail` 恒 undefined → `Number(undefined)||0` 恒 0 → 松手 `seek(0)` 把视频拨回开头、拖动文案恒 0:00、逐秒判重恒「同一秒」故一次不震——三个看起来毫不相干的现象同一个因，且 `npm run check` 全绿、2262 例测试全过（静态与单测都测不到跨运行时的载荷形状·根因#8）。规矩：① 新接一种平台回调时，**第一件事是把整个入参打到控制台看真实形状**，别照抄同页面另一种回调；② 兜底默认值（`|| 0`）会把「读错字段」伪装成「值就是 0」，凡是有兜底的取值点，形状错误必然静默——这类点要么加断言要么先实测；③ 取值走单源 helper（本批 `wxsArgs()` 兼容两种形状）+ 守卫禁裸读，别让形状知识散在各处。**元教训**：我在拿到真值日志前，已按「现象推断」连发两批修复（批O 落位兜底、批P 量轨重试），实测日志把两批的假设都推翻了——**能插一行 log 就别推断三轮**；批O 的推测性代码已当批退役。首踩：2026-07-20 批Q。
