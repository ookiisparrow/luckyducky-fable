# 执行者错题本（refactor-batch 附册）

> 历任执行者/子代理**真实踩过**的坑。每个批次执行者开工前必读；收尾时把本批新踩的坑记回来（一条一行、带首踩批次）。同类坑满三条 → 按元模式收敛成机器守卫，并在此标「已机器化」。已机器化的条目仍保留——机器拦得住行为，拦不住浪费一轮返工。

- **E1 方法体正则守卫必须走 `methodBody()`+`stripComments()` 共享 helper**（scripts/check-structure.mjs）：裸写正则会咬到注释假绿——2026-07-08 播放页批/客服批连栽两次。已机器化：`guard-strip-single-source`。
- **E2 抄既有模式要抄全两半**：wxml 事件绑定（如 `catchtouchmove="noop"`）与 ts 方法定义是一对，抄一半=真机报错。首踩：2026-07-08 播放页批（noop 漏定义）。部分机器化：catchtouchmove⇒同页方法存在断言。
- **E3 改动文件 ⊆ 批次白名单**：误碰立即 `git checkout` 撤回并在返回值自报。首踩：2026-07-08 四拍板批（误 touch 现状与路线.md，自行撤回）。已机器化：batch-pipeline 边界断言。
- **E4 金额单位跨界必须显式换算**：mp 前端 cart/展示层是「元 number」，云端与订单链路是「分整数 Fen」——别把分塞进 cart.add、别把元写进库。首踩：无（四拍板批评审预防项），列此防患。
- **E5 规格矛盾/歧义 → 记 openQuestions 停手**：不擅自设计、不脑补规格。正例：2026-07-08 客服批（openCustomerService 参数两处描述冲突，执行者停问后按更安全字面执行）。
- **E6 账本双源要同批同步**：改 docs/待办与债.md 条目时，它指名的「明细源」（如 rewrite/mp/README.md §已知未搬面）必须同批改齐，否则两账互相矛盾。首踩：2026-07-08 客服批（README 漏同步）。
- **E7 债目勾销前重新 grep、不凭上批记忆**：账本写「现场核实确系修后版、无残留」前必须当场对目标文件跑一遍 grep/复读，不能沿用上一批的核实结论——上一批可能只改了部分行、遗漏点位下一批未必会自动补上。首踩：2026-07-09 批E round2（待办与债151 行债目声称 SKILL.md「无 next 时代旧文残留」，但第33行「样板房」实际仍在，上一次核实未真 grep 全文）。
- **E8 元守卫自指边界要抗非目标关键字绕过**：用「下一个 XX 关键字出现处」划 span 终点时，只认一种声明关键字（如 `function`）会被同语义但不同写法（`const`/`let`/`var` 箭头函数）的裸写插入绕过——插入点落在「前一个 span 终点之前」即被误判在界内放行。反向自检必须覆盖「非目标关键字写法插在两个 span 之间」这一类，不能只测「文件末尾追加」（那种远离 span、必被抓到，测不出边界漏洞）。首踩：2026-07-09 批E round2（`guard-strip-single-source` 的 `findSpan()` 只认 `function` 关键字，const 箭头函数裸写绕过；修复：边界关键字集扩到 `function|const|let|var`）。
- **E9 新增守卫 id 会连带触发 guard-count-synced——白名单必须带上 docs/自动化验证系统.md**：repoChecks 数组长度变化要求同步该文档计数，文档不在白名单守卫就落不了地（战役 II 批A 原型守卫因此撤回、批E 带上文档才正式落地）。首踩：2026-07-10 战役 II 批A。
- **E10 取真源断言必须对「剥注释后的函数体」匹配**：花括号配平时跳过注释但返回原始切片，注释文本会让 toMatch/indexOf 假命中——把保护代码注释掉断言仍绿（E1「注释假绿」病根在测试层的翻版）。helper 返回值必须剥注释。首踩：2026-07-10 战役 II 批I（admin g/h/commerce-c 与 agent desk 四份复制 helper 同修）。
- **E11 worktree 开工先 `npm install`**：新 worktree 无 node_modules，命令沿祖先目录借主仓的包**碰巧半绿**（npx vitest 能跑、npm run check 到 vue-tsc 才炸）——半绿比全红更误导。顺带：反向自检篡改**未跟踪新文件**后 `git checkout` 还原不了（不在 git 里），得手工还原。首踩：2026-07-11 flip-demo 批。
- **E12 只读评审员/验证者在共享工作区禁用 git checkout/restore/stash 类还原命令**：验证性篡改必须「自己备份→自己精确恢复（cp 回写）」，绝不可用 `git checkout -- <file>` 还原——它清掉的是**全部未提交改动**（含本批执行者成果），且并行评审员会在事故窗口读到瞬态工作区、产出「改动缺失」类假 P0（连锁污染两个评审）。首踩：2026-07-11 播放器重设计批D（评审员 checkout 清空 player.ts 后靠早前捕获的 diff `git apply` 侥幸救回；另一评审员在窗口内报假 P0）。部分机器化：batch-pipeline 评审员提示词已加禁令（2026-07-11）。
- **E13 执行者自己做反向自检也适用 E12**：篡改→验红→还原的「还原」必须用与篡改对称的精确编辑（Edit 加哪行就 Edit 删哪行），不可 `git checkout -- <file>`——文件里还躺着本批未提交改动，checkout 连它们一起冲掉。首踩：2026-07-12 投屏取缔批（验 rw-mp-no-casting 咬红后 checkout player.ts，冲掉本批四处编辑，靠上下文重放救回）。
