# 容量体检（扛不扛得住量）

> 元模式 health 环的**规模专版**（`docs/元模式.md`）：`/deep-audit` 管安全/钱/根因守卫、`小程序真机走查（rewrite/mp/README.md）` 管真机/多端/UX；本 skill 管**「构建过 + 单人能用，但量上来扛不扛得住」**那层。逼近上线/灰度/扩量、或用户问「能扛多少」时走它。**关键认知（根因#8 同理）：单人真机能用 ≠ 千人同时能用——没压测的容量结论都是「推断」不是「证明」。**

## 0. 先把「量」翻译成机器语言（别被大数字唬住）
- 日均用户 → 峰值：电商流量是尖的，峰值小时 ≈ 日量的 10–20%。
- 每用户一程触发多少次云调用（逛+登录+下单+看课 ≈ 20 次）。
- 算出**峰值 QPS + 爆发 QPS**。例：9,999/天 ≈ 峰值每秒几十、瞬时爆发约 100——对 serverless 是小场面，「马力」基本不是瓶颈，**瓶颈在下面几条、与马力无关**。

## 1. 扫规模杀手（读热路径、逐个看真实现，不凭记忆）
逐个读最热的读路径 + 写量大的函数，找这几类（本仓真踩过的标注）：
- **内存聚合封顶**：看板类 `getDashboard` 各表 `limit(1000)` 内存聚合——4 位数当天就溢出、GMV 少算（`待办 §上量就绪`）。
- **无界全表 `.get()`**：`getProducts`/`getCourses` 无 limit（现 catalog 小、安全；长大成千要分页）。
- **N+1 / 扇出**：循环里逐条查库。
- **固定 limit 静默挤出**：列表无 cursor/limit 契约（`paging-contract` 守，但核口径）。
- **热文档争用**：所有人抢写同一条计数器/文档。
- **缺索引**：非 `_openid` 的 `where`（如 `closeExpiredOrders` 的 `{status,createdAt}`）在增长表上退化成扫表（`_openid` 微信云开发自动建、无忧）。
- **无界增长**：只写不读的流水（`events`——现已 P2 限速 + `cleanupEvents` TTL 收口，根因#13/债#9）。

## 2. 基建维度（代码看不到，必须去控制台/部署侧确认）
- **数据库套餐档位**：读写全经云函数→库（16 集合锁后库是总闸），峰值 QPS 上限取决于套餐——**代码里查不到，去控制台确认能扛峰值**。头号未知。
- **视频带宽**：视频是不是还裸云存储直吐（R21「VOD/CDN」搁置）？观看上量带宽/流量费是压倒性大头、可能先顶爆——视频产品头号成本。
- **冷启动**：钱链热函数（createOrder/pay/payCallback）冷启动延迟影响支付体验，必要时预置并发。

## 3. 分清「正确」与「容量」（关键，别混）
- **并发正确性**（代码能证、本仓已稳，别重报）：确定性 id 防重扣 / 条件抢占防重激活 / fail-closed 闸 / 状态机 / 限速防刷。一千人同时涌进**数据不会错乱**。
- **容量**（马力 / QPS / 带宽，**没压测就是推断**）：上面 §1/§2 那些。
- 两者分开报——「正确性稳」不等于「扛得住量」。

## 4. 分级 + 业务语言汇报（所有者非技术）
- 按风险排，每条标**代码能改 / 归控制台·部署 / 归运营**——别把控制台的事说成代码能搞定。
- **诚实红线**：没跑过压测就说「推断、非证明」（根因#8）；给「上量前必清」清单，落 `待办 §上量就绪`。
- 真要赌大量：正经顺序 = ① 控制台查套餐/视频 ② 跑一次峰值并发压测把推断变证明 ③ 补 §1/§2 的代码项。

## 红线
- **「构建过」「单人能用」≠「扛得住量」**——没压测的容量结论必须标「推断」（根因#8）。
- 基建维度（DB 套餐 / 视频带宽 / 索引）代码里看不到，**别假装代码能下结论**——明确标「去控制台/部署侧确认」。
- 别把「并发正确性」当「容量」糊弄——本仓正确性已稳，问题在马力 / 带宽 / 套餐。

> 一句话：把量翻成峰值 QPS、读热路径扫规模杀手、基建维度标「去控制台」、分清正确性 vs 容量、没压测就说推断、分级落 `待办 §上量就绪`。
