# 数据库 / 云存储 权限期望表（根因账本 #9，固化验收单 X）

> 「敏感写一律走云函数」的前提是**客户端不能直接写库**。这道闸在云开发控制台设置、**代码看不到**（CLI 也读不到安全规则）——故是 git 外资产，须人工核对。本表是期望档位的正册副本（生产仓验收单 X 固化），上线前/迁移后逐项核一次。
>

> ⚠️ **更正（2026-06-14 验收单 X 实测·计数随 COLLECTIONS 同步=22）：上表 22 个集合一律「所有用户不可读写」(仅管理端·`inventory` 库存#1/`auditLog` 审计#4 同档)**——小程序**零直接读库**（全走云函数 getProducts/getCourses/getContent），故 products/courses/content **不需要公开读**；原标「所有人可读」是错的，且让 `courses` 里的 `videoFileId` 被客户端直查泄露（付费视频漏口）。商品图/头像显示走的是**存储**的 `cloud://` 公开读，与集合读权限无关。

## 云存储

- `products/` `avatars/`（商品图/头像，公开展示）：**仅创建者可写、所有人可读**。
- `videos/`（**付费课视频**，审计 P1）：客户端经云函数 `getPlaybackUrl` 服务端鉴权后拿短时效 URL，不直接 `getTempFileURL`。
  - **保护靠「fileID 不可得」而非存储路径私有**（2026-06-14 验收单 X 实测更正：云开发存储规则只认创建者、不认路径，`resource.path` 写法报 InvalidParameter，做不到「products 公开读 + videos 私有读」）。
  - 实际保护链（四条合起来 = 客户端永远拿不到 videoFileId）：① `getCourses` 剥 videoFileId（只返 `hasVideo` 布尔）② `getPlaybackUrl` 只发短时效 URL、不发 fileID ③ `getMyCourses` 不带 ④ **`courses` 集合 仅管理端**（客户端直查也拿不到）。⇒ videos/ 存储即便公开读也无人能取。
  - 核对（验收单 X 扩展，已实测）：客户端 `db.collection('courses').get()` 被拒（-502003）= videoFileId 不可得；未购课账号付费段 `getPlaybackUrl` 返 `NOT_ENTITLED`。

## 备注：debt #14 与确定性 _id

- 写侧确定性 _id（submitReview `orderId__productId`、afterSales `orderId__productId`、qrcodes/code、seed `_id`=业务 id）让「仅创建者可读写」即够（撞 _id 抛错=天然唯一）。
- users/progress 已接确定性 _id（login/updateProfile/trackEvent 经 `ensureDoc`，**债#14 已根治**：并发首写撞号即幂等、不重复建档）；仍先 `where` 命中老随机 _id 档作惰性迁移，权限模型不变（仍仅创建者/仅后台）。
