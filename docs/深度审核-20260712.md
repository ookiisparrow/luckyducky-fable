## 收尾判断（结论先行）

- **最该先处置的两条 P1**：① 退款/发货不联动是**真资损路径**（钱退了货还能发），逻辑修单点、收益高；② 部署闸绕过是**唯一 tcb 机器防线在常见命令形态下失守**，正则修单点。两者都不需新架构。
- **P2 里成群的是一类病**：mp「网络失败伪装成空态/不存在」5 处 + admin/退款审核假绿判据——都是「失败落 fallback 空值继续走」，仓内已有正确范式（detail.ts / aftersales.ts loadFailed 分治），逐处对齐即可，风险低。
- **守卫元审揭示系统性缺口**：3 条守卫（频控/钱链漂移/需求追溯）仍盯已清退的冻结旧线 `packages/`，对唯一在迭代的 `rewrite/` 是**假绿**——不是当前有 bug，是「防线扫错了线」，补 `rw-` 镜像即闭环（先例：`rw-order-transitions-declared` 2026-07-05 已做过同款补丁）。
- **够不到（须人工/真机核）**：库集合权限、云存储私有读、真支付退款、真机投屏 state 字符串枚举——代码证不了，按验收单 X/Y/Z 上线前控制台+真机核。
- 逐条修按 `/refactor-batch`（先立守卫红→改到绿→反向自检→记账）。建议批次：批1 两条 P1；批2 mp 失败态家族 + Refunds 假绿；批3 守卫补 rw- 镜像；P3 随手批或并入相关批。
