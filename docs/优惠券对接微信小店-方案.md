# 优惠券系统对接微信生态原生能力 — 方案文档

> 状态：**决策材料，未落地**（用户已拍板做成真系统，本文档只出方案，不写实现代码）
> 提问口径：用户要求"对接微信小店"，本研究发现**更贴合当前架构的不是微信小店本身、而是微信支付的"商家券"**——原因见下文研究结论。请用户对着"推荐方案"一节做选择。

## 一、现状：当前占位券链路

**链路**：结算页固定显示"-¥20.00"抵扣 → 云端 `createOrder` 用同一个硬编码常量现算 → 若抵到货款为 0 走 fail-closed 拒单。

**代码位置**（全部是占位，注释已自述"真券系统落地时重写"）：
- `rewrite/shared/src/checkout.ts:16` — `export const COUPON = 20`（元，无条件抵扣，非"券"，是全场固定立减）
- `rewrite/mp/lib/checkoutConst.ts:14` — 同值副本（mp 包进不了 `@ldrw/shared`，靠 `rw-mp-checkout-consts-synced` 守卫焊死逐值一致）
- `rewrite/shared/src/errors.ts:30` — `COUPON_EXCEEDS_GOODS`（深审 2026-07-05 新增的 fail-closed 闸：券抵到 0 元即拒单，防"上架低价 SKU 即白送"）
- `rewrite/cloud/src/functions/app/actions/orders.ts:147-150` — 唯一现算点：`amountFen = goodsFen + shipFen - couponFen`，云端现算不信前端
- `rewrite/shared/tests/parity.test.ts:28-31` — parity 测试焊死 `COUPON`/`SHIP`/`CHECKOUT_ADDONS` 与旧线（`packages/`）逐值一致

**当前券的本质**：不是"用户领取/核销"的券实体，是结算页固定展示的一条抵扣行，人人一样、无门槛、无有效期、不可配置（改值要改代码 + 过守卫 + 过 parity 测试）。

## 二、研究结论：微信生态两套不同的"优惠券"

用户提到的"微信小店""小程序连接小店"官方文档里，**优惠券实际分属两套互不相通的体系**，接入方式和适配度差异很大：

### 体系 A：微信小店自己的优惠券（`<store-coupon>` 组件 / 小店营销 API）

- 属于**微信小店**账号体系（`developers.weixin.qq.com/doc/store/shop/`），创建/核销走小店自己的营销管理 API，权限集 132，需小店商家 `access_token`。
- **前置强制**：小程序要用小店的商品卡/优惠券组件，必须先完成"小程序连接小店"——这是**小店侧发起邀请、小程序侧接受**的人工关联动作（微信公众平台/小店后台操作，代码做不了）。
- **关键限制（决定不可行）**：`<store-coupon>` 组件展示的券，用户点击后**跳转到微信小店自己的下单结算页**核销，不是在当前小程序自己的结算页里抵扣。也就是说：要用这套券，商品必须先搬进小店后台管理、订单必须走小店自己的交易系统。
- 与本仓 T3（云为唯一真相，商品/订单单源在自建 CloudBase）、以及 2026-06-30 已拍板的决策（`docs/关键决策记录.md` §19："**微信小店＝分发渠道、非实物引擎**"——材料包+激活码+课程捆绑、激活链绑 openid，装不进小店，自建订单/课程/激活体系不迁）**直接冲突**。核心商品线（钩织材料包+课程+激活码）本来就不打算搬进小店卖，这套券自然用不上。

### 体系 B：微信支付"商家券"（独立于微信小店，绑定商户号）

- 属于**微信支付**（`pay.weixin.qq.com`），是商户自定义规则的营销工具（满减券/换购券/折扣券三种），**不要求商品或订单挂在微信小店下**，只要求商户已有微信支付商户号——这一点本项目**已具备**（`console-assets/01-支付退款工作流.md` 记录的连接器 `wxpay_33nb7su`、子商户号）。
- 完整流程：商户 API **创建券批次** → 发放（H5/小程序发券插件/API 推送到用户微信卡包）→ 用户领取后卡包可见 → **商户在自己的下单流程里查询该用户卡包中满足条件的券**（`根据过滤条件查询用户的券` API）→ **商户自己判断门槛、自己算折扣金额**（官方原文："用户是否满足优惠使用门槛条件，以及对应优惠金额或商品均由商家系统自行判断"）→ 商户按算好的折后价发起支付 → **支付成功后调用"核销用户的券"接口核销**（传 openid + 批次号 + 券码）→ 退款场景可调"申请退券"把券状态从 `USED` 退回 `EFFECTIVE`（对称覆盖售后退款场景）。
- **这条链路与本仓现有架构高度同构**：现在 `createOrder` 已经是"云端查库自己算价、fail-closed 不信前端"的模式；商家券只是把"查一个写死常量"换成"查用户卡包里的券、按官方返回的门槛/折扣规则算"，核销动作天然对应订单状态机里"支付成功"这个转移点（`transition()`），退券天然对应现有退款流程——不需要改变交易引擎在哪、订单在哪，**只是结算页多算一步、支付回调多调一个接口**。

### 关键差异小结

| | 微信小店优惠券（体系 A） | 微信支付商家券（体系 B） |
|---|---|---|
| 归属 | 微信小店账号 | 微信支付商户号 |
| 前置 | 必须"小程序连接小店"（人工邀请/接受） | 商户平台申请开通"商家券"产品权限（人工，约 1 个工作日审核） |
| 核销地点 | 跳转小店自己的结算页 | 商户自己的下单/支付流程里核销 |
| 是否要求商品/订单搬进小店 | **是**（与决策§19"小店仅渠道"冲突） | 否 |
| 与本仓结算页兼容 | **不兼容** | **兼容** |

## 三、可行性判断

**用户要求的"结算页抵扣"场景，微信小店原生优惠券（体系 A）不可行**——它的核销天然跳转到小店自己的下单页，会把交易入口从当前小程序自建结算页挪走，与决策§19"小店仅渠道、不迁引擎"正面冲突，也绕不开"先把商品搬进小店"这个前置，不是技术细节问题、是路线问题。

**微信支付商家券（体系 B）可行，是推荐路线**，但有三点需要如实标注的限制：

1. **灰度状态未定**：本次研究检索到的官方"商家券接入前准备"页面文案含"当前处于灰度测试阶段、暂未开放使用"字样（检索时点 2026-07-12）。这可能是文档未及时更新的残留信息，也可能是真实当前状态——**这一点代码/研究都验不出，必须用户或后续人工去微信支付商户平台核实当前是否已全量开放商家券产品权限申请入口**，不确认此项之前不建议排期开发。
2. **不是"传参自动扣"，是"自己查自己算"**：商家券不会像"代金券/立减金"那样自动接入统一下单接口自动核算，商户必须自己调"查询用户券"接口、自己按官方返回的规则字段判断门槛/折扣、自己算出折后价——这正好符合本仓"价格云端现算、不信前端"的既有不变量（T·价格闸），改造量可控，但也意味着这不是"接个开关就好"的轻量对接。
3. **需要新的出站签名调用点，但底子已经在**：`rewrite/cloud/src/kit/wxpay.ts` 已经实现了商户 APIv3 直连的核心原语——`wxpaySign()`（RSA-SHA256 签名）、`normalizePem()`（还原控制台粘贴易塌陷的私钥换行）、`getSecureConfigFields()`（读 `secureConfig.wxpay.mchPrivateKey`/`mchSerial`，决策 2026-07-12 已支持 admin 页面填写生效，不需要再走"改环境变量重新部署"）。这套凭证管道是给"S16 外部对账拉交易账单"用的，但**商家券所需的签名机制完全相同**（同一商户私钥、同一签名算法），只是要新增 POST 请求体签名（现有 `wxpaySign` 已支持传 `body` 参数，`fetchTradeBill` 只是恰好只用了 GET，扩一个 POST 调用点不难）。

**结论**：走商家券路线，凭证/签名地基已经有七成，缺的是（a）用户去微信支付商户平台确认并申请商家券产品权限，（b）新增几个 API 调用点（创建/查询/核销/退券），（c）结算页从"固定展示"改成"查券/选券/展示动态门槛"。不是从零开始。

## 四、推荐方案

### 主推：微信支付商家券（体系 B），若灰度未开放则走退而求其次方案过渡

**分层设计**（尽量复用现有架构语汇，不新建平行体系）：

1. **凭证与接缝**：复用 `secureConfig.wxpay.mchPrivateKey`/`mchSerial`（已有，无需新增字段）；新增出站调用点收口在 `kit/wxpay.ts` 同文件或紧邻的 `kit/wxpayCoupon.ts`（遵循本仓"平台接缝单点"原则，且现有 `wxpay-seam-single` 守卫的精神应该扩面覆盖新调用点，不要在业务码里散落直拼签名）。
2. **createOrder 阶段（`orders.ts`）**：把现在硬编码读 `COUPON` 常量的两行（147/195 行）换成：若前端传了 `couponCode`（用户在结算页选中的券），云端查该用户卡包里这张券的详情（门槛/折扣规则），按官方规则字段自己算 `couponFen`（不信前端传来的折扣数字），其余 fail-closed 逻辑（`COUPON_EXCEEDS_GOODS` 判断、分整数运算）原样保留、只是 `couponFen` 从常量变成动态值。**不核销**——只读、只算价，核销挪到下一步。
3. **支付成功回调（paynotify）阶段**：这是新增的一次性副作用点，天然对应现有 `transition()` 状态机"首次转移绑副作用"的模式——支付成功回调触发订单状态 `pending → paid` 的同一次转移里，调用"核销用户的券"接口核销该订单绑定的券（幂等：同一订单只应核销一次，靠 `transition()` 现有的原子性保证，不需要新的幂等机制）。
4. **退款阶段**：现有退款流程完成后追加"申请退券"调用，把已核销的券状态退回可用（对称覆盖售后场景，用户没白扔一张券）。fail-soft：退券失败不阻断退款本身，走 `observe.alert` 留痕告警（沿用病根14"失败必可观测"规则），不静默吞。
5. **错误码**：`COUPON_EXCEEDS_GOODS` 保留（含义从"占位固定券抵到 0"变成"动态券抵到 0"，行为不变）；可能需要新增 `COUPON_NOT_FOUND`/`COUPON_EXPIRED`/`COUPON_ALREADY_USED` 一类新码（选中的券在结算这一刻已失效/已用/查不到时的用户反馈），按 `errors.ts` 头注"新增只追加"登记，parity 测试同步登记新码名单。
6. **mp 端结算页 UI**（`rewrite/mp/pages/checkout/`）：
   - 从"固定显示一行优惠"改成"我的优惠券"入口——查用户卡包里适用本单商品/满足门槛的券列表，用户选一张（或不选）。
   - 选中后前端只做**展示**用（沿用现有"云端权威、前端展示口径对齐"惯例，`resolveOrderAmountFen` 已有"回包缺失才回退前端估算"的防御范式，可直接复用）——真正金额仍由 `createOrder` 回包权威值决定，前端展示与之对齐，不允许前端自己算折扣当真。
   - 无券可用/未领券时的空态（"暂无可用优惠券"），符合本仓"空数据有空态"的质量尺子。
   - 若要支持"发券"（比如首单券、活动券），还需要一个发放入口（H5/小程序发券插件、或客服/活动页触发 API 推送）——这属于运营侧新功能，若用户暂无发券场景（比如先只做"后台创建的通用券，用户扫码/点击链接自助领取"）可以先不做主动推送这一层，缩小首批范围。

### 退而求其次：若灰度未开放 / 商户平台暂不批准

若用户去核实后发现商家券产品权限当前申请不到（灰度限制、或商户资质/类目原因），建议两级降级，**不建议自建一套完整的优惠券中台**（过度工程，呼应本仓"防过度工程"纪律——没有真实多档规则需求前不要先建机制）：

- **降级 A（推荐先做这个）**：保留"结算页抵扣"这个用户体验，但把现在的硬编码 `COUPON = 20` 换成**可配置**——挪进现有 `config` 集合（或仿照 `secureConfig`/`configChecklist` 的 admin 页面填写模式），后台可改抵扣金额/门槛（比如"满 X 元减 Y 元"），仍是"全员统一一条规则"，不做"用户级券实体"（不做领取/多张/有效期）。改动量小，云函数侧只是把常量读取换成读一个 config 文档，`COUPON_EXCEEDS_GOODS` 闸门逻辑不用动。这条路线随时可以作为后续升级到商家券的过渡态，不浪费。
- **降级 B（用户明确要"券实体"体验但商家券批不下来才考虑）**：自建券系统（`coupons`/`userCoupons` 两张集合，管理端建券规则，用户端"我的优惠券"页，`createOrder` 里查询+核销走自己的库，不依赖微信任何官方 API）。这是最大的工作量，且是本仓从零发明一套现成体系已有官方实现的东西，**不推荐作为首选**，只有在商家券路线被证实走不通时才考虑。

## 五、需要用户做的人工配置清单（代码做不到，照 `console-assets/` 正册惯例列出）

> 以下步骤全部要在微信支付商户平台 / 微信公众平台人工操作，Claude Code 拿不到登录凭证也代劳不了。建议走通后把结果（产品权限开通状态、批次号规则、回调地址等）补进 `console-assets/` 新增一份 `03-商家券配置.md` 正册（沿用 01/02 的编号习惯）。

1. **核实商家券产品权限是否已开放申请**：登录微信支付商户平台（对应本项目商户号，与 `console-assets/01-支付退款工作流.md` 记录的连接器 `wxpay_33nb7su` 同一商户号体系），查营销工具 / 产品中心，确认"商家券"入口是否可申请（本文档研究时点显示可能仍在灰度，需现场核实为准）。
2. **申请开通商家券小程序发券插件权限**：若走"小程序内发券"场景，需在小程序管理后台申请该插件权限，官方文档标注约 1 个工作日审核。
3. **设置商家券事件通知回调地址**：微信支付商户平台配置回调 URL（用于接收发放/核销等异步通知），需要一个新的云函数 HTTP 触发器（对应现有 `paynotify`/`refundnotify` 转发节点模式，参照 `console-assets/forward-node.js` 现成写法新增一个转发节点即可，不是从零设计）。
4. **确认 APIv3 密钥可用于解密新回调**：现有 `wxpay.mchPrivateKey`/`mchSerial` 已在 `secureConfig` 里（admin 配置清单页可填），若商家券回调走同一套 APIv3 密钥体系应可直接复用；若官方要求单独的商家券密钥，需要在 admin 配置清单页新增一个字段（技术上是小改动，走 `saveSecureConfig` 单口写入即可）。
5. **决定券的初版规则**（业务决策，非技术）：满减门槛/折扣力度、每人限领张数、有效期、是否分商品券/店铺券——这些决定了创建批次 API 传什么参数，需要用户先拍板规则再排期实现。
6. **决定发放渠道**：后台批量发放给存量用户？活动页领取？首单自动发？客服触发？这决定 mp 端是否需要新增"领券"入口页面，还是只做"结算页选用已有券"。

## 六、云函数侧改造大致设计（复述第四节，供排期参考）

- 新增/扩展接缝文件：`kit/wxpay.ts` 扩展支持 POST 签名请求，或新增紧邻文件 `kit/wxpayCoupon.ts`（收口创建批次/查询用户券/核销/退券四个调用）。
- `orders.ts::createOrder`：`couponFen` 从常量改成动态查询+校验（前端传选中的 `couponCode`，云端查详情算折扣，不信前端传的折扣数字）。
- 支付成功回调（现有 paynotify 转发节点触达的云函数处理点）：追加核销调用，绑定订单状态 `pending→paid` 首次转移（复用 `transition()` 原子性防重复核销）。
- 退款流程：追加"申请退券"调用，fail-soft + `observe.alert` 留痕。
- `errors.ts`：视需要新增 `COUPON_NOT_FOUND`/`COUPON_EXPIRED`/`COUPON_ALREADY_USED` 等码，`parity.test.ts` 登记名单同步。
- 若要支持"我的优惠券"列表：新增一个查询用户可用券的 app 网关 action（只读，查微信侧、不落自己的库——避免和微信卡包状态产生双源漂移）。

## 七、mp 端 UI 改造大致设计

- 结算页（`rewrite/mp/pages/checkout/`）：把 `couponLabel` 固定展示改成"选择优惠券"交互（弹层/跳转选择页），选中态回显门槛/折扣文案；未选/无可用券时的空态文案。
- 若做"我的优惠券"独立入口：`我的` tab 下新增一个列表页，展示卡包里适用本商城的券（可用/已用/已过期分组），复用 `MediaSlot`/空态组件等现有模式，不新建一套展示体系。
- 所有金额仍由云端 `createOrder` 回包权威值驱动展示，前端不自行算折扣当真（沿用现有 `resolveOrderAmountFen` fail-soft 回退范式）。

## 八、预计工作量分解（按 `/refactor-batch` 颗粒度，每批可独立验收）

> 前提：批次 0 是硬前置——用户没走完人工配置清单第 1-2 条之前，后续批次都无法真机验证（只能写代码 + mock 测试），建议先做批次 0、拿到真实开通结果再排后续。

- **批次 0（用户侧·非代码）**：核实商家券产品权限开放状态 → 若开放，申请开通 + 设置回调地址 → 拍板券规则（门槛/有效期/发放渠道）。**这批不产生代码提交**，是本方案能否往下走的分岔点。
- **批次 1（云函数侧地基）**：`kit/wxpay.ts` 扩展 POST 签名支持 + 新增创建/查询/核销/退券四个调用函数（先写通、用 mock fetch 测试签名逻辑正确，不依赖真实开通）。加测试 + 视情况扩展/新增 `wxpay-seam-single` 类守卫覆盖新调用点。
- **批次 2（下单侧改造）**：`orders.ts::createOrder` 接入动态查券算价，替换硬编码 `COUPON` 常量读取；`errors.ts` 新增券相关错误码 + `parity.test.ts` 登记；`shared/checkout.ts`/`mp/lib/checkoutConst.ts` 的 `COUPON` 常量退役（或降级为"无券兜底值"，视批次 4 降级方案是否并行保留）。
- **批次 3（回调侧核销/退券联动）**：支付成功回调追加核销调用（绑定 `transition()` 首次转移）；退款流程追加退券调用（fail-soft + alert）；新增回调转发节点（参照 `forward-node.js` 模式）+ 补进 `console-assets/03-商家券配置.md` 正册。
- **批次 4（mp 端结算页 UI）**：结算页从固定展示改交互式选券，空态/回显文案，`/browser-check` 或真机验证（微信原生单源，走 `/frontend-check` 真机验证范式）。

每批遵循本仓纪律：根因优先 → 先加/扩守卫（红）→ 改到绿 → `npm run check` → 反向自检 → squash 提交 → 记 `重构日志.md`；涉及钱的批次（1-3）按 P0 尺子过审（金额分整数、fail-closed、幂等核销）。

## 九、风险与不确定性（如实标注，不假装已验证）

- **最大不确定性**：商家券产品权限当前是否已全量开放，本次研究检索到的官方文档片段显示可能仍在灰度——**这一条必须用户去商户平台现场核实，本文档给不出确定答案**（根因#8"拿到≠用通"，此处是"文档说的≠现在能不能申请到"）。
- 商家券"查询用户券"接口的完整请求/响应字段（本次研究未取得逐字段文档，只确认了接口存在和大致语义），批次 1 实现时需要用户或后续会话细读该接口的官方参数文档。
- 是否要求"普通商户"与"服务商-子商户"模式有不同的调用参数（`fetchTradeBill` 现有注释里已经踩过这个坑——"若商户号是服务商-子商户模式，账单 API 须服务商凭证 + sub_mchid 参数"，商家券大概率有类似的模式差异，需要批次 1 落地时对照当前商户号的真实挂载模式核实）。
- 微信小店体系（体系 A）的判断是"不可行"而非"暂不确定"——这一条置信度高，因为核销跳转小店结算页这个限制是官方文档明确写的架构性约束，不是配置或权限能绕开的。
