# 关键决策记录

本文件记录“为什么这样做”。`CLAUDE.md` 只保留决策摘要，详细背景放这里。

## 1. 使用 uni-app

决策：使用 uni-app + Vue 3。

原因：起步设想微信小程序 / H5 / App 多端发布、一份代码省成本。**2026-06-14 复核校正（需求清单 R22）：T1「微信原生单源」已把发布目标收窄为只发微信小程序——H5/App 非发布目标（H5 仅内部预览）；uni-app 一份代码的能力保留、只发微信。**

影响：

- 页面、API、组件都优先使用 uni-app 跨端能力。
- 避免某一端私有能力，必要时用条件编译隔离。

## 2. 使用 JavaScript，暂不使用 TypeScript

决策：当前使用 JavaScript。

原因：降低新手理解成本，保持项目推进速度。未来如果业务复杂度提高，可再评估迁移 TypeScript。

## 3. 使用 px，不使用 rpx

决策：统一使用 px。

原因：当前设计稿和布局是流式方案，px 更便于 1:1 对照设计稿，减少转换误差。

## 4. 自绘 TabBar 和导航

决策：使用自绘浮动药丸 TabBar 和自绘导航。

原因：原生 tabBar 难以还原浮动药丸设计；项目也需要更细的顶部安全区控制。

影响：

- `pages.json` 使用 `navigationStyle: custom`。
- 顶部避让统一走 `utils/systemBar.js`。

## 5. 使用 MediaSlot 管理图位

决策：非 hero 图位优先使用 `MediaSlot`。

原因：当前缺真实素材，灰占位能保持结构稳定；未来替换图片、视频、动图时只改数据或组件入参，减少页面改动。

> 收敛（2026-07-16）：本决策仅对旧线 `packages/` 参照物有效。**新线 `rewrite/mp` 已弃 MediaSlot**（UI 方向 A 拍板·cover 直渲 + CSS 灰底占位）；新线图片/资产加载策略 canonical 见 `rewrite/mp/README.md` §「图片/资产加载策略」。

## 6. 主色统一为幸运紫

决策：主色统一为紫色，主题变量放在 `src/uni.scss`。

原因：详情、购物车、结算、个人中心等页面需要统一品牌感，也贴合当前视觉方向。

约束：

- 不要在页面散写 `#a371ea`。
- 优先使用 `$purple`、`$brand` 等 token。

## 7. Sass @import 到 @use 暂缓

决策：暂缓把 Sass `@import` 迁移到 `@use`。

原因：uni-app 自动注入 `uni.scss` 变量，直接迁移到 `@use` 可能导致 `co.scss` 作用域和变量访问变化，存在 scoped 样式静默回归风险。当前告警不阻断构建，收益低于风险。

后续条件：

- 等 uni-app Sass 集成更稳定。
- 有足够时间做 H5 + 小程序页面眼校。

## 8. 顶部安全区使用 systemBar.js

决策：微信小程序顶部安全区不只依赖 CSS `env(safe-area-inset-top)`，改用 `utils/systemBar.js` 动态计算。

原因：`env()` 在小程序尤其安卓上不可靠，且只包含状态栏，不包含右上角胶囊。自绘导航需要同时避让状态栏和胶囊。

影响：

- 非沉浸页已使用动态状态栏和胶囊数据。
- 沉浸页仍需单独处理。

## 9. 敏感业务走云函数

决策：未来订单、支付、积分、审核、售后等敏感业务走云函数，不信任前端传值。

原因：前端传入的 openid、角色、价格、订单状态都可以被伪造。云函数可通过 `cloud.getWXContext()` 获取可信用户上下文，并在服务端完成业务校验。

影响：

- 前端直连数据库只用于非敏感读写。
- 数据库和存储必须配置安全规则。
- 下单、支付、退款、状态变更必须服务端校验。

## 10. 业务数据上云：小程序优先 + store/api 收口 + 本地回退

决策：业务数据上云时，小程序端走云函数取数，H5 / App 端回退本地 `data/*`；不把同步的 `catalog.js` 改成异步，而是新增 `store/products.js` 叠加云端数据，页面从 store 取。

原因：

- `wx.cloud` 只有小程序端有，H5 / App 端没有；为兼顾多端，api 层用条件编译，小程序调云函数、其它端回退本地。
- `catalog.js` 是全店同步单一来源（cart、详情推荐位等同步依赖），改成异步会牵一发动全身、风险大。用 store 叠加云端数据，既拿到云端真值，又不破坏既有同步依赖。

影响：

- 调云函数统一收口到 `utils/cloud.js` 的 `callCloud`（`wx` 引用集中一处 + 条件编译）。
- api 返回 canonical 数据（价格为数字），展示层负责格式化（拼 ￥）。
- 课程 / 订单照此「云函数 + api + store」三件套模式推进。

## 11. 管理后台：CMS 管数据 + 自建管二维码/看板

决策（2026-06-10）：管理后台不全自建。数据管理用云开发 CMS（内容管理扩展，部署在自己环境里）；业务动作（二维码批次生成/导出、发货绑定订单条目、条目级退货处理）与数据看板自建。

原因（收益）：

- **数据 CRUD 零开发**：商品、FAQ、评价等集合的管理界面安装即用，省掉后台最大的一块开发量；P3 的自建范围收窄到「码批次 + 发货绑定」一小块。
- **同源无同步**：CMS 直接管云开发数据库里同一份集合，小程序读的就是后台改的，没有两套库的同步问题。
- **内置账号与角色权限**：运营/管理员分权开箱即有，不用自建后台登录体系。
- **媒体上传顺手**：图/视频传云存储拿 fileID，正好对上 segment.videoFileId 的素材填充流程。
- **退出成本低（关键）**：CMS 只是自己集合上的一层界面，数据始终是自己的；哪天不合用随时弃 CMS 改自建，数据零迁移。

代价（与对策）：

- **深嵌套编辑体验差**：courses 是三层嵌套 JSON，CMS 表单擅长平铺字段，嵌套要么 JSON 手改（易错、无契约校验）要么拆多集合（动 getCourses 数据形状）。→ 对策：**courses 暂不入 CMS**，近期以 seedCourses 种子（git 版本管理）为准源。⚠️ 同日补充需求「商品款式持续新增、配套课程同步新增」（规格 §四-6）给这条对策定了**到期点**：上新常态化前须拍板课程编排形态（拆集合入 CMS / 自建编排页，进规格 §七 待确认），种子方案只是素材期过渡。
- **业务动作做不了**：生成码、发货绑定、退货处理是带校验的业务操作而非 CRUD，必须自建。→ 这本就是方案的「自建」半边；敏感集合（orders/qrcodes/activations）在 CMS 设只读或不暴露，写一律走云函数（与决策 #9 安全底座一致）。
- **资源与开通成本**：CMS 部署占环境配额（静态托管 + 云函数/数据库读写），个人版安装扩展可能要求开通静态网站托管等按量项（以安装时控制台提示为准，待验证）；初期用量很小。
- **通用界面非定制**：核销率、按课程维度的统计视图 CMS 没有，归入自建看板（P5）。
- **维护跟随上游**：CMS 是腾讯开源扩展，升级节奏不受控，大版本升级可能有迁移成本；由「退出成本低」托底。

影响：

- P3 自建范围明确：码批次生成/导出/绑定（云函数 + 简单页面）；P5 自建看板。
- 安装 CMS 时设好角色：敏感集合只读或隐藏。
- 课程内容管理近期流程不变（改种子 → tcb invoke）。

> **§11 对账注记（2026-06-11，用户问询触发）**：原计划「CMS 管数据」的高频场景（商品/课程/文案/卡片/批次/看板）已全部被自建控制台覆盖（§14–§16），CMS 安装由「计划」改为**按需暂缓**——当前没有它独有的高频需求，安装反增一套账号与维护面；将来出现「批量人工修数据/客服改单」类场景再装（双向成本都低，随时可进退）。

## 12. 激活码不绑订单（备货封码）+ 退货验收扫码兜底

决策（2026-06-10）：唯一激活码按「一款产品的配套课程」批量生成、打印，**备货阶段**封入该产品每个包装并封装——码只属于（courseId, batchId），**不与订单/订单条目绑定**（取消原「发货时绑订单条目」环节）。条目级退货权的执行改为两层：① `confirmEnter` 时**启发式自动失效**（确认者本人订单中含该课程对应商品的可退条目，取最早一条 `refundable=false`）；② **强制兜底 = 退货验收扫码**（退货须寄回完整包装，验收扫包装内码：已激活 → 该件拒退；未激活 → 正常退）。

原因：

- 用户明确的备货流程：码在备货时封入包装——发货拣货时拿到的是已封装的盒子，无从知道盒内是哪个码，「发货绑定」物理上不成立；内外双码方案被否（多一份物料 + 多一个发货扫码动作）。
- 验收扫码让退货权的最终裁决回到**物理凭证**：码随包装走，谁退货谁交回码。不依赖订单绑定，自购 / 送礼一致有效（收礼人激活 → 买家手里没有完整包装可退；买家退未拆封件 → 码未激活，正常退）。

代价（与对策）：

- `refundable` 自动失效只在自购场景命中（启发式），送礼场景订单条目标记不更新 → 该字段降为**展示性标记**，App 内提示用，裁决以验收扫码为准。
- 退货权执行依赖运营纪律：**退货必须要求寄回完整包装 + 验收必扫码**，需写进退货政策与验收 SOP；客服需理解「标记」与「裁决」双轨。
- 启发式匹配在「同用户多笔同商品订单」时取最早可退条目，可能与用户心理预期的那一单不一致 → 展示层面说明即可，不影响裁决。

影响：

- `qrcodes` 集合去掉 `orderId` / `orderItemRef` 字段；发货环节零额外动作。
- P3 自建范围再收窄：码批次生成 / 导出打印 / 验收扫码查询（无发货绑定界面）。
- 管理后台订单管理 = 条目级标记展示 + 验收扫码查询裁决。

## 13. 二维码类型：普通链接二维码（一码一地址）

决策（2026-06-11，用户拍板）：印刷物料走**普通链接二维码**——每个激活码对应一个网址（形如 `https://<域名>/q/<激活码>`），把网址清单交给印刷厂，「一个地址印刷一个二维码」。放弃原推荐的小程序码（`wxacode.getUnlimited` + scene）。

原因（收益）：

- **印刷流程最简**：交付物是一列文本网址，印刷厂自己生成二维码图样，我们不用逐码调接口生成图片文件再交付。
- **码本身可读**：网址尾段就是激活码，客服 / 退货验收人员肉眼可读、可手工查询，不依赖扫码设备。
- **不占小程序码接口**：无生成量与频率的接口依赖；URL Link 方案则因微信已不支持永久有效链接，印刷物料（要活数年）直接出局。
- **域名落地页有余地**：微信外扫码会落到该网址，可放一页「请在微信中打开」引导（静态托管即可）。

代价（与对策 / 依赖）：

- **依赖一个 ICP 备案域名**：微信「扫普通链接二维码打开小程序」要求域名已备案 + 在小程序后台配规则（前缀绑定 welcome 页 + 上传校验文件）。→ ✅ 用户确认已有备案域名（2026-06-11）；剩后台规则配置 + 真机扫码验证。
- **规则配置前印刷不可开机**：先配好规则、真机扫码验证通过，再批量交印。→ 测试期用 体验版启动参数 / 编译模式模拟进入（welcome 已兼容 `?code=`）。
- 微信内扫码进小程序时链接在启动参数 `?q=`（urlencode）里 → welcome 已增 `?q=` 解析（取 `code=` 参数或路径最后一段），与 `?code=` / scene 三轨兼容。

影响：

- `genQrcodes` 职责不变（只产码）；印刷交付物 = 按批次导出「码 → 网址」清单（拼前缀即可，归管理后台批次导出）。
- 印刷物料的剩余依赖收敛为：① 确认/备案域名；② 小程序后台配「扫普通链接二维码打开小程序」规则；③ 真机扫码验证（验收单 F 补扫码项）。

## 14. 视频上传控制台：自建网页编排页 + 云存储 v1

决策（2026-06-11，用户提出需求并定方向）：视频上传控制台做**网页版自建**（要求：逻辑清晰、支持拖拽排序、方便命名），形态为「课程列表 + 三层编排页（章节/课时/小段）+ 分段上传 + 草稿/发布分离」。子规格见 史料「设计规格-管理控制台.md」，UI 设计稿 `design/console.pen`。（2026-06-11 同日追加：二维码卡片设计板块并入本控制台，见子规格 §六——印刷交付物从「码 → 网址清单」升级为「卡面设计 + 一码一张卡」，补全决策 §13 的印刷链路。）

这同时落定主规格 §七 两个挂账待确认项：

- **课程编排形态 = 自建编排页**（决策 §11 当时的对策「courses 暂不入 CMS、种子当准源」到期——上新常态化前须有编排工具，本决策即答案）。
- **视频存储 v1 = 云开发云存储**（fileID 挂 `segment.videoFileId`，与既有数据模型零冲突；VOD / 防盗链 P5 再评估，不阻塞素材灌入）。

原因（收益）：

- 三层嵌套结构 CMS 表单做不了（§11 已论证），自建编排页是唯一能把「拖拽排序 + 原地改名 + 分段上传」做顺手的路。
- 网页版（而非小程序内管理页）：电脑上传大视频文件、批量整理素材的体验远好于手机；部署在云开发静态托管，同环境同数据零同步。
- 草稿 / 发布分离（`coursesDraft` → 发布覆盖 `courses`）：编辑过程不影响线上学员，小程序端零改动。

代价（与对策）：

- 多一个独立小工程（admin/）要维护 → 复用同一套 Vue 3 + Vite 技术栈与云函数安全底座，能共享的契约共享。
- Web 端无微信 openid，管理员鉴权要走 CloudBase 登录态 → 云函数按 uid 对照管理员名单二次校验（不信任前端，实现时细化）。
- 旧视频替换后的存储清理 → 记技术债，不阻塞 v1。

影响：

- 「seedCourses 种子当准源」降级为初始化工具；课程内容准源逐步移交控制台。
- 小程序播放页需配套「按段取 fileID 播放」改造（素材灌入后单独排里程碑）。
- 侧边栏为「二维码批次」「数据看板」留位（决策 §13 管理半边与 P5 看板的落点）。

## 15. 控制台主轴 = 商品上新流水线（商品建档进自建控制台）

决策（2026-06-11，用户明确逻辑层级）：管理控制台以**商品**为主轴组织：创建商品 → 上传产品图片 → 填写 SKU → 上传编排配套教学视频 → 设计二维码卡片（可自定义尺寸）→ 批量生成二维码地址，**打包下载 SVG 设计图 + 地址清单**交印刷厂。子规格 §七。

原因（收益）：

- 与真实业务动线一致：上新就是「一款商品走完六步」，比按工具分模块（课程/卡片各一摊）更不容易漏步骤。
- 商品图片在控制台上传后直接喂小程序商品页——真实素材替换灰占位有了正式入口。
- SVG 矢量交付印刷厂，配合「一地址一码」的既定工作方式（§13），印刷链路全自动化。

代价（与对策）：

- 商品建档 / 图片 / SKU 进自建控制台，§11「CMS 管商品数据」降级为兜底查看 → CMS 仍装（FAQ/评价等平铺数据 + 应急直改），商品的写路径收口到控制台云函数。
- 控制台 v1 范围变大（多了商品三步）→ 分期消化：v1 流水线骨架 + 视频编排；v1.5 卡片 + 印刷包。

影响：

- `products` 模型扩展（images / skus）；小程序详情页规格组件后续从样例切真。
- 设计稿补 S5 商品工作台（六步导航）；S1 课程列表职责并入商品列表。

## 16. 管理台形态升级：一比一所见即所得橱窗

决策（2026-06-11，用户提出）：小程序信息（排序、上下架、文案等「长什么样」的内容）的管理形态定为**一比一所见即所得**——控制台内嵌手机框预览（iPhone 17 比例，用户指定），与小程序同一份云端数据渲染，在画面上直接拖动/编辑，与右侧面板双向联动。子规格 §八。

原因（收益）：非技术运营者零学习成本（看到的就是用户看到的）；数据本就同一份、设计 token 共用，等价渲染成本低；为「小程序信息全面可管理」铺路（逐块接入路线见 §八）。

代价（与对策）：等价组件需随小程序 UI 改版同步维护 → 只复刻「可管理区块」关键视觉；区块组件小程序侧已是纯 props（技术债 #4 成果），形状稳定。

影响：products 的 featured 语义 = 首页橱窗位；后续内容块上云时小程序侧从静态 data 切 store（照三件套样板）。

## 17. 代码按技术分层组织（package-by-layer），不按业务链重组

决策（2026-06-11，用户提出并拍板）：代码保持**按技术分层**组织（`pages / store / api / data / components / utils / cloudfunctions`），**不**改成「按业务链分文件夹」（package-by-feature）。每条业务链的「整体视图」由 `docs/业务逻辑架构.md` 提供——**逻辑上集合、物理上分布**的折中。

背景：用户提出「物理就近 → 人更易整体理解」（业界 proximity / package-by-feature 原则，方向正确）。但全量物理集合在本项目**做不到**。

原因：
- **硬约束**：① 每个云函数是独立部署单元（`cloudfunctions/<name>/` 自带 package.json、`tcb fn deploy` 单独发），其「云端那一半」无法并入某条链的功能文件夹；② 管理控制台是独立工程（`admin/`，独立构建/部署）；③ 共享代码（cart / catalog / utils）跨多条链复用，按功能切会被迫复制（破坏单一来源）或进 `shared/`（又散回去）。
- **软代价**：按功能重组要改全部 import 路径，回归风险高，支付前不值得（同「不为整洁动工作代码」纪律）。

影响：
- 新增功能仍按层放置（页面进 `pages/`、状态进 `store/`、接口进 `api/`、云端进 `cloudfunctions/`）。
- 页内自用代码就近 colocate 于 `pages/<page>/components/`（如 `detail/components/`、`player/components/HelpSheet/`）——局部就近仍坚持。
- **业务链或状态机变化时，先改 `docs/业务逻辑架构.md`（按链地图）再动代码**，让「整体视图」不漂移。

## 22. 故意不抽象：投机泛化默认拒绝（YAGNI / Rule of Three）

决策（2026-06-25，用户提「过度工程防治」并拍板）：本仓**默认拒绝投机抽象**——内联具体优先；新增接口/层/泛型/配置项须有当下真实需求，理由含「万一以后/方便将来」即删；相似两段先问「本质相同还是碰巧长得像」、宁可先重复，到 Rule of Three 第三次方向看清再提。横切方法论决策（与 §17 同级），和 CLAUDE §7「抽象简约」尺子互为表里——**ADR 答为什么 + 攒判例库，尺子是写码时的尺**。

为什么需要这条：治理框架（元模式 + 十三类病根 + 64 守卫）整套在**防「做得不够」**，每条守卫都逼把该做的做到；这套「每主张配守卫」范式自带反向风险——守卫越堆、抽象越厚。过度工程＝替用户没给的数字（规模/寿命/用途）瞎猜天花板：约束一旦明确（一次性脚本 / 3 个内部用户 / 半年必重写）就不该按企业级铺。且错误抽象比重复更贵——「碰巧像」当「本质同」合并，后续拆解代价高。

判例（本仓既有「故意不做」实例，后续同类决策引此为锚）：
- **S11 商品不引 xCounts**（`94c797b`）：草稿量小全量加载，计数/筛选/搜索前端做即诚实，不为「将来量大」预建后端聚合层。
- **cs 客服域不立 P3 空册**（§21 / `5d64446`）：查证无自有 status 状态机 → 不为凑数立册（元模式「别摆设守卫」实例）。
- **§11 CMS 按需暂缓**：当前无独有高频需求 → 不预装一套账号与维护面，将来真出现再装。

边界补丁（2026-06-25 本会话）：**未来式 ≠ 投机式**。`MediaSlot` 的 video/gif/lottie 类型、`logger` 的错误上报钩子、`TroublePanel` 真素材说明，均属于已明确的业务预留，不按「方便将来」删除。流程上遇到「以后/预留/将来」先做三分流：① 能追到需求/ADR/现状路线/用户明确意图＝保留；② 只有泛泛未来收益＝删或延后；③ 来源不明＝先问，不直接改。

## 30. tab 页误触退出提醒：机制定选 `page-container`、`enableAlertBeforeUnload` 判废（2026-07-14·终结三次翻烧饼）

决策（2026-07-14，用户「开一个彻底的研究，到底能不能做」后，16-agent 最大化研究——官方文档 + 社区真机 + 仓内存档 + 4 路对抗验证——定论）：

- **能做，但只有一条路**：tab 栈底页（首页/购物车/我·返回=退出小程序）的误触退出提醒**只能用隐形 `<page-container>` 的「再按一次退出」**，且**只可靠覆盖安卓返回键**。`wx.enableAlertBeforeUnload` **判废、不再用于本用途**。
- **为什么 `enableAlertBeforeUnload` 无效（双死因）**：① 它是"返回上一页"确认框、语义绑 `navigateBack`/左上角原生返回按钮，tab 栈底页无上一页；② 本 app `navigationStyle:custom`（home.json 自绘导航）**藏掉了它要挂的原生返回按钮**（社区证"只有非自定义标题栏才触发"）→ 首页永远不弹。**铁证**：仓内 史料「调试日志-至20260616.md」（2026-06-14）早已真机测出「真机无弹窗」。
- **为什么 `page-container` 有效**：拦返回看"本页有无武装容器"、**与栈深无关**，故栈底页照拦（`<page-container show overlay="{{false}}" duration="{{0}}" bindbeforeleave>`·基础库 ≥2.16.0）。武装时给 Page 注入 `position:fixed`、锁页面级滚动（overlay=false 不解锁）→ **内容须套 `<scroll-view style="height:100vh">`**。旧 uni-app 线（`packages/miniapp` useExitGuard.js）安卓真机验证过、是唯一带真机标签的结论。
- **平台天花板（任何机制都拦不住·非缺陷）**：胶囊「×」/ Home 键 / 上滑杀进程 / **鸿蒙 HarmonyOS**（官方明说不能拦）。iOS 边缘侧滑退出=未证实、须真机（可能半滑卡死/白屏）。
- **翻烧饼溯源（三方各对一半、互相说岔）**：「平台不可能」(mp-7fixes)对 enableAlertBeforeUnload/×/Home 对，但过度泛化、忘了 page-container；「反转误判」(62ac602)对"native 源码里没写过该 API"，但错在把已真机失败的 enableAlertBeforeUnload 搬回来、还称旧结论"从没真用过"（存档证明测过/失败）；旧 uni-app「page-container 有效」是唯一经真机确证的正解。**根子：devtools 里 page-container 显示不工作（只真机生效·根因#10）→ 屡屡假阴性。**
- **✅ 已落地（2026-07-18·用户「tabbar页需要防止误触退出」·过程账见 archive 重构日志 fable 分册）**：撤 `armExitAlert` 死代码 → 三 tab 页移植 page-container 三件套 + 整页套 `<scroll-view>`（home 连带迁 refresher/scroll-into-view/bindscroll）；守卫 `rw-mp-tabbar-exit-guard` 六断言焊死（含判废 API 防回潮）。**要撤掉退出提醒须先改本 §30 再退役该守卫**（删与加对等）。详见 `重构日志` 2026-07-18 同日条；真机 7 项待验（根因#8）。
- **现状与修法（落地前的记录·留作经过）**：main（PR #24·c8d2680）线上曾是那版**无效的 enableAlertBeforeUnload = 回退/死代码**，须撤 + 移植 page-container。**移植代价**：me 低（套 scroll-view）、cart 中（固定"去结算"栏留 scroll-view 外当兄弟+安全区）、**home 高**（强制 scroll-view 破坏下拉刷新 P0/滚动回顶 P1/`wx.pageScrollTo` 锚点跳转×4 P1/横向 rail 嵌套——单独排期）。用户测试用**安卓**、安卓返回键这条对他有效。落地待用户拍板走 `/refactor-batch`（详见 `待办与债` 同条）。

## 31. 视频转码管线拍板：腾讯云点播 VOD（2026-07-18·转码=播放丝滑第一杠杆的落地方向）

决策（2026-07-18，用户「落地。」——此前同日三轮 workflow 调研：拖动实时跟随可行性 → VOD 成本模型 → 落地风险评估，共 15 agent、载荷断言全过对抗核验）：

- **选型：腾讯云点播 VOD**（弃数据万象 CI）。三路证据同向：① 能力面——转码+自适应+雪碧图+封面一站（雪碧图正补 R36、封面补 poster 缺口，CI 的雪碧图只是截帧输出模式）；② 成本面——VOD 媒资存储 0.144 元/GB/月 vs 云开发容量 3 元/GB/月（20 倍差），CI 产物存回云开发桶会把最大省钱项吃掉；③ 风险面——云开发桶直绑 CI 无官方文档背书。转码后每学员每遍观看流量缩 4~8 倍（百学员月账约 187→30 元），转码费一次性个位数元/课。
- **「拖动画面实时跟随」定性为伪需求、正解=雪碧图预览**：微信 `<video>` seek 秒级无精确选项、iOS 回退 1s、`seekcomplete` Android 不可靠、生态零成功案例、微信自家 controls 都是松手才跳；抖音/B站/YouTube 的「拖到哪看到哪」全部是预生成雪碧图（官方文档/逆向证实）。播放器两段式拖拽与守卫 `rw-mp-player-immersive` 不动。
- **架构**：平台接缝单点 `kit/vod.ts`（镜像 callFlow·守卫 `rw-vod-seam-single`）；`isVodFileId`（纯数字 FileId vs `cloud://`）判据单源 `@ldrw/shared`，播放分流/GC 删除/admin 转码中显示三处共用；Key 防盗链签名纯本地 md5（有效期 6h > mp 25min 地址缓存 + 课时长，< GC 24h 缓期）、playKey 未配 fail-closed url:null（守卫 `rw-vod-sign-fail-closed`）；上传=admin 浏览器 vod-js-sdk-v6 直传 + UGC 签名（云函数纯 HMAC 派发）+ procedure 自动触发任务流；转码感知=拉模式（admin「同步转码状态」on-demand DescribeMediaInfos·**刻意不配回调**——云函数无公网入口，不为此开 HTTP 访问服务）；发布闸 VOD_PROCESSING（传完≠可播·防伪成功）。**未配置全链自动回退云存储老路，配置填入即切换（零部署·灰度零风险）**；**帮助视频线恒走云存储不迁**——其播放侧只认云存储 fileID（无转码同步无发布闸兜底），故共用的 `uploadVideo` 走 VOD 须显式 opt-in、默认老路（守卫 `rw-admin-help-video-stays-cos`·批2 自纠详见重构日志同日条）。
- **运营模式变化（已知代价）**：上传完须等转码（1-2 分钟短段几十秒到几分钟、100 段自动排队约几十分钟、大半被上传时间掩盖）才能发布；发布前多一步「同步转码状态」。
- **遗留与前置**：批3（mp 拖动预览窗消费 vodSpriteVtt + poster 进播放器）依赖真实转码产物后另立批；**人工前置**=secureConfig 集合锁权限（最高优先旧洞）→ 控制台五步（`console-assets/04` 重建手册）→ 真实视频全链验证（TC3 手签/防盗链签名的正确性只有真云证得了·根因#8）。
