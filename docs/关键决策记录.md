# 关键决策记录

本文件记录“为什么这样做”。`CLAUDE.md` 只保留决策摘要，详细背景放这里。

## 1. 使用 uni-app

决策：使用 uni-app + Vue 3。

原因：项目目标是微信小程序 / H5 / App 多端发布，一份代码多端更符合当前阶段的成本收益。

影响：

- 页面、API、组件都优先使用 uni-app 跨端能力。
- 避免某一端私有能力，必要时用条件编译隔离。

## 2. 使用 JavaScript，暂不使用 TypeScript

决策：当前使用 JavaScript。

原因：降低新手理解成本，保持项目推进速度。未来如果业务复杂度提高，可再评估迁移 TypeScript。

## 3. 使用 px，不使用 rpx

决策：统一使用 px。

原因：当前设计稿和布局是流式方案，px 更便于 1:1 对照设计稿，减少转换误差。

## 4. 自绘 TabBar 和导航

决策：使用自绘浮动药丸 TabBar 和自绘导航。

原因：原生 tabBar 难以还原浮动药丸设计；项目也需要更细的顶部安全区控制。

影响：

- `pages.json` 使用 `navigationStyle: custom`。
- 顶部避让统一走 `utils/systemBar.js`。

## 5. 使用 MediaSlot 管理图位

决策：非 hero 图位优先使用 `MediaSlot`。

原因：当前缺真实素材，灰占位能保持结构稳定；未来替换图片、视频、动图时只改数据或组件入参，减少页面改动。

## 6. 主色统一为幸运紫

决策：主色统一为紫色，主题变量放在 `src/uni.scss`。

原因：详情、购物车、结算、个人中心等页面需要统一品牌感，也贴合当前视觉方向。

约束：

- 不要在页面散写 `#a371ea`。
- 优先使用 `$purple`、`$brand` 等 token。

## 7. Sass @import 到 @use 暂缓

决策：暂缓把 Sass `@import` 迁移到 `@use`。

原因：uni-app 自动注入 `uni.scss` 变量，直接迁移到 `@use` 可能导致 `co.scss` 作用域和变量访问变化，存在 scoped 样式静默回归风险。当前告警不阻断构建，收益低于风险。

后续条件：

- 等 uni-app Sass 集成更稳定。
- 有足够时间做 H5 + 小程序页面眼校。

## 8. 顶部安全区使用 systemBar.js

决策：微信小程序顶部安全区不只依赖 CSS `env(safe-area-inset-top)`，改用 `utils/systemBar.js` 动态计算。

原因：`env()` 在小程序尤其安卓上不可靠，且只包含状态栏，不包含右上角胶囊。自绘导航需要同时避让状态栏和胶囊。

影响：

- 非沉浸页已使用动态状态栏和胶囊数据。
- 沉浸页仍需单独处理。

## 9. 敏感业务走云函数

决策：未来订单、支付、积分、审核、售后等敏感业务走云函数，不信任前端传值。

原因：前端传入的 openid、角色、价格、订单状态都可以被伪造。云函数可通过 `cloud.getWXContext()` 获取可信用户上下文，并在服务端完成业务校验。

影响：

- 前端直连数据库只用于非敏感读写。
- 数据库和存储必须配置安全规则。
- 下单、支付、退款、状态变更必须服务端校验。

## 10. 业务数据上云：小程序优先 + store/api 收口 + 本地回退

决策：业务数据上云时，小程序端走云函数取数，H5 / App 端回退本地 `data/*`；不把同步的 `catalog.js` 改成异步，而是新增 `store/products.js` 叠加云端数据，页面从 store 取。

原因：

- `wx.cloud` 只有小程序端有，H5 / App 端没有；为兼顾多端，api 层用条件编译，小程序调云函数、其它端回退本地。
- `catalog.js` 是全店同步单一来源（cart、详情推荐位等同步依赖），改成异步会牵一发动全身、风险大。用 store 叠加云端数据，既拿到云端真值，又不破坏既有同步依赖。

影响：

- 调云函数统一收口到 `utils/cloud.js` 的 `callCloud`（`wx` 引用集中一处 + 条件编译）。
- api 返回 canonical 数据（价格为数字），展示层负责格式化（拼 ￥）。
- 课程 / 订单照此「云函数 + api + store」三件套模式推进。
