# 代码标准（Lucky Ducky）

> **用途**:代码审核 + 新代码自查的统一尺子——判断代码是否干净、整合、位置正确、拓扑清晰、可长期维护。
> **怎么用**:写完/审查时对照各维度;命中问题按 §20 定级(P0–P3)。日常只看文末「red flags 速查」即可快速过一遍。
> **与其他文档分工**:本文是"代码该怎么写";`CLAUDE.md` = 项目硬约定(技术栈/目录/命名/token);
> `调试日志.md` = bug 账本;`技术债与重构.md` = 欠债 backlog;`设计规格-课程电商系统.md` = 新系统设计。
> 标 **`▸ 本项目`** 的行是把通用原则对到真实代码/现状,优先看它。

## 1. 模块边界
- 页面只负责编排,不承担大量业务细节。
- 通用组件不包含页面专属业务。
- 页面专属组件放对应 `pages/xxx/components`。
- store 不依赖页面数据;utils 不依赖 store/pages/components。
- data/mock 不直接泄漏到深层展示组件;API/service 层与 mock data 有明确边界。
- ▸ 本项目:`store/address.js` 现从 `data/checkout.js` 引 `SAMPLE_ADDRESS`,是越界(技术债 #3,待解耦)。

## 2. 依赖方向
理想:`pages → feature components → shared components`;`pages → store/services`;`services → api/data`;`store/components → utils`。
不理想:`store → page data`、`global components → business mock`、`utils → store`、`shared components → page-specific logic`。
- ▸ 本项目:整体无循环依赖(优点)。违例集中在「组件直接取数」(§6)与「store→checkout 数据」(§1)。

## 3. 组件纯度
- 纯展示组件只收 props、发 emits;容器组件负责取数/拼状态/副作用。
- 一个组件别同时做取数 + 状态 + 复杂交互 + 展示。
- 插槽用于合理扩展,不让接口变隐式难懂;组件名表达**业务角色**而非 UI 形态。
- ▸ 本项目:`AddressBlock`/`OrderItem`/`PriceSummary`/`QuantityStepper` 方向正确;`HelpSheet` 是 player feature 却在全局且过大(技术债 #2)。

## 4. 状态归属
- 跨页面状态进 store;页面临时态留 page;组件交互态留 component。
- 可恢复状态持久化;临时草稿不持久化(除非有明确恢复策略);状态变更入口集中。
- ▸ 本项目:`cart.items`/`address.list`/`user.profile` 持久化;`checkoutItems` 草稿**故意不持久化**(冷启动不残留)。

## 5. 副作用归属
- 跳转、toast、modal、storage、timer、request 都是副作用,别散在深层纯组件里。
- timer 必须清理;storage key 不散字面量、用常量/service 收敛;网络请求统一入口 + 错误处理;生命周期逻辑不混入展示组件。
- ▸ 本项目:`ld_video_intro_seen` 等 storage key 仍散在页面(技术债 #7,待收敛);日志已有统一出口 `utils/logger.js`。

## 6. 数据流清晰度
- 数据自上而下走 props,事件自下而上走 emits。
- **展示组件不直接 import mock 数据**;同一派生结果别多处各算;金额/数量/状态标签有单一计算来源;query 解析集中或有校验。
- ▸ 本项目:`FeatureProducts`/`Reviews`/`FAQ`/`Reassurance`/`TrustStrip` 仍直接 `import data/*`(技术债 #4)——接云开发时统一改 props + adapter。

## 7. 业务不变量
- 购物车数量必须是**正整数**(非小数);金额必须是有限数字;空商品不能结算;无地址不能提交。
- 订单 id 必须能关联订单详情;同一状态不应有多个真相来源;过期订单不能支付;删除默认地址后默认状态可恢复。
- ▸ **云开发新增不变量**(见设计规格):只能查看自己激活的课程;二维码**一码一用**、不可重复激活;"确认进入即失退货权"必须留时间戳;价格/数量/激活/支付**不能仅靠前端信任**,后端校验。

## 8. 文件位置
跨页面组件 `src/components`;页面专属 `src/pages/xxx/components`;业务服务 `src/services`;后端请求 `src/api`;mock `src/data`;状态 `src/store`;纯工具 `src/utils`;常量 `src/constants`;组合逻辑 `src/composables` 或 feature 内 `useXxx.js`。
- ▸ 本项目:暂无 `services`/`constants`(接云开发时建);`HelpSheet` 位置待修(§3)。

## 9. 文件大小（参考,非硬线）
纯组件 `<150` 行;页面 `<250–300`;store `<150–200`;单 feature `>300` 考虑拆;`>500` 需明确理由或拆分计划。

## 10. 函数粒度
- 一个函数只表达一个意图;函数名说**业务动作**而非实现细节。
- 别一个函数同时做校验 + 状态变更 + 跳转 + toast;复杂流程拆 `validate → mutate → side effect`;公共逻辑别复制粘贴多页。
- ▸ 本项目:`checkout.onSubmit` 已是"校验(空草稿/地址)→ 扣减 → 跳转"的清晰序。

## 11. 命名一致性
- 领域名统一(product/item/goods 别混);订单状态 key 统一(pending/toship/toreceive/done);组件名表语义(`AddressBlock` 优于 `InfoCard`);CSS 前缀统一;mock/demo/sample/fallback 准确区分。

## 12. 样式拓扑
- 设计 token 全局(`uni.scss`);组件私有样式留 scoped;真正通用的 utility 才全局(`co.scss`)。
- **不要同名样式在全局和组件内双写**;页面样式不承担组件内部结构;复用通过组件/token 而非复制 class。
- ▸ 本项目:`co.scss` 与组件 scoped 有同名双写(技术债 #6);注意 scoped 与 co.scss **互相够不到**的铁律(见 CLAUDE.md §8 样式规则)。

## 13. 错误处理
- 请求失败统一处理;storage 解析失败不崩;空数据有空态;query 缺失有 fallback 或拦截;用户动作失败有反馈。
- 日志统一出口,别散落 `console`。
- ▸ 本项目:已建 `logger` + 全局 `errorHandler`/`uni.onError`;`persist` 坏数据不崩 + `sanitize` 清洗;`checkout` 空草稿守卫。

## 14. 测试价值
- 测**业务不变量**而非实现细节;store actions/getters 要测;sanitize/validate 测脏数据;金额/数量扣减/默认地址要测。
- 测试**调用真实函数**、别复制生产逻辑;demo seed 行为若保留,显式测试或注释。
- ▸ 本项目:`tests/` 已覆盖 cart/persist/validate/format;`persist.test` 用真实 `sanitizeCart`(非复制)。

## 15. 可替换性
- mock 能否替换成 API 而不改大量组件?图片占位能否换真实媒体?支付 mock 能否换真实支付?订单静态数据能否换 store/service?是否有硬编码路径/状态/文案使替换困难?
- ▸ 本项目:`data/*` → 云函数 是下一步主线;§6 的"组件直接取数"是替换成本的主要来源。

## 16. 演进成本
- 新增页面能否复用现有组件?新增订单状态是否只改配置?新增商品字段是否影响多组件?新增接口是否有清晰 service 入口?新增主题/样式是否依赖 token?

## 17. 注释质量
- 注释解释**为什么**而非重复做什么;Demo/mock 行为必须标注;TODO 有归属和方向;过期注释要清理;注释不能掩盖不清晰的结构。

## 18. 安全与鲁棒性
- 不信任 storage/query/API 返回;所有外部输入都校验;禁 `eval`/`v-html` 等危险入口(除非强约束);用户资料/token/手机号等敏感信息不乱日志;价格/数量不能仅靠前端信任。

## 19. 性能与资源
- 长列表是否需分页/虚拟化;大图是否懒加载/占位;timer 是否按需启停;页面切换是否残留订阅/定时器;重计算是否 computed 缓存;大组件是否致不必要重渲染。

## 20. 审核分级
- **P0**:会崩溃 / 数据错乱 / 交易错误 / 安全风险。
- **P1**:主流程明显错误 / 状态不一致 / 架构边界严重错误。
- **P2**:维护成本高 / 重复明显 / 位置不当 / 未来扩展困难。
- **P3**:风格不统一 / 命名不佳 / 注释过期 / 轻微技术债。

## 21. uni-app / 小程序专项（本项目硬约束）
- 只用 uni 跨端 API(`uni.xxx`),不用某端私有能力。
- **不内联 `<svg>`**;图标用 `<image src=svg>`(见 `Icon`)。**CSS 不 `background-image:url(本地)`**;本地图用 `<image>` 或纯色占位(见 `MediaSlot`)。
- 避 `backdrop-filter`/`color-mix()` 等端不兼容属性,用近似色/半透明。
- **顶部安全区/胶囊用 `utils/systemBar.js`(动态),不要硬编码 `env(safe-area-inset-top)`**;H5/App 用条件编译保 env 兜底。
- 跨端差异用条件编译 `// #ifdef MP-WEIXIN`(scss/js/template 各有写法)。
- 单位用 **px**(本项目流式布局、与设计稿 1:1)。
- 动态值经 `:style` 注入 CSS 变量、scoped 里 `var()` 取(scoped 拿不到 JS 值)。

## red flags 速查（看到就警惕）
| 信号 | 问题 | 维度 |
|---|---|---|
| 展示组件里 `import @/data/...` | 取数混进展示 | §6 |
| 页面专属组件躺在 `src/components/` | 位置错(如 HelpSheet) | §3/§8 |
| `store` 里 import `data/某页` | 域耦合(如 address←checkout) | §1 |
| `co.scss` 与组件 scoped 同名 class | 双写易漂移 | §12 |
| `uni.setStorageSync('字面量')` 散落 | storage key 未收敛 | §5 |
| `env(safe-area-inset-top)` 硬编码 | 该用 systemBar | §21 |
| 裸 `console.*` | 该走 logger | §13 |
| 一个函数又校验又改状态又跳转 | 粒度过粗 | §10 |
| 数量/金额未校验就入库或计算 | 业务不变量缺失 | §7/§18 |

## 简化评分表
| 维度 | 好代码 | 差代码 |
|---|---|---|
| 边界 | 页面编排,组件展示,store 管状态 | 页面/组件/store 互相串 |
| 数据流 | 上下清晰,单一来源 | 多处 import data,各算各的 |
| 副作用 | 集中、可追踪、可清理 | toast/storage/timer 到处散 |
| 样式 | token + 私有 scoped | 全局/组件同名双写 |
| 测试 | 锁业务不变量 | 只测表面或复制逻辑 |
| 位置 | feature 归 feature,shared 归 shared | 页面专属代码进全局 |
| 演进 | 替换 API 成本小 | 接后端要改一堆组件 |
