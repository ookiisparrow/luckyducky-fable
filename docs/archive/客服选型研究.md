## §0 TL;DR / 结论

- **客服不是空白**：项目早已建好并真机验过「独立微信客服」智能客服（`cs/kfCallback`+`kfBind`+`kit/wecom.ts`+`dispatch.ts`+4 入口），基础免费 + 企业微信认证 ¥300/年。短板＝话术写死、仅关键词无 AI、转人工自动分配卡权限、查订单未通。
- **自建底子已大半具备**：现有 Vue3 admin + 24 集合（orders/users/afterSales/activations/events/progress/inventory…）+ 可复用 CS helper（`resolveOpenid`/`summarizeOrders`），足以作「360 坐席工作台」的起点。
- **微信生态枢纽＝微信小店**：一处铺货跨 视频号/公众号/小程序/搜一搜/朋友圈，自带客服/订单/物流/会员；现有自建小程序可经官方「小程序连接小店」API 接入。**微信客服**覆盖 6 入口＝天然跨节点统一接待中枢（H5 例外，需 WeCom 兜底）。
- **三条主路**（构建 vs 采购，按需求五层评）：① **自建**（现有 admin 上长 360 工作台 + 微信客服跨节点 + 接微信小店）——最省钱最灵活、最贴现有资产；② **付费一线客服 API 喂数据**（七鱼/Udesk WeiPeng）——5+ 坐席、要托管客服时再评；③ **电商中台**（有赞/微盟）——最省事但数据锁死、与 T1 自建路线冲突。
- **三件「未拍板」**：AI 路线（元宝·Hunyuan / Claude / FastGPT·Dify / WorkBuddy）、构建 vs 采购、是否上微信小店并推翻 **T1「只发微信小程序单源」**。本文给利弊，决策权在用户。

> **2026-06-30 定案（用户拍板·实现各为后续批次）**：① **改写 T1（非推翻）**——代码不变量 `api-cloud-only` 不松，放开分发范围接微信小店+视频号/搜一搜，**H5 自有官网暂缓**；② **360 坐席工作台＝自建**于现有 admin（加 `getCustomer360`+搜索+RBAC），不采购付费一线/不上中台；③ **AI＝元宝/Hunyuan、时机押后**（量起来、关键词兜不住再上）。详 `关键决策记录.md §19`。

> **2026-06-30 爆破澄清（用户逐问深挖根本需求·结论沉淀，详见 §5/§6/§6.5）**：
> - **微信小店＝分发渠道、非实物引擎**（倾向）：核心商品是「材料包+激活码+课程」捆绑，激活链绑 CloudBase openid，**课程/激活/视频体系装不进小店**（Q3/Q4 文件实证）→ 数据天然双源（小店实物 + 云端课程/激活/画像），**自建工作台的使命＝把双源并成统一 360**，小店是数据源之一、非替代品。
> - **360 数据底座≈8 成已就位**（Q5）：真要新建四块——`getCustomer360` 聚合 / 客户检索+画像 / 坐席承面(A 企微侧栏 vs C 自建工作台·**未定**) / 坐席 RBAC（现为单超管口令）。**只读「客户360查询页」天级、完整工作台周级**。
> - **链②（external_userid→openid）为 A/C 共同命门、且是先验头道**（Q6）：要 `kfBind`+小程序绑开放平台；未通则两承面都只能手动搜、跳不了自动 360。「侧栏取 external_userid」只是 **A 专属疑虑**（C 自建工作台天然知会话身份、无此卡点）。
> - **学习上下文进 360**（Q7）：`progress.last{lessonId,segmentId,at,dur}`+`done`+`updatedAt` 已精到「看到第几秒」（resume 在用），可低成本接进 360「学习位置/卡点」栏，助客服精准指导钩织学习。

> **2026-06-30 爆破收敛 v1（用户拍板·客服方案成型，详见 §6.6/§6.7/§6.8）**：客服＝「微信生态多节点接住 + 统一 360 + 按意图分流」服务体系。已定：
> - **双轨**：订单/售后类→微信客服（机器人+360 自动答）；**钩织学习求助类→引导加企微做 VIP 私域沉淀**（Q8）。
> - **承面 A+C 混合（S1 更新）**：内部 VIP 学习指导走企微侧栏(A)；**外包订单/售后客服走自建独立后台(C)**（外包不进你企微）。承面↔意图干净映射：订单类→C(外包)、学习类→A(内部)。
> - **沉淀机制＝官方「联系我」插件**（免费可用）；获客助手（计费/小程序集成排期未定·待问销售）押后（Q8）。
> - **入口话术功能化**（Q9）：小程序入口用「向讲师请教这个针法」功能性描述；**VIP/专属/私人一对一 价值感挪到加完好友后的企微内兑现**——直接当小程序钓饵＝踩诱导添加红线会被拒。
> - **合规走得通**（Q9）：Flow A 全安全；Flow B 三个「做对才过」＝插件不 scheme + 话术功能化 + 隐私声明数据共享。**待用户核**：课程类目资质（钩织兴趣类·门槛通常低·项目已生产或已处理）。
> - **功能候选**：Q7 学习位置 + **Q10 关键节点拍照留底 + 二分诊断**（结构化诊断·反向加固 VIP 可持续；MVP 小切口；**新合规点 `imgSecCheck` 必加**）。
> - **未定**：微信小店仅渠道 vs 引擎化（倾向仅渠道）；实施步进先只读 360 查询页 vs 直奔完整工作台。

> **2026-06-30 场景覆盖审核 S1–S5（用户逐场景审是否被方案覆盖·详 §6.9）**：S1 外包客服→**承面升级 A+C 混合**(外包走自建后台 C·RBAC 需外包细化)；S2 VIP 直播教学→直播是**客服方案外的新内容模块**(推荐企微群直播·别自建 RTC)；S3 设施托 CloudBase 将来可迁→**守 T2 kit 收口即可·别建迁移抽象**；S4 机器人分流固定流程→**✅ 方案核心**(读类闭环·写类/判断类只路由)；S5 沉淀痛点→知识库→**新增知识库模块**(升格写死 FAQ 为单源·喂 AI·反哺课程·先人工后 AI)。

## §1 现状盘点（项目已有什么）

### 客服（已建·真机验过）
- **云端**：`cs/kfCallback`（验签解密 fail-closed → `sync_msg` 拉取去重 → `dispatch` 分流 → `send_msg`）+ `cs/kfBind`（unionid→external_userid 身份桥，rate-limited）+ 接缝单源 `kit/wecom.ts`（token 缓存/sync/send/transfer/idconvert/加解密）。集合 `kfState`（token/cursor/去重痕）、`kfIdentity`（external_userid→openid）。
- **机器人能力**（`cs/kfCallback/dispatch.ts`）：进会话欢迎菜单、4 类关键词识别（物流/激活码/售后/教程）、4 条 FAQ、小程序卡片、查订单（`summarizeOrders`）、转人工。
- **前端**：4 入口（详情坞 `DetailDock`/「我」页/售后页/播放页 `ServicePanel`）统一走 `utils/customerService.js` 的 `openCustomerService()`。
- **成本**：基础免费 + 企业微信认证 ¥300/年。
- **配置 runbook**：见 `微信客服配置手册.md`；客观接口/集合事实见 `系统事实.md §5.6`（本文不重抄）。
- **短板**：① 话术/FAQ 写死代码、无可视化后台（改话术要改 `dispatch.ts` 重部署）；② 仅关键词匹配、无真正 AI 理解；③ 转人工自动分配卡微信 48002 权限（现靠接待台手动接入）；④ 查订单卡「小程序绑微信开放平台」未完成。未完成项多为「靠人」运营配置（env 密钥/后台配接待人/开放平台绑定），非代码。

### 后台/数据资产（自建 360 工作台的底子）
- **Vue3 admin**（`packages/admin`，独立 Vite，共用云环境）：已有 Login/ProductList/Wizard/Showcase/**Orders**（列表/筛选/搜单/发货/批量）/**Refunds**（审批）/**Dashboard**（KPI/待办/漏斗/近况）/**Inventory**（CAS 库存）/**Reconciliation**（对账）/HelpVideos/Externals。API 经单 facade `api/cloud.js` → `POST /adminApi` 派发 43 个 action。
- **24 集合**（`kit/collections.ts`，`系统事实.md` 为单源）：360 视图可用——`users`(画像基底)/`orders`(订单+shipping{company,trackingNo})/`afterSales`/`activations`/`reviews`/`progress`(学习)/`events`(行为·90 天自动清)/`inventory`(SKU 库存)/`kfIdentity`(身份桥)。
- **可复用 CS helper**：`resolveOpenid(db, externalUserId)`（身份反查）、`summarizeOrders(db, openid)`（近 3 单状态+运单）——正是坐席「查客户订单」所需雏形。
- **ERP**：仅基础进销存（`kit/inventory.ts` CAS 防超卖 + 阈值），**无多仓/调拨/供应商/采购**。
- **多节点**：无。**T1「微信原生单源」已把发布目标收窄为只发微信小程序**（H5 仅内部预览），无 视频号/微信小店/公众号 接入。

### 并发视角（扛量分两层）
- **机器人层**：云函数天然弹性 + 现有去重（`seen:<msgid>`）+ 单批 50 条/15s 预算，消息洪峰本就能扛。
- **人工接待层**：真正的并发瓶颈——单人接待不可水平扩，需**多坐席 / 排队 / 工单 / 转人工自动分配**。现有转人工自动分配卡 48002（属「客户联系」权限域，企业内部接入密钥无此权限）。

## §2 真实需求（五层·2026-06-30 修正口径）

用户多轮澄清后的真需求，**不按「一人轻量运营」假设**，五层：
1. **统一后台 + 360 坐席工作台**：客服要能看**订单详情 / 物流情况 / 用户记录 / 用户画像**（CRM 级、与业务数据打通），不是孤立聊天框。
2. **协同 ERP 信息**：客服/后台对接 ERP（库存/订单/履约）。
3. **微信生态多节点**：搭建并打通**微信小店、视频号、小程序、H5 自有官网、企业微信/WorkBuddy、元宝(腾讯 Hunyuan)**，目标＝**最大程度利用微信生态**，客服与后台跨节点统一接住。
4. **承接大量客服并发**：容量前置（多坐席/排队/分流/工单）。
5. **AI 能力未拍板**：中立陈列、不替用户决定。

**与旧记录的重大差异（provenance）**：旧文档口径是「精品店初期免费内置、**重型能力上量后再议**」（决策§19）、「**从未压测**·一人小团队」（待办/运营规划）、「**只发微信小程序单源**」（T1）。本需求把客服**升格**为「统一后台 + 微信生态多节点 + 扛并发」体系的一维 → **T1 单源决策与「上量后再议」均需用户重新拍板**（本文只标冲突、不替决策）。

**代表产品**（定位·价位·2025-2026 公开口径，以官网为准）：
- **网易七鱼**：电商向、抗高并发。在线客服 ¥1888/3888/5888 /坐席/年；AI 机器人 ¥5888/28888/158888/年；SaaS 起 ¥19800/年（10 坐席+全渠道）。有开放 API（客户信息/订单查询/CRM 绑定）。👁⚡

## §5 微信生态多节点（如何最大化利用微信生态）

### 微信小店＝官方统一电商中枢
- 2024.08 起 视频号小店可升级为**微信小店**；2025-2026 已成微信统一电商面，一处铺货跨 **视频号 / 公众号 / 小程序 / 搜一搜 / 朋友圈**。
- 自带：**客服**（知识库·2026）、订单全生命周期、物流跟踪、**会员系统**（2026.04，会员等级/积分/权益，免开发）。
- **现有自建小程序可接入**：官方「**小程序连接小店**」API——展示小店商品卡、插优惠券、订单回调推到小程序、订单数据双向同步（一个小程序可绑多个小店；一个小店可绑至多 100 个小程序）。
- 取舍：用微信小店＝白得跨节点分发 + 内置客服/会员/物流；纯自建小程序则失去这层官方分发。**技术服务费按类目抽佣**（以官网为准）。
- **本项目定位＝分发渠道、非实物引擎（2026-06-30 爆破·文件实证）**：现状电商（`createOrder`/`pay` 走云开发支付工作流/库存 CAS/状态机/对账/发货合规）与学习（`activateCourse`/`qrcodes`/`courses`/`progress`）**全自建在 CloudBase、深度耦合**；核心商品是「材料包+激活码+课程」捆绑、激活链绑 openid，**课程/激活/视频体系装不进微信小店**。故小店只宜做「视频号/搜一搜 引流 + 卖纯实物/引流款」，**不迁已运转的自建引擎**（那是拆承重墙）。→ 数据天然双源，自建工作台使命＝双源统一 360（详 §6.1）。
- **平台税收口（Q2）**：碰微信的接缝（小店连接/微信客服/客户联系权限）交「有界平台税」（一次性配置 + 偶发跟微信规则变·权限按应用一次性授予非反复申请）；靠项目既有「平台接缝单点 `kit/wecom.ts` + `console-assets/` 正册」（根因#12）把反复重配面**结构性收成一个点**。能力主体跑自有 CloudBase＝微信治不到、近一劳永逸。

### 视频号 / 公众号 / 搜一搜 / 朋友圈
- **视频号**：直播/短视频挂商品 → 微信小店；视频号订单咨询经企业微信客服统一接待。

### 微信客服＝跨节点统一接待中枢（关键）
- **覆盖 6 入口**：视频号 / 小程序 / 公众号 / 搜一搜 / 支付凭证 / 企业客服微信——**同一套微信客服可同时接住多个微信节点的会话**（项目已用的正是这个独立产品）。

## §6 统一后台 + CRM/ERP 360 坐席工作台：构建 vs 采购

**360 数据源现状**：坐席要看的「订单/物流/画像/记录」**现有集合已覆盖大半**（§1）——`orders`(含 shipping)/`users`/`afterSales`/`activations`/`progress`/`events`，且有 `resolveOpenid`/`summarizeOrders` 雏形。缺的是**一个聚合查询 + 坐席搜索 + 角色权限**的工作台壳。

**三方案对照**：

| 维度 | A. 自建(现有 admin 上长) | B. 付费一线 API 喂数据(七鱼/Udesk WeiPeng) | C. 电商中台(有赞/微盟) |
|---|---|---|---|
| 工时 | 数周(壳+聚合+搜索+RBAC) | 中台 1-2 周对接 | 迁移 1-2 周 |
| 成本 | 基本设施 <¥100/月 | ¥5 万-10 万/年 + 设施 | 有赞 ¥1.38-2.68 万/年(+抽佣)·微盟 ¥3800 起+模块 |
| 360 视图 | 自建(数据现成) | 内置侧栏(喂数据后) | 内置 |
| 客服扛并发 | 仍用微信客服多坐席 | 自带工单/排队/多坐席 | 内置客服 |
| 灵活/定制 | 全自由(贴钩织业务) | 受厂商限 | 受厂商限 |
| 数据归属/迁移 | 自有·易导出 | 半锁定 | **锁死·难迁** |
| 与 T1/现有资产 | **最贴**(复用 admin) | 并存 | **冲突**(等于换中台) |

**SCRM 侧栏 360**（尘锋/微伴/微盛，¥600-1800/人/年）：在企业微信侧边栏给坐席看订单/物流/画像，与微信客服/企微原生整合度高；适合「私域 + 坐席 360」但同样要把自有数据喂进去。

**ERP 分档建议**（按月 GMV，2025-2026 行情）：
- **<¥10K/月、<50 SKU**：CloudBase 自建进销存（现有 `inventory`）足够。
- **¥10K-10 万/月、50-500 SKU**：轻量 SaaS ERP（万里牛 lite 等，¥600/月级）+ API 对接。
- **>¥10 万/月、500+ SKU**：聚水潭/旺店通等全 ERP 才划算。
- 「客服协同 ERP」典型含义＝坐席在会话里实时看库存/物流、审批退款联动 ERP 退货退款。

**落点提示（仅建议·本批不写代码）**：自建路线可加一个 `getCustomer360` admin action（组合查 `users`+近 N 单+物流+退款+激活+行为摘要）+ 坐席按 openid/手机号/订单号搜索 + `auditLog` 加 agent_id + RBAC 按角色过滤。

### §6.1 距 360 工作台多远（2026-06-30 爆破·文件实证）

**数据底座≈8 成已就位**（决策②自建因此是「拼装」非「造数」）：`getOrderDetail` 已联查 `activationFor`（逐商品激活/进课态）、`resolveOpenid`（身份桥）、`shipOrder`/`approveRefund`/`rejectRefund`/`recordAudit`（动作+审计）、Vue3 admin 壳 + 43 action。**真要新建四块**：

| 待建 | 量级 | 说明 |
|---|---|---|
| ① `getCustomer360` 聚合 | 小（~1-2 天） | 按 openid join 现成数据，无新造 |
| ② 客户检索 + `getUser` + 画像 rollup | 小-中 | 现仅按单号搜；`users` 连读取 action 都没有；画像（总消费/单数/激活率/最近活跃）待 rollup |
| ③ 坐席承面 A/C | 中-大 | **未定**：A 企微侧栏 / C 自建工作台 |
| ④ 坐席 RBAC | 中 | 现为**单超管口令**（`adminConfig` 一个 keyHash），无多账号/角色 |

**距离两层**：只读「**客户 360 查询页**」（坐席搜一下看全貌）≈**天级**（数据+动作全现成）；**完整坐席工作台**（带 RBAC + 贴聊天）≈**周级**（卡 RBAC 模型 + 承面 A/C）。

**两运营前置（靠人·根因#8）**：① `kfBind` 绑微信开放平台（链②·见下）；② 解 48002「客户联系」权限（转人工自动分流/多坐席排队）。

### §6.2 认人两链 + A/C 卡点区分（2026-06-30 爆破）

- **链① 会话→external_userid**：`kfCallback` 每条消息已带（`extUserId()` 在用·后端已验）。其 UI 绑定是 **A 专属疑虑**——企微侧栏在「微信客服接待」上下文能否吐 id **未真机验**（根因#8）；**C 自建工作台从 callback 收会话状态、天然知身份、无此卡点**。
- **链② external_userid→openid→订单/360/学习**：要 `kfBind`+绑开放平台。**A、C 共同命门、更根本**；未通则两承面都只能手动搜单号/手机、跳不了自动 360。
- **先验头道结论**：别先纠结 A/C，**先打通/先验链②**（绑开放平台 → 测试用户小程序登录写 `kfIdentity` → 发微信客服消息 → `resolveOpenid` 出 openid + `summarizeOrders` 出单）——它是 360 命门，且顺手解锁搁置的「查订单」。

### §6.3 学习上下文进 360（2026-06-30 爆破·Q7）

客服需看学员**当前课程节点位置**以指导钩织学习。**数据已现成且精到秒**：`progress`（每用户每课一条·`_id=openid__courseId`）的 `last{lessonId,segmentId,at,dur}`（视频看到第几秒）+ `done{segmentId:true}` + `updatedAt`，由 `trackEvent`（segment_done/watch_at）折叠写入，`last.at` 已被「继续观看」resume 用着＝真数据。**落点**：`getCustomer360` 加「学习位置」子面板，segmentId join `courses`（chapters→lessons→segments{name}）→「《课》第X章·第Y节『名』·看到 m:ss/共 m:ss·该段未完成·最近学习 N 分钟前·本课 a/b 段」。**成本小**；学习数据同按 openid → **搭同一条链②解锁、零新增微信依赖**。诚实区分：取「最后位置」（`progress.last`，扎实便宜）非「严格实时帧」（要 live 推流·不必要）。

## §6.5 目标动线（爆破推演·🟢现成 / 🟡待建 / 🔴运营前置）

一条横跨双源的实景（客人问「材料包没到 + 课程打不开」）走完整动线：

1. **客人进会话**（视频号/小程序/公众号任一入口）→ 落同一微信客服会话 🟢。
2. **机器人挡轻咨询** 🟢（`dispatch.ts`）：欢迎菜单 + `recognize()` 关键词高亮气泡 + FAQ + `summarizeOrders` 查单（🔴 需链②）。轻咨询到此消化、不占坐席（抗并发第一层）。
3. **转人工** 🟢（`transfer()`·🔴 自动分配卡 48002，现接待台手动认领）。
4. **坐席接手，360 自动浮出** 🟡：拿会话 `external_userid` → `resolveOpenid` → `getCustomer360`。一屏含：画像头（🟡 rollup）、订单+物流 🟢、**激活/课程态** 🟢（双源在此并屏）、售后 🟢、**学习位置/卡点** 🟡（§6.3）。
5. **判断 + 一键动作** 🟢：`shipOrder`/`approveRefund`/补运单…经 `recordAudit`（🟡 加 agent_id）。
6. **回复客人**、会话结束 🟢。
7. **并发/多节点** 🟢：多入口汇一接待池，多坐席各自认领（🔴 48002）、侧栏各按会话加载 360、水平扩。

**承面差异**：A 企微侧栏＝借企微聊天 + 自建侧栏（卡链①侧栏取 id·未验）；C 自建工作台＝聊天+360 一屏全自建（无侧栏卡点·但聊天 UI/实时全自建）。**承面已定 A**（Q8：客户联系侧栏 `getCurExternalContact` 标准能力·卡点解除）。

## §6.6 双轨 + 按意图分流 + 合规卡点（Q8/Q9·已拍板）

**双轨 + 意图分流**：点客服按意图走两路——
- **订单/售后类**→ 微信客服（`openCustomerServiceChat`）+ 机器人(`dispatch.ts`)+360 自动答、必要转人工。✅ 全程现成、项目已过审。
- **钩织学习求助类**→ 课程/播放页入口「向讲师请教」→ **官方「联系我」插件加企微好友** → VIP 客服在企微侧栏看 360+学习位置+（Q10）节点照片精准指导 → 沉淀（标签/群/复购）。

**沉淀 vs 接待的本质**：微信客服＝接待（不"拥有"联系人）；加企微好友＝外部联系人（可主动再触达/拉群/朋友圈）＝真"沉淀"。意图分流把高沉淀价值的「学习求助」精准投去企微，不在「问快递」的人身上浪费获客。

**「加企微」机制（已查验）**：无"一点直达"API；走「联系我」插件 `plugin://contactPlugin` 或获客链接 scheme→**跳出小程序→用户手动确认「添加」**（不能静默·微信全局如此）。**用插件、不用 scheme**（scheme 有诱导跳转+外链限制风险）。`idconvert`(unionid↔external_userid) 适用、前置同链②。获客助手计费/小程序集成排期未定（待问销售）→ 先用免费的「联系我」活码。

**合规卡点地图（已查验）**：

| 步骤 | 判定 |
|---|---|
| Flow A 订单类全程 | ✅ 全安全（微信客服标准用法·已过审） |
| Flow B 入口话术 | ⚠️ **诱导红线**：用功能化描述、**不**用「VIP专属/加我领」钓饵（已拍板话术功能化） |
| 加企微机制 | ⚙️ 用「联系我」插件（预审过）·不用 scheme |
| 360/照片 数据给客服看 | 📋 隐私协议**声明**数据共享 + 授权 + 可撤回（有 `privacy-authorize-wired` 底子） |
| 私域主动触达 | ⚠️ 别高频/别"专属优惠仅限好友"·留退订 |
| 课程类目资质（流程外） | 📋 **待用户核**（钩织兴趣类·门槛通常低·项目已生产或已处理） |

**结论**：合规不改变策略、只约束「怎么呈现」——插件 + 功能化话术 + 隐私声明，均可做到。

## §6.7 关键节点拍照留底 + 二分诊断（Q10·功能候选）

**做什么**：教学视频在关键节点引导用户拍钩织成果留底；求助时客服**逐节点二分查证**（看该节点照片→出问题给挽回办法→否则进下一节点）。

**要新建**：关键节点定义 +「正常样子/常见问题/挽回办法」内容策展（admin·每课一份）、用户侧节点拍照上传 + 新集合 `checkpoints`（`known-collections-only` 登记）、360 面板扩展、⚠️ **`imgSecCheck` 内容安全检测（全仓现零·UGC 图片上线微信硬要求·必加新合规点）**。

## §7 寓言：哪条路最合适，为什么

> **小棉鸭的钩织小铺要扩张了。她想要的不再只是「门口有人答话」，而是一个能接住四面八方客人、又能随时翻出客人底账的前台。她面前三条路。**
>
> **第一条，请一支一线豪华礼宾团**（付费 SaaS / 有赞·微盟中台）。西装革履、自带叫号机和工单本、人多也不乱，侧台还能立刻调出客人的订单和画像。可这团队站在**自家铺子之外**——要把自家的账本一页页誊给他们看（API 喂数据），月月付的钱也不便宜；更要命的是，若哪天想换团队，账本被他们攥着、誊不回来（数据锁死）。
>
> **第二条，自己挖井、自己修水管、自己造一套前台系统**（自建开源客服）。听着免费又自由，可水管天天漏要她一个人爬上爬下；最尴尬的是——**这些水管接头压根拧不上微信的水龙头**（开源客服几乎都不接微信小程序），想引微信各扇门里的客人进来，还得自己焊一截转接管。
>
> **第三条，把自家已经盘好的灶和账台打通，并在微信的每一扇门口都设一个迎客的**。小棉鸭忽然想起：她家**灶早砌好了**（微信客服已建好、真机验过），**账台也现成**（admin + 24 本账册：订单、物流、会员、行为），只是灶和账台之间还没修一条短廊（一个 `getCustomer360` 聚合），迎客的也还只守着小程序一扇门。她要做的，是把短廊修通（坐席一眼看全客人底账），再把迎客的请到微信的**每扇门口**——视频号、公众号、搜一搜，甚至**直接在微信官方的大集市租个摊位**（微信小店），让一处上架、各门通卖，而微信客服本就能同时接住这许多扇门的客人。人潮真涌来时，她再多请几位迎客的站到队列里（多坐席/排队），灶台和账台纹丝不动。
>
> **至于要不要给前台配个「会听人话、又会自己翻账」的帮手**（AI：元宝、Claude、或自建知识库）——小棉鸭还没拿定主意。帮手能省力，但请哪一个、要不要请，她想再想想。本文不替她拍这个板。
>
> **三条路没有绝对优劣**：客流真的要爆、且不在乎把账本交出去，礼宾团最省心；要绝对自由又不缺运维的手，自挖井也行；但**对一个已经砌好灶、备好账台、且整个生意都长在微信里的小铺**，把现成的打通、再贴着微信生态铺开，通常是花得最少、退路最宽的一条。最终选哪条、AI 请不请、要不要推翻「只守小程序一扇门」的旧约（T1），都由小棉鸭自己定。

## §8 AI 选项（中立陈列·未拍板）

> **明确：AI 能力尚未拍板。本节只陈列利弊，不下推荐、不替用户决定。**

| 路线 | 利 | 弊 / 代价 |
|---|---|---|
| 元宝 / 腾讯 Hunyuan API | 腾讯生态原生·与微信/云亲和·新号 100 万 tokens/月免费 | 能力/生态绑腾讯·需评效果 |
| Claude API 直连 | 强·最轻·无新基建·贴现有云函数(在 `dispatch.ts` 关键词兜底处调用) | 跨境调用/合规与成本需评 |
| FastGPT / Dify 自建 | 知识库 RAG 强·可控 | 要自建并运维一套服务 |
| 腾讯 WorkBuddy | 自动化 agent·可接 WeCom/微信 | 偏企业自动化·小铺可能过重 |

**共性影响**：AI 主要改善「答错兜底质量」（从关键词升级为真听懂）+ 削机器人层重复咨询（间接助扛并发）+ 知识库自答。无论选哪条，都建议保留「AI 答不准 → 转人工」兜底。**选不选、选哪条，留待用户拍板。**
