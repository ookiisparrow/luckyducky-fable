### 2026-06-11 · 控制台传图失败 UPLOAD_FAIL（HTTP 服务请求体 ~100KB 上限,原 F）　✅ 已结案
- **现象**:上传真实产品图报 UPLOAD_FAIL;此前 AI 的 E2E 用 1×1 测试图通过。实测 50KB 过/200KB 拒(`EXCEED_MAX_PAYLOAD_SIZE`),真实照片 base64 200–700KB 必超。
- **教训**:① **E2E 样本要贴近真实数据尺寸**——最小样本只验逻辑不验边界;② 同通道响应体也有上限,listDrafts 商品多后需分页;③ 网关层错误与业务错误形状不同,适配层必须兜底。
- **状态**:✅ 分片上传修复(`uploadChunk`/`uploadFinish`,~80K/片)已部署;用户随后用真实封面图上架成功、验收单 M 通过 = 复验关账(2026-06-11)。

### 2026-06-11 · 控制台传视频「直传失败 HTTP 403」（直传凭证只认 POST 表单,原 G）　✅ 已结案
- **现象**:第④步上传 mp4 报 403 终止。根因:`getUploadMetadata` 签发的凭证按微信 HTTP API 约定只认 **POST 表单上传**(签名串含请求方法),前端用 `PUT + Authorization 头` → 签名不符被拒;直传路径此前从未被真实走通(E2E 只验到取凭证成功)。
- **教训**:① 凭证类接口要按签发方约定的**完整请求形状**使用(方法/字段都参与签名,不能只对上 token);② 「双保险」要覆盖主路径的**全部失败点**,只兜第一步等于没兜;③ E2E 过了「拿到凭证」≠ 过了「用凭证」(与上条"最小样本只验逻辑"同根)。
- **状态**:✅ 直传改 POST 表单 + 上传失败自动回落分片(PR #52)并重发线上控制台;用户重传成功——云端小熊课程已带真实视频段并发布 = 复验关账(2026-06-11)。
