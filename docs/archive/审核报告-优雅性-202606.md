## A. 云函数

### A-1（P3，✅ 已清理）adminApi 的 `str` 截断助手重复定义 4 次
- **问题**：`cleanProduct` / `cleanCourse` / `saveCard` / `saveHomeContent` 各自在函数内定义一份 `const str = (v,cap)=>...`，同一文件 4 份完全相同。`trackEvent` 同款助手已是模块顶层（正确范例）。
- **清理**：提为 adminApi 模块顶层一份，删 4 处副本；34 处调用不变，`node -c` 语法通过。零行为变化。
- **代价**：需重新部署 adminApi（纯整洁化、无紧迫，方便时跑）。

### A-2（可接受，记录）openid / admin 闸样板重复
- `const {OPENID}=cloud.getWXContext(); if(!OPENID) return {ok:false,error:'NO_OPENID'}` 在 10 个函数重复；admin 闸（须 `users.isAdmin`）在 4 个重复。
- **为何不抽**：微信云开发**每个云函数独立打包部署、无共享运行时**。要共享得把助手作为 npm 包或源文件**复制进每个函数目录**——徒增构建/同步复杂度，收益（省几行）不抵成本。**判定：保留是正确取舍**，不是债。
- 缓解：样板极短、语义单一、各函数顶部一眼可辨，复制风险低。

### A-3（可接受，记录）createOrder ↔ api/order.js 金额/订单号镜像
- 云端 `createOrder`（定价权威）与客户端 `api/order.js`（H5/App 回退）各有一份订单号生成 + 金额计算，逻辑镜像。
- **为何不抽**：跨「云函数 / 客户端」部署边界，物理上无法共享同一模块；客户端那份本就是**演示用回退**，且注释已标注「与云函数同规则」。**判定：有意镜像**，可接受。

## B. 管理控制台（admin/）

### B-1（P2，✅ 已清理）防抖自动保存三件套散三处
- **问题**：`watch(deep) + setTimeout 防抖 + saveState 状态机 + onBeforeUnmount flush` 在 `StepVideos` / `StepCards` / `Showcase` 各抄一份。上一批安全审给三处**逐个手补 flush**，正是「同一逻辑散落、改一处易漏」的实证。
- **清理**：抽 `admin/src/composables/useAutosave.js`（`useAutosave(source, saveFn, {delay, ready, afterSave})` → `{saveState}`，内含 flush），参照小程序侧 `src/composables/useTimers.js` 范式。`StepVideos`（delay 800 + afterSave 写 videoStats）、`StepCards`（delay 700 + afterSave 写 cardStatus）改用之，删内联 timer/watch/flush。行为完全保持，admin build 通过。
- **未纳入 Showcase 的原因**：它是**两条保存路径（排序+上下架 命令式触发 / 首页文案 深监听）共享一个 saveState**——形状与单源单存不同，强套封装反而更绕。保留原样，已在 `useAutosave.js` 注释说明。收益仍达成：今后 flush 修复对接入页统一生效，同类 bug 不再各漏一遍。

### B-2（P3 → 技术债）三个大文件
- `StepVideos.vue`(583) / `Showcase.vue`(576) / `StepCards.vue`(509) 偏大。但都**功能密集**：三层树编辑 + 拖拽 + 视频直传 / 一比一手机预览 + 双向拖拽 + 内容编辑 / 双面 SVG 卡片设计器 + 实时预览。拆分属判断题（拆预览组件 / 拖拽逻辑），非机械整洁化，有回归风险。**记技术债，待触碰时顺手拆**。

### B-3（正向确认）admin 结构清晰
- `api/cloud.js` 单一适配层（换后端只改这里、双模式）、`store/products.js` 收口、按步骤分文件、CSS token 化（tokens.css）——结构良好。
