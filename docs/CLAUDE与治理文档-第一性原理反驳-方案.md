# CLAUDE.md 与治理文档——第一性原理反驳报告（待拍板）

> 2026-07-23 立。任务：对 CLAUDE.md 与治理方案文档做第一性原理的深度反驳，目标是**提高未来 AI 工作时的注意力，消灭无用信息、低效信息**。本报告只分析不动手：每条指控附可核证据，改法标注守卫耦合面，拍板后走 `/refactor-batch` 落地。

## 0. 审视框架：上下文经济学（第一性原理推导）

CLAUDE.md 的物理本质：**每个 AI 会话开局强制注入的 ~24KB（约 1.2–1.5 万 token）**。它的每一行有两个成本：① token 本身；② 注意力稀释——AI 的注意力对上下文近似零和，读了墓碑就少读代码。每一行只有一种价值：**改变本会话行为的概率 × 改变的价值**。

由此推出信息四分类与各自的正确归宿：

| 类 | 定义 | 正确归宿 | 成本 |
|---|---|---|---|
| ① 闸外约束 | 机器守卫能拦的 | check-structure / hook / 测试 | **零上下文成本**（最优） |
| ② 行前必读 | 不先知道就会白干一轮的（危险性、设计期取舍、靠人判断题） | CLAUDE.md 正文 | 每会话付费，值得 |
| ③ 按需知识 | 特定任务才需要的 | 独立文档 + 一行指针 | 近零 |
| ④ 历史叙事 / 墓碑 / 自证 | 解释过去、防御已消失的风险、向读者证明自己没错 | **应清零**（git 历史即归档层——体系自己的话） | 每会话纯损耗 |

**判据（准入测试）**：删掉这行，一个典型会话会变差吗？——已有编辑 hook 当场反馈违例的规则，答案通常是「不会」。

## 1. 总反驳：体系把「防漂移」当最高价值，但真正的稀缺品是「注意力」

这套体系最深刻的洞见是它自己写下的两句话：

- 「靠人记的迟早漂移，靠机器守的不会」（元模式 A2）——守卫的本质是**把约束搬出上下文窗口**，是注意力的外包。
- 「每轮进回路的上下文要瘦」（元模式 A6 杠杆②）——静态层应是薄的单源。

**反驳的核心：这两条原则没有对体系自身的核心文书执行。**已被机器焊死的规则，CLAUDE.md 仍在为其支付 prose——这是对同一条约束的**重复支付**：机器付一次（零上下文成本），prose 每会话再付一次。守卫存在的全部意义就是让 prose 可以退到一句话。

自曝的例子：§3 明文「机制在守卫里、本文不复述」，§4 频控条却复述 `throttleLocked`/`throttleFail`/`throttleReset` 三个函数名和调用顺序——机制细节正是守卫 `rw-admin-login-throttled` 焊死的东西。写新代码的 AI 被闸拦下时会得到处方级报错，prose 里的函数名从未被谁需要过。

下面按严重度逐项反驳，每条附证据。

## 2. 逐项反驳

### 2.1 CLAUDE.md 身份段：历史叙事冒充操作约定 【P1·最大单块浪费】

身份段（前 ~20 行）里，未来会话真正需要的操作事实只有五个：

1. 本仓是生产源，部署/建表/删表**真实生效**；
2. 活代码在 `rewrite/`；
3. 旧线参照 = next 仓（本仓不留）；
4. 8 件控制台资产 git 外、`console-assets/` 为权威源；
5. AppID / 云环境 ID。

其余 ~60% 是**切换史与墓碑**：「2026-07-03 从 luckyducky-next HEAD ff02afb 完整复制」「2026-07-09 M5 切换执行完毕」「无真实用户前提下用户拍板压缩 runbook 前置闸」「旧线 26 函数与 /admin2 /agent2 并行路径已清退」「packages/ 已于 2026-07-23 瘦身拍板批删除」「线上旧版 v0.9.4 调用的旧函数已删」「（生产线史工作日志已冻结入 git 史料层）」「design/ 与旧线 packages/ 已删·git 历史可查」……

这直接违反体系自家的文档规则③「**正册只放仍会被查的内容，过期即删，不留考古层**」。墓碑的辩护是「防 AI 复活旧线」——但该风险的机器面已有 `rw-toolchain-no-oldline` 守着，且随 packages/ 物理删除，风险本体已消失：没有东西可以复活。墓碑的防御价值随时间指数衰减，token 成本永恒不变。

**改法**：身份段压缩到上述五事实（约 6 行）；切换史一句话指针「切换/瘦身经过见 `重构日志`」。**建议立「墓碑 TTL」元规则**：任何「已删/已清退/已退役」防御性注记，要么配守卫要么登记到期删除——见 §5。

### 2.2 计数手抄违反自家规则⑥ 【P2·实锤三处】

CLAUDE.md 规则⑥：「客观事实（接口/集合/数据/**计数**）只在系统事实机器维护、**别处引用不手抄**（防 stale）」。而「17 云函数」在 CLAUDE.md 手写了 **3 处**（L5、L8、L58），每处旁边还挂着「函数数以 `rewrite/cloud/build.mjs` 产物为准」的免责——既然以产物为准，写数字就纯粹是在制造 stale 面。

这不是理论风险：`系统事实.md` 已出现 17/18 口径分裂（17 产物 + 1 部署别名 adminApiV2）；`待办与债.md` 正躺着「17→14 拓扑收编提案」待拍板——一旦拍板，CLAUDE.md 三处全部变错。`guard-count-synced` backstop 只在**数字错时**红，防不了「数字还对着的 stale 预备役」，更防不了口径分裂。

**改法**：三处全部去数字，只留「拓扑以 build.mjs 产物为准」。同理排查身份段的「8 件控制台资产」（该数有 `console-assets-present` 守着，属守卫锚，可留）。

### 2.3 §9 skills 注册表：与自动注入的描述双源，每会话双倍付费 【P1·最大纯冗余块】

Claude Code 的 harness **每会话自动把全部 SKILL.md 的 description（含触发词）注入上下文**——这已经是 13 个 skill 的完整可发现性机制。CLAUDE.md §9 又用 ~25 行把每个 skill 的用途、区别、组合关系复述一遍，其中大量与 SKILL.md description 逐句雷同（如 patrol 的「/anti-entropy-loop 的 /loop 降档版」两处都有）。

同一信息每会话进入上下文**两次**，这是全仓最贵的一块纯冗余。§9 里唯一不可替代的内容是一条纪律：「**本仓任何改动都走 /refactor-batch**」。

**守卫耦合**：`skills-referenced-exist` 解析 CLAUDE §9 的 `/skill-name` 引用防悬空指针——瘦身 §9 须同步调整该守卫扫描面（改为扫 `.claude/skills/` 目录自身即可，且更根本：目录才是单源，CLAUDE 列表本来就是手抄镜像）。

**改法**：§9 缩为 3 行——改动纪律一句 + 裸名清单一行（或干脆删清单，守卫改扫目录）。预计净省 ~20 行 × 每会话。

### 2.4 §1 文档地图：自相矛盾 + 嵌套括号 prose 【P2】

- **活矛盾**：规则②「卷档机制暂停、待重新设计（2026-07-05 拍板，决策§25）」与同节「**退役制**（2026-07-23 第一性原理批**改制**）」并存。退役制就是重新设计的结果，②是没清走的尸体——同一文件内两段规则打架，AI 读到只能猜哪条现行。
- **同一规则两处两个置信度**：「条目 ≤8 行」在 §1 规则① 是硬规则口吻，在 §4「文档防膨胀」又降为「写作纪律（靠人自律，无机器守卫）」——双源且互相削弱。
- 地图本体是嵌套括号长 prose（「重构日志.md（批次账本，里程碑记这）· …（生产线史工作日志已冻结入 git 史料层）」），找「X 记在哪」要人肉解析整段。

**改法**：删规则②（或改写为「退役制即卷档机制的替代设计」一句）；≤8 行只在一处出现；地图改两列表格（文件 → 一句话职责），描述里的历史括号全删。

### 2.5 §11 失效指针：「云开发 → 文档梳理」 【P3·但暴露病根#16 活标本】

CLAUDE.md L148 路由表写「云开发 → 文档梳理」——`docs/` 下不存在此文档，现名 `云开发参考.md`。守卫 `doc-refs-exist` 没抓住，因为它只扫「`docs/`+实名」格式的引用，**裸中文名引用逃逸扫描面**——这正是病根#16「守卫绑定漂移：扫描面够不着，空样本＝绿」在治理文书层的活标本。

**改法**：修文本之外，把「文档引用必须用可 resolve 的 `docs/`+实名格式」立为写作纪律（守卫已在，只是引用姿势要喂进它的扫描面）。

### 2.6 元模式 §B 绑定点整段失效 【P2·「单源」自身漂了】

元模式 L72（§B「文档退役-唤起」绑定点）：「`docs/archive/`（退役层）+ 史料「README.md」（唤起索引）+ 守卫 `archive-index-synced`」——三个绑定**全部失效**：`docs/archive/` 已于 07-23 撤出工作区（`ls` 确认不存在）；索引已改名 `docs/史料索引.md`；`archive-index-synced` 已不在守卫注册表（grep 仅存注释残留）。07-23 两个大批改完，自称 canonical 单源的元模式没有回写。

讽刺点在于：元模式开篇宣称「都引用本文，不再各自复述——复述即病根#5」，但**单源模式的前提是单源自身必须新鲜**；单源 stale 时，所有引用它的地方一起错，比复制漂移更糟（复制至少有一份可能是对的）。

**改法**：L72 重写为现行退役制（史料索引 + tag 锚 + `doc-refs-exist`）。**元规则**：凡大批改动治理机制，「回写元模式 §B」应进 `/refactor-batch` 的记账清单——治理文书是每会话被读的，它们 stale 的伤害半径最大。

### 2.7 AGENTS.md：自称只路由，实际复述；且漏接判例机器 【P3】

- 开篇「本文件只做路由，不复述，防双源漂移」，随后「铁律」节复述了五条 CLAUDE 已有约定（check 全绿、不碰 check 脚本、走 refactor-batch……）——又一处声明与行为不一致。
- 更实质的缺口：「动手前查判例」指向 `关键决策记录`/`待办与债`/`根因账本` 三份原文，**唯独没指 `docs/判例索引.json`**——专门为「动手前查判例」建的 grep 机器，进场手册没接上。工头手册 §二 倒是接了。双源路由各自漂移的实证。

**改法**：铁律节压缩为差异项（AI 车队特有的三条：验收语言=check、不碰裁判、部署是人工动作），其余一句指 CLAUDE；判例入口统一指 `判例索引.json`。

### 2.8 工头手册 vs CLAUDE §8：两份活约定对 git 流程给出相反指令 【P2】

- CLAUDE §8：「每批一个 squash 提交**直接上 main**（不走异步 PR）」。
- 工头手册 §五：「GitHub main 分支保护：**必须走 PR**、CI 绿才可合并」。

正面冲突，AI 读到只能二选一猜。且工头手册 §五 的老板设置清单 checkbox 全部未勾——多模型车队制度（考核、发合并账号、施工单派活）没有启用证据，属于自家 §7 明令的「**别为不存在的需求建机制**」：为尚未存在的组织形态维护了一份 48 行的活职位手册，还被 CLAUDE §1 列为「治理工具」每会话广播。

**改法**：拍板二选一——(a) 车队制度真要上：先解决与 §8 的流程冲突，勾掉 §五 清单；(b) 暂不上：整册降级为待拍板方案或退役入史料，CLAUDE §1 撤下指针。

### 2.9 根因账本：病史段 = 死指针考古，约 40% 篇幅 【P2】

17 条病根每条带「病史」段，密集引用「债#14」「日志 I」「K-③」「定向 D-1」「优雅性 A-1」——这些缩写指向**已退役入 git 史料层的册子**，未来 AI 不做 `git show museum-20260723:...` 根本解不开；解开了对当前行为也无增益。病根的行为承载全在「本质 / 根治 / 绝迹证明」三段；病史的真实功能只有一个——「这条守卫为什么不许删」的证据链，而这用**一行**足够（如：#1 病史：五案并发重复写，最险为退款双执行·详见史料）。

`guard-coverage` 只解析 §一 的病根编号与守卫 roots，不读病史正文——压缩无守卫耦合。预计 158 行 → ~95 行。

**改法**：每条病史压为一行 + 史料指针；「§四 完整性自查」的人工对照表（L154–158，全是旧线日志字母映射）已被 guard-coverage 机器化取代，自己都标注「降级为可读复核」——按退役制应整段删除。

### 2.10 系统事实.md L40：单行 1500+ 字符，事实被过程账淹没 【P2】

自称「客观事实总源：机器维护」的册子，其 admin action 清单行把批次号、日期、闸门语义、翻案史全部内联成一条括号链（「+ 2 净新增 saveSecureConfig/savePayConfig·2026-07-12 人工配置清单可填写化·敏感凭证/支付接缝配置入库单源·未登记 ACTION_CAPS 默认仅超管·写类自动审计·payload 嵌套防审计日志泄露 + …」×20 段）。守卫 `admin-action-count-synced` 只核计数与名单——括号里的全部叙事是无守卫 prose，正是「该退没退的过程账」。L26–L34 函数表同病（每格塞部署史）。

**改法**：清单行只留名字；每 action 的闸门/cap 语义本就在代码与 ACTION_CAPS 单源里；「净新增于哪批」沉重构日志。事实册应当无聊——无聊才说明没混入叙事。

### 2.11 现状与路线「当前阶段」：入口文档需要考古式阅读 【P3】

CLAUDE 承诺「当前完成度与下一步只看现状与路线」，但该文件 §当前阶段 是 9 行无分段长文，混合四个「已落」批次叙事 + 8 项下一步（其中 ① 还是删除线划掉的）。新会话为了回答「现在该干什么」，要先读完三段历史。

**改法**：三行现状（阶段一句话 + 距上线还差什么）+ 纯未完成清单；「已落」叙事全部沉重构日志（那本来就是它家）。

### 2.12 巡逻日志：8 轮里 7 轮「0 findings 干净」，且停摆 18 天无信号 【P3·双重教训】

- 按自家规则「镜头轮换状态 = **最新一条**」，前 7 条（且全是 07-05 同日、多数零发现）是纯过程账，按退役制应删——治理器官自己没执行退役制。
- 更深一层：patrol 自 07-05 停摆至今（18 天）**无任何信号**——「持续义务有传感器」（病根#17）没覆盖到巡逻本身。要么给 patrol 配心跳（超 N 天未跑即红/提醒），要么承认它是按需工具、从「自动节拍」的宣传里降级——现状是宣传了一个不在跑的自动机制，AI 读了会形成错误的系统模型。

### 2.13 判例索引：概念正确，死判例稀释命中 【P3】

「grep 一处命中全部判例」的设计是对的。但 979 行里，决策 §1–§8（uni-app/JS 不用 TS/px 不用 rpx/Sass/MediaSlot/systemBar…）全是**旧线判例**，旧线已物理删除，多数永不再被问；只有 §3 标了 superseded。死条目的成本不是行数，是**污染 grep 命中**——搜「样式」「框架」先撞上一堆已死拍板。另有关键词农场倾向（根因#17 配了 32 个关键词，含「减肥」「越长越大」「仓库变大」）——同义词有边际收益，但每个关键词也是一次误命中机会。

**改法**：旧线判例条目标 `"status": "oldline-dead"` 或移出（`precedent-index-synced` 计数守卫需同步）；关键词以「别人真会用的说法」为限，砍拟人化长尾。

## 3. 钢人：这套体系做对了什么（反驳的边界）

反驳≠推翻。以下资产是真金，砍 prose 时一根手指都不能碰：

- **守卫体系本身**：110+ repoCheck、guard-coverage 元闭环、金丝雀夹具、基线棘轮——17 条病根全是真钱真事故换的。本报告的立场恰恰是「守卫越硬，prose 越该薄」——是对守卫的**更彻底信任**，不是怀疑。
- **单源纪律、退役制、判例索引概念、「验证不是相信」**——全部保留；本报告指出的只是「原则未自用」。
- **改动前必查守卫耦合**：CLAUDE.md 不是纯 prose，它是三个守卫的解析对象——`guard-coverage` 读 `[机器守:id]`（16 处）与 `[靠人:#N]` 锚（6 处）、`skills-referenced-exist` 读 §9、`doc-refs-exist` 读 `docs/`+实名引用。**任何瘦身批必须保锚或同步改守卫**，否则闸红。这也再次证明正确顺序是「先守卫后 prose」：锚在，围绕锚的叙事都可删。

## 5. 喂回体系的元规则建议（按体系自己的语法：痛 → 不变量 → 守卫/靠人）

1. **准入测试成文**（痛=本报告 §2 全部）：CLAUDE.md 每行须过「删了这行，典型会话会变差吗」；已有守卫的规则只许留「不变量一句 + 锚」。`[靠人——「会不会变差」是判断题；执行节拍挂 /doc-audit]`
2. **墓碑 TTL**（痛=2.1）：防御性「已删/已清退」注记须 (a) 配守卫，或 (b) 登记到期日，到期即删。不变量：「活文档中无超期墓碑」。`[可半机器化：墓碑句式（已删/已清退/已退役·git 历史可查）可正则扫描出候选清单供 /doc-audit 亲核]`
3. **治理文书优先体检**（痛=2.5/2.6）：每会话被读的文书（CLAUDE/元模式/AGENTS），其 stale 伤害半径最大，/doc-audit 应将其列为第一优先扫描对象；大批改动治理机制后「回写元模式 §B」进 /refactor-batch 收尾清单。
4. **计数禁令扩面**（痛=2.2）：受守卫维护的计数（函数数/action 数/守卫数）在系统事实之外的一切文档一律以「以 X 为准」指代，不写数字——`guard-count-synced` 防错数，本规则防「还没错的 stale 预备役」。`[靠人写作纪律，backstop 已在]`
5. **裸名引用禁令**（痛=2.5）：文档引用文档一律「`docs/`+实名」可 resolve 格式，禁裸中文名——让既有 `doc-refs-exist` 的扫描面够得着。

## 6. 一句话总结

这套治理体系的第一性原理是对的——「靠机器不靠人记」；它输给了自己的推论：**机器守住之后，prose 就该退场**。当前的 CLAUDE.md 与治理文书群同时保留了机器和 prose 两份支付，外加第三份「历史叙事」的纯损耗；三处已核实的失效指针（文档梳理 / 元模式 §B / AGENTS 判例缺口）证明厚 prose 不但贵，而且自己也在漂。瘦身不是删治理，是**把治理彻底交给它自己发明的机器**。
