## 2026-06-18 · 📦 发货合规 #26 代码侧自动化：shipOrder 接微信 upload_shipping_info（休眠态·fail-soft·根因#12）

**痛（合规债#26·🔴上线必做的代码侧半边）**：微信「实物 + 微信支付」小程序支付后须上传发货信息，否则**订单资金冻结/无法结算** + 后台反复弹「待接入发货管理」。原 `shipOrder` 只把 company/trackingNo 写自己 `orders` 表、**没调微信发货接口**＝合规缺口（测试单没事，真顾客订单的钱会被锁）。
- **关键事实**：本仓是**微信云开发**云函数（wx-server-sdk），调微信发货走「云调用」`cloud.openapi`——access_token 由云环境注入、**无需 AppSecret**（区别 kit/wecom 企业微信走 qyapi 须自管 token）。故自动化无外部密钥阻碍，能纯代码就位。
- **接缝（根因#12 平台接缝单点）**：新建 `kit/shipping.ts` 的 `uploadShippingToWx`（云调用上传·收口微信发货接缝一处）+ `wxDeliveryCode`（物流公司中文名→微信编码映射·未命中 OTHER·与快递100 com 编码不同套）。`shipOrder` 本地翻 shipped 成功**后**追调上传。
- **fail-soft（钱链不反噬·根因#3 边界）**：上传失败绝不回滚本地发货——留痕 `wxShipUploaded/wxShipError` + 打 `[LD_ALERT] sev=money` 告警人工补录，发货本身照成（return `{ok:true, wxShip}`）。缺 transactionId 直接不打微信、留 `NO_TRANSACTION_ID` 痕。
- **守卫（先红后绿·反向自检×3 全咬）**：① 结构 `shipping-info-uploaded-to-wx`（roots#12）＝上传调用 `uploadShippingInfo` 收口 kit/shipping.ts 单点（别处直调即红）+ shipOrder 成功路径须**调** `uploadShippingToWx(`（收紧到带括号·防只 import 不调假绿——反向自检首轮暴露并修）；② 行为 `shipping-upload-fail-soft`（roots#12·test）＝ `tests/cloud/shipOrder.test.js` 锁上传失败本地仍 shipped + 留痕 + 告警。反向自检：去 shipOrder 调用→红 / 别处直调上传→红 / 删告警→行为测试红，还原均绿。

## 2026-06-18 · 🔄 优化批0618·T-F3：数据页下拉刷新（parallel-b·根因#8 双机制）

**痛**：数据页无下拉刷新——后台流转（订单付款/发货、退款状态、新评价）后用户只能退页重进才看到最新。
- **关键多端坑（根因#8·分两机制非一刀切）**：`onPullDownRefresh` 只由**页面级滚动**触发；内容装在整页 `<scroll-view>` 里滚动的页（index/me），页面本身不滚→页面级 `enablePullDownRefresh` 在 mp-weixin **根本不触发**（H5 还假绿）。故：
  - **页面级滚动数据页**（catalog/order-list/reviews/aftersales/order）→ `pages.json enablePullDownRefresh` + `onPullDownRefresh` 强刷 store + `uni.stopPullDownRefresh()` 收尾；
