## 2026-07-20 · 批P 拖动彻底失效且无震动（量轨一次不中即永久放弃·根因#8）

**用户第二轮反馈**：批O 后「进度条依然失效，并且缺失震动反馈」。**「没有震动」是决定性线索**——震感写在 `onSeekTick`（WXS 过秒回桥）里，它不响＝WXS 一次都没回桥＝拖动逻辑压根没被激活，不是「跳错位置」而是「完全没启动」。批O 那两条（落位/续播兜底）修的是提交之后的事，够不着这里。

- **根因**：`_seekRect`（进度条量轨）只在 `playSegment` 的 `state:'playing'` setData 回调里量一次（`if (!this._seekRect)`）。而该回调触发时 `.lp-seek` 常常尚未完成布局提交 → `boundingClientRect` 回 `width:0`。代码正确地拒绝缓存坏值（`if (r.width > 0)`，注释里写明了这个坑），但**没有任何重量路径**：`_seekRect` 就此永久为 null → `buildSeekCfg` 恒回 null → `seekCfg` 永不下发 → `seek.wxs` 的 `st.rect` 恒 null → `onStart` 首行直接 return。四个现象一并解释：无位移、无回桥、无震动、无 seek；且只有换段才可能碰运气量到（与历史「有时要切下一集才能加载」同一族）。
- **修**：`onTimeUpdate` 内 `if (!this._seekRect) this.measureSeekRect()` —— 播放推进是天然节拍，未量得就再试，布局提交后必然量到即自行停止（无定时器、无新状态）。置于 `seeking`/节流两道早退**之前**：拖动死掉时 `seeking` 恒 false，放在早退之后会自锁。
- **守卫**：折进 `rw-mp-player-immersive`（第三条 seek 类断言）——`onTimeUpdate` 方法体内须出现 `measureSeekRect(`。
- **验证**：`npm run check` 全绿（208 repoCheck + 10 fileRule·246 文件 2262 例）。**反向自检**：删 `onTimeUpdate` 那处重量 → 红 → 对称还原 → 绿。
- **自检事故（已记错题本 E21）**：首次反向自检用整行做锚点，而 `onPlaying` 里存在一份**字面完全相同**的调用，`replace(...,1)` 删错了那处 → 守卫**正确地保持绿**却被误读一轮；随后的「还原」又插到另一处，造成一处丢失＋一处重复＋一处缩进被改，靠 `git diff` 逐行核对才发现并修复。教训：锚点必须唯一、还原后必须 diff 核字，守卫不红先怀疑篡改没落到目标处。
- **诚实边界**：本条根因是**静态可证**的（代码里确实没有第二条重量路径），但「用户那次是否正是量到 width:0」仍需真机/模拟器复测确认；若重测后拖动仍死，下一步就该上 `miniprogram-automator` 实连开发者工具读 `page.data.seekCfg` 取真值，而不是继续推断。

## 2026-07-20 · 批O 拖动进度条后视频回到开头（seek 只发起不保证落住·根因#8）

**用户反馈现象**：拖动进度条 → 视频回到开头、进度条冻在松手位置不再动（看起来像「拖动失效」）。另有一条 `AbortError: play() request was interrupted by a new load request` 在**打开页面时**就出现——用户明确说拖动前已有，故与本 bug 非同一件（那条是 autoplay 与 playSegment 显式兜底 play 的抢跑，视频照常播，暂不动，属模拟器噪声面）。

**先证伪再定位**（避免猜 WXS）：写临时脚本把 `seek.wxs` 整个模块 `new Function(module, Math, src)` 加载，喂假 `ownerInstance`（记录 setStyle/callMethod）+ 假几何（轨 left40/width300·durSec120·两节点）驱动一次真实拖动 → 提交的是 `onSeekCommit {sec:90}`（含磁吸命中），**渲染层这半是对的**，问题在逻辑层/平台侧。

- **根因**：`ctx.seek(sec)` 只是「发起」，平台不保证落住——媒体源不支持 Range 请求时（或部分模拟器实现）seek 会触发整段**重新加载**：位置回 0 且播放停摆在首帧。而播放一停就没有 `timeupdate`，`seekPct` 不再更新 → 进度条永久冻在 WXS 最后 setStyle 的位置。两个现象同一个因。commit 路径此前既没人确认位置真落住、也没人确认播放续上。
- **修**（两条都在 `onSeekCommit`，各复用既有机制不另起炉灶）：① `this._resumeAt = sec` —— 真发生重载时 `loadedmetadata` 必再触发，由 `onVideoMeta` 既有双保险（原为兜 `initial-time` 失效）把位置拉回目标秒；② `if (!this.data.paused) ctx.play()` —— 把停摆的播放推回去（幂等·与 `playSegment` 的 autoplay 停摆兜底同型；用户主动暂停态不打扰，`segDone` 分支原有 play 保留）。
- **守卫**：折进既有 `rw-mp-player-immersive`（不新起 id·该守卫 desc 本就覆盖 onSeekCommit 语义），新增两条断言 + desc 同步。**断言收紧一次**：初版查 `.play(` 出现过，被 `segDone` 分支那处 play 顶绿（错题本 E17 同型：一个方法体内两处同名调用，存在性断言测不出删了哪处）——改钉 `!this.data.paused) … .play(` 这条路径本身。
- **验证**：`npm run check` 全绿（208 repoCheck + 10 fileRule·246 文件 2262 例）。**反向自检四步**（E13/E14/E19：全程 python 精确增删 + `grep -c` 核实，未用 git checkout）：删补推 play → 红 → 还原绿；删 `_resumeAt` 赋值 → 红 → 还原绿。
- **诚实边界**：① 根因是**基于现象的推断**（模拟器 seek 重载不可在仓内复现），唯一证明是用户重测——真机 seek 通常不重载，故真机可能本来就没这个问题；② `onVideoMeta` 的兜底判据是「离目标 >2s 才补 seek」，用 `_at`（上次 timeupdate）近似当前位置——**拖动跨度 <2s 且真发生重载时兜不住**（位置留在 0），已知洞、未修（改判据会动到首挂载续播路径，收益不抵风险）。

## 2026-07-20 · 批N WXS 语法子集守卫（播放页真机编译失败·根因#8）

**症状**：微信开发者工具打开播放页即报 `./pages/player/seek.wxs:14:28: Unexpected identifier '('` + 连带 `__route__ is not defined`（渲染层脚本没注入＝整页白）。第二条是第一条的连带现象，非独立 bug。

- **守卫**（先立后修）：新增 repoCheck `rw-mp-wxs-runtime-subset`（roots `#8`），扫 `rewrite/mp/**/*.wxs`，剥注释（`stripComments` 单源·E1）+ 掩蔽字符串字面量（`maskStringLiterals`·防文案里的 `=>`/`...` 假红）后逐行咬 14 类非法构造：`Number()/String()/Boolean()/Array()` 调用、`Object.*`、`new`、箭头函数、`let/const`、模板串、展开符、`try/catch`、`class`、`import/export`、`Promise/async/await/setTimeout`、`wx.*`、ES6 字符串数组方法。扫描面为零个 `.wxs` 亦红（病根#16「空样本＝绿不发信号」——WXS 全撤属刻意退役、该同批删守卫）。
- **修**：`seek.wxs` 与 `lib/player.ts` 的 `clampSeek` 双源**同改为同一句 WXS 合法式** `Math.floor(value || 0)` / `Math.max(0, Math.floor(durSec || 0))`——入参已是 `number`，`|| 0` 收 NaN 足够，行为对测试全网格逐点不变；「同式同构」由此字面成立，不再留一个 TS 合法而 WXS 非法的写法当陷阱。`seek.wxs` 头注 + `player.ts` 行注各写清「别写 Number()」及其为什么。
