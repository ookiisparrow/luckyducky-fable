## 2026-06-24 · ⭐ 评价汇总真全量精确：getReviews 去 200 样本近似（债#13 后半收尾·根因#7）

**痛**：评价列表早已 cursor 分页，但**汇总**（评分/计数/星级分布/标签）仍基于 ≤200 最近样本（`approx` 标注）——某商品评价破 200 时评分与星级分布失真。
- **设计抉择（关键）**：原债提「写入时维护 `reviewStats/productId` 增量统计」，深查发现增量方案有**首建回填 + 并发 CAS + 半步漂移 + 标签 dot 字段路径**多重分布式坑（且测试桩无 `_.inc`、codebase 本就弃用 inc 走 CAS）。改用 **stateless `count()`+`aggregate(sum)`**（与 dashboard GMV 同范式·#18续）——全规避那些坑：零写路径耦合、零新集合（不占库权限锁）、零并发/迁移/半步问题、always 精确。
- **实现**：首页汇总＝4×`count()`（星级桶·1 星并入 2 星档·含 `_.in([1,2])`）+ 1×`aggregate.group({_id:null,sum:$rating})` → 评分/计数/分布全量精确、不读全部文档；`approx` 恒 false。**标签仍取近样本 top-5**（标签是模糊「印象」·精确需 unwind 聚合·收益低，诚实保留近似）。**兜底**：聚合/计数异常退回 bounded 样本 buildSummary（零回归·根因#8 失败有兜底）。
- **真库可信度**：count/aggregate(group _id:null,sum) 正是 loadtest `capacity` 模式已对真测试库证「破千不封顶精确」的两个 op——**不引入未真证的新聚合形状**（不用 unwind/group-by-field），故无 root#8 桩≠真 缺口。
- **守卫（test·先红后绿+反向自检）**：`reviews-summary-exact`（roots #7/债#13）＝`tests/cloud/getReviewsSummaryExact.test.js` 灌 **250 条**（>200 样本上限·已知分布 100/60/40/30/20）锁 count===250、score "3.8"、dist 精确、approx false。反向自检：强制走 bounded 兜底 → count 截到 200 红 → 还原绿。既有 `getReviewsPaged.test.js` 仍绿（小样本下精确口径同值·向后兼容）。
