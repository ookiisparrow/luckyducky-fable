## Context
现状:前端主干 + 云开发 P0 + 静默登录 + 商品上云已完成;课程仍写死一门(`data/course.js`,两层结构)、欢迎页只是"看过"标记、订单/支付未上云。目标:长出完整业务——**实物购买 → 收货扫唯一二维码 → 跳该课程欢迎页 → 确认开始观看即失去该件商品退货权 → 分步视频教学,只能看自己激活的课**,并配套管理后台。

## 一、端到端业务流程
微信静默登录(openid 登记账户,**不采集手机号**)→ 买**实物**材料包(微信支付)→ 收货,包内含**唯一激活二维码** →
微信扫码进小程序,**跳转该码对应课程的欢迎页**(按码解析课程,渲染该课程的标题/封面)→
欢迎页**两页式**:第 1 页课程引导;第 2 页明示「确认开始观看视频教程即失去**该件商品**的退货权」→
用户点击确认(记时间戳)→ 课程激活并绑定当前账户 → 课程目录 / 分步播放,**只显示自己激活的课**。

## 三、数据模型（云数据库集合）
- **users**:`{ _openid, nickname, avatar, createdAt }`（**不采集手机号**,2026-06-10 定）
- **products**:实物商品 `{ _id, name, price, was, tag, courseId(关联课程) }`（基数待确认:纯实物可否无课 / 套装是否多课）
- **courses**（**三层结构**,2026-06-10 定）:
  `{ _id, title, chapters:[{ id, title, lessons:[{ id, name, segments:[{ id, name, dur, videoFileId, free }] }] }] }`
  - 命名对应:**大章节 = chapter、中章节 = lesson、小片段 = segment**。
  - **目录只显示 chapter + lesson 两层**;segment 只在播放页出现(顶部分段进度条)。
  - **每个 segment 是独立视频文件**(各自 `videoFileId`):播放器逻辑干净(原生 ended 即段尾),素材按小片段剪辑。
- **orders**:`{ _id(订单号), _openid, items:[{ productId, name, spec, price, qty, refundable }], goods, coupon, ship, amount, address(收货快照), status, createdAt, paidAt }`
  - **`refundable` 下沉到条目级**(2026-06-10 定:退货权按单件商品,不影响同单其他商品)。
  - **云端定价**(2026-06-10 细化):前端只传 `productId/qty`,价格与金额一律云函数按 `products` 现算(不信任前端);
    商品名/规格快照进条目(历史快照惯例);地址快照只收白名单字段。
  - **模拟支付**(P4 前过渡):`createOrder` 直接 `status:'paid'` + `paidAt`;接真实支付后拆 `pending → 支付回调 → paid`。
    时间戳统一存 epoch 毫秒(避免 serverDate 跨 callFunction 序列化问题)。
- **qrcodes**(激活码,核心):`{ _id, code(唯一), courseId, batchId, status:'unused'|'activated', activatedBy, activatedAt }`
  - **码不绑订单**(2026-06-10 修订:备货阶段封码入包,发货时无从知道盒内是哪个码):码只属于(courseId, batchId);
    退货权执行改「确认时启发匹配 + 退货验收扫码兜底」,见 §四-3。
- **activations**(用户↔课程):`{ _id, _openid, courseId, qrcodeId, enteredAt(确认开始观看=退货失效点), createdAt }`

## 四、核心机制设计
1. **微信登录**:静默 openid upsert `users`(已跑通);头像 / 昵称授权可选补充;**手机号不获取**。
2. **二维码激活 + 重复扫码**:管理后台**按一款产品的配套课程批量生成** `qrcodes` → 打印 → **备货阶段每个该产品包装封入一码并封装**(2026-06-10 定,码不与订单绑定)。用户扫**小程序码**(`scene=code`)→ **该课程欢迎页**。云函数 `activateCourse(code)`:
   - 码未用 → 建 `activation` 绑定当前 openid + 标码 `activated`(一码一用、防复制;并发用「条件更新抢占」保证);
   - **同一用户再扫已激活码**:已确认(`enteredAt` 非空)→ 按钮变「继续学习」直接进课(不再出现退货权文案);
     **未确认 → 仍走确认页**(2026-06-10 细化:确认是进课的唯一闸门,激活≠可看,防"激活后绕过退货确认进课");
   - **他人扫已激活码 → 提示「该码已被激活」**,不可重复激活、不可进课。
3. **退货权节点（按单件商品;实现 2026-06-10 修订,码不绑订单后）**:欢迎页第 2 页「确认开始观看」→ 云函数 `confirmEnter`:
   写 `activation.enteredAt`(时间戳留证)+ **启发式自动失效**——若确认者本人订单中存在该课程对应商品(`products.courseId` 反查)的可退条目,取最早一条 `refundable=false`。
   - **同单其他商品退货权不受影响**;
   - **无论谁点击确认均生效**;送礼场景(确认者名下无订单)不自动失效,由验收兜底;
   - **强制兜底 = 退货验收扫码**:退货须寄回完整包装,验收扫包装内码——`status=activated` → 该件拒退;码未激活 → 正常退。
     不依赖订单绑定,自购 / 送礼一致有效,是退货权的**最终执行点**(`refundable` 降为展示性标记,裁决以验收扫码为准);
   - 有法律意义,需留时间戳证据;按钮旁明示后果(是否再加二次确认弹窗,实现时定)。
4. **课程权限**:「我的课程」/目录调 `getMyCourses`(按 `_openid` 查 `activations`,**只返回已确认 `enteredAt` 非空的**,2026-06-10 细化);播放页校验该课在激活集内;`videoFileId` 经云函数换**临时 URL**(防盗链,先校验已激活)。H5 / App 端无云回退为"视为已激活"(演示模式)。
5. **分步播放交互（用户体验重心）**:一个动作一个动作学。
   - 播放页**顶部分段进度条** = 当前 lesson 的 segments(如「认识工具」分 4 段);**下方完整进度条** = 当前 segment 的播放时长。
   - **segment 播完自动停止 → 弹「重复播放」按钮**;用户也可点「下一段」播本 lesson 的下一个 segment。
   - 待定细节(实现时拍板):最后一段播完是否引导进下一 lesson;是否限制拖动快进;~~学习进度记忆建议记到 **segment 粒度**~~(✅ 2026-06-11 已按 segment 落地,trackEvent/progress;为「计数陪伴」方向留位)。
   - 现有 `player/index.vue` 已有分段进度 + 自绘控件底子;**改造前先修调试日志 B(黑块)+ 真机验同层渲染**。
6. **持续上新（成对上架,2026-06-10 补充需求）**:商品款式**不断新增**,配套教学视频课程同步新增——上新是常态运营动作,不是一次性建库。
   - **上新流水线**:新增 product(CMS)→ 新增 course 三层编排 + 按段传视频 → `products.courseId` 关联 → 按课程生成码批次 → 印码随货发售。
   - **引用模型**:课程内容更新对已激活用户自动生效(activation 绑 courseId,非快照);segment 结构变更时旧进度按 segment id 失配作废,可接受(进度记忆按 segment id 记)。
   - **对后台的影响**:课程三层编排 + 按段上传是**复发性运营操作**,「seedCourses 种子当准源」只撑到上新常态化之前(见 §七 已定 4 与 关键决策记录 §11)。

## 安全底座（贯穿所有机制）
- **不信任前端**:openid / 角色 / 价格 / 订单状态 / 激活状态,一律云函数用 `getWXContext` + 数据库校验。
- **安全规则**:公共内容前端只读、用户只读写自己的;敏感集合(`orders`/`qrcodes`/`activations`)前端禁写、全走云函数。
- **云函数鉴权**:默认 `invoke: auth != null`;函数内再校验业务权限。
- **视频防盗链**:`videoFileId` 经云函数换临时 URL,先校验已激活。

## 七、关键技术决策
**已定（2026-06-10）**:
1. **退货权按单件商品**:`refundable` 在订单条目级;任何人确认均生效。~~发货时码绑订单条目~~ → **2026-06-10 修订**:码不绑订单(备货封码入包),实现改「确认时启发匹配 + 退货验收扫码兜底」,见 §四-3 与 关键决策记录 §12。
2. **小片段 = 独立视频文件**:每 segment 一个 `videoFileId`,素材按段剪。
3. **放弃获取手机号**:登录只用 openid,免企业认证依赖与按次付费接口。
4. **管理后台 = CMS 管数据 + 自建管二维码/看板**:平铺数据集合(商品/FAQ/评价等)用云开发 CMS 管;
   业务动作(码批次生成打印/验收扫码查询/条目级退货)自建;敏感集合(orders/qrcodes/activations)CMS 只读或不暴露,写一律走云函数;
   **courses 三层结构暂不入 CMS**(深嵌套编辑易错),近期以 `seedCourses` 种子为准源。
   ⚠️ 2026-06-10 补充需求(§四-6 持续上新)给该对策定了到期点:**上新常态化前须拍板课程编排形态**
