> 立 2026-06-14。本文 = **第二次重构的 live 计划/地图**，接替 `archive/重构总计划-v5.md`（第一次重构 B0–B9 作战计划，已完工归档）。

## 一、为什么有第二次重构
项目三阶段：① 原版 demo 反向生长（整合需求）→ ② 第一次重构 v0.9.1（架构重组 + 立自动化规范）→ ③ **第二次重构：把需求·功能·架构·自动化更深结合，从高维全局找更有效架构**。
**北极星标尺**：让没有技术基础的所有者更容易逼近专业项目结果。

## 二、scope 边界（本次只做技术整合）
- **做**：需求·功能·架构·自动化的**结构/技术**整合。
- **不做**：产品重心（承重柱 电商+锁视频 vs 制作陪伴）——那是 `archive/项目思路重构-上帝视角.md` 提的**生意决策**，独立 owner track；本次正交、不碰。技术整合反而让将来真要扶正承重柱时更便宜。

## 三、方向（演进，非推倒重写）
- 北极星 **A：需求当单一源、向下派生**（声明式产品模型 → 功能/类型/接口/守卫从它派生或对它校验）。
- 路径 **C：把元模式从「痛→守卫」长成「需求→功能→守卫」**。
- 铁律：**只在安全处生成**（数据/类型/接口契约），**永不重生成真机验过的运行时/UI**（根因#8）；**在分支做、不碰 live**（v0.9.1 生产跑真钱，沿用第一次重构的隔离护法）。

## 四、四道缝（高维诊断 = 整合的杠杆点）
1. **需求还在文档、没长进代码**：元模式只闭环「痛→守卫」，缺「需求→功能→守卫」。
2. **真相散在多本文档**（设计规格×2 / 业务逻辑架构 / 关键决策…），靠人对齐会漂。
3. **架构按技术分层（决策§17）、非按业务域**：业务↔代码隔一层翻译，非技术所有者离代码远一层。
4. **自动化在「守」不在「生」**：seed 单源能派生数据（T3），但类型/接口/页面骨架仍手写。

## 六、纪律
- 每阶段**先压测设计再动手**；每批走 `/refactor-batch`（先守卫后实现 → 反向自检 → squash 上 main）。
- 记录：本地图标阶段进度；批次证据记 `重构日志.md`（第二次重构批次同一账本，新开「## 第二次重构（演进）」段）。记录类账本（工作/重构/调试日志）按时间/季度卷档，不在 P0 强动。
