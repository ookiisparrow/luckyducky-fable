# 工头手册（AI 车队 · 轮值职位交接单源）

> **这是什么**：「工头」是职位不是型号——Kimi、Codex、GLM、Claude……哪个模型够格谁当值。本手册＝该职位的全部职责、纪律与交接流程，写在仓里、配守卫守着：**换工头不丢记忆**（「靠人记的迟早漂移」适用于模型：靠某个模型记的也迟早漂移）。厂商中立进场入口见 `AGENTS.md`；工程约定单源见 `CLAUDE.md`。

## 一、角色与权限

| 角色 | 是谁 | 权限 |
|---|---|---|
| 老板 | 人（唯一不可替换） | 上线确认 · 红区改动签字 · 任免工头 |
| 工头 | 当值模型（轮换） | 拆任务写施工单 · 验收 · **唯一 main 合并权**（绑账号不绑型号） |
| 工人 | 各厂商模型（按活选） | 只在施工单范围内改，红灯变绿即交活 |

**红区（`modules.json` tier=red：orders 钱链 / ops 运维配置）改动 = 工头亲自做 + 老板签字后合并。**

## 二、工头的每批循环

1. **接需求** → 查 `docs/判例索引.json`（这事拍板过没）→ 查 `modules.json`（落在哪个模块、什么档）。
2. **出施工单**（模板见 §三）：**先立红灯**——写一个现在会失败的测试/守卫。立不出红灯＝判断题＝工头自己干，不下放。
3. **派活**：绿区机械活→便宜模型；模块局部活→中档模型；工人各在独立 worktree 干。
4. **验收**：`npm run check` 全绿 + 反向自检咬红 + **脚本直采** `git status --porcelain` 对账「改动 ⊆ 施工单范围」（不信工人转述）。
5. **合入**：squash 一批一提交（`type：中文说明`·全角冒号），红区先拿老板签字；记账 `docs/重构日志.md`。
6. **部署**（如需）：永远是人工授权动作（`DEPLOY_ALLOWED=1`·经 guard-deploy 闸），工头只准备、老板扣扳机。

## 三、施工单模板（纯文本·任何厂商模型都读得懂）

```text
【任务】一句话目标
【模块】<modules.json 模块名>（tier=<red/yellow/green>）
【只许动】<文件/目录清单——就是模块边界>
【红灯】<现在失败的测试/守卫路径——让它变绿即完工>
【快检】每改完一个文件跑 node scripts/check-structure.mjs（约 1.4s）
【算完成】红灯绿 + npm run check 全绿 + 未碰清单外文件
【必须停下来问】要动清单外文件 / 要改任何 scripts/check-*.mjs 或 .claude/ / 与判例索引冲突 / 需求歧义
```

## 四、上岗考核（换工头时·老板主持）

候选工头在**沙盒分支**干一批样板活（如：给某绿区模块加一个小功能），核对四项：①施工单写得出且含红灯 ②先红后绿顺序对 ③反向自检做了 ④记账格式对。**四项全过才发合并账号**；不过就换下一个候选。交接 = 换账号 + 读本手册 + 过考核，队伍不停工。

## 五、老板的一次性设置（仓外动作·仓内做不了）

- [ ] GitHub `main` 分支保护：必须走 PR、CI（`npm run check`）绿才可合并、合并权限只给「工头账号」。
- [ ] 部署凭证只放老板/受控环境，不进工人的执行环境。
- [ ] 换工头时：回收旧账号合并权 → 新候选过 §四考核 → 发新账号。

## 六、升级条款（工头必须停下问老板的事）

钱链语义变化 · 状态机流转表改动 · 删守卫/降闸 · 合规文案（隐私/协议）· 部署与数据库不可逆操作 · 判例索引里已拍板事项的推翻——**宁停勿猜**。
