## 一、十二类病根（病史 → 本质 → 根治 → 落点 → 绝迹证明）

### 1. 并发「先查后写」、一次性副作用不幂等
- **病史**：债#14（login/updateProfile/trackEvent 并发首写可重复建档，读侧取第一条不稳定）；日志 I（confirmEnter 失退货权块在幂等守卫外，重复调用多扣一笔）；K-③（退款审批无原子抢占曾可双退款）；K-④（订单号 doc.set 撞号会覆盖旧单）；定向 D-1（submitReview count 预检 TOCTOU，修成确定性 `_id=orderId__productId` 成为孤例正确范例）。
- **本质**：把「检查」与「写入」当两步做，并发窗口必然存在；一次性副作用没有和「首次」原子绑定。
- **根治**：确定性 `_id` 通例化（kit `ids.js` 规范：`${OPENID}__${courseId}` 等，撞 id 即并发方已写，天然幂等）+ 写抢占一律条件更新（`transition()`）+ 一次性副作用绑定首次状态转移；存量数据惰性迁移（命中新 id 用之，未命中原子搬迁，任意时刻部署都安全）。
- **落点**：B1（kit ids/transition）/ B2（全函数接入 + 迁移脚本 dry-run）。
- **绝迹证明**：反向测试「并发双调用只产生一条文档」——恢复先查后建写法必红。

### 2. 状态转移散写、副作用反查状态集窄化
- **病史**：日志 H（退货权失效反查写死 `status:'paid'`，真实流程走到 shipped/done 时静默断链）；K-⑥（六处状态流转各自手写条件更新：confirmReceive/shipOrder/confirmEnter…）；订单状态机散在 6+ 函数无统一闸。
- **本质**：状态机没有单一权威表达，每处转移各自背诵规则；「正向流转窄条件」被误用于「副作用反查」。
- **根治**：kit `transition(coll, id, fromStatuses, toStatus, patch)` 携带合法流转表，钱相关转移自动留痕（stateEvents lite，单据仍是唯一账本——账本拆分已否决勿捡回）；TS 状态联合类型让写错状态名=编译失败。
- **落点**：B1（kit + shared 类型）/ B2（钱链函数全部换用）。
- **绝迹证明**：流转表 from 改宽一档对应 BAD_STATUS 测试必红；编译期拒绝非法状态名（**限调用方用 `ORDER_STATUS` 常量**；`transition()` 边界收 raw string、不挡 typo——跨集合泛型的已知折中，2026-06-13 深审收准）。

### 3. 信任边界靠每个函数自觉
- **病史**：K-①（payCallback/refundCallback 曾可被客户端伪造支付成功——全场最险）；定向 A-1（seed/init 三函数裸奔，任意登录用户可把生产商品表抹回种子）；日志「login 漏写 _openid」；18 处 openid 三行闸、4 处 admin 闸纯手抄；债#15（admin 口令 bootstrap 抢占窗口）。
- **本质**：安检不是缺省而是每次手写的自觉——忘一处就是钱/权限的洞；越「不像业务」的函数（seed/回调）越容易忘。
- **根治**：kit gate fail-closed 缺省——`withOpenId`（不带身份进不来）/`withAdminGate`/`assertServerCall`（回调防伪内置：见 OPENID 即拒）/`defineNotifyCallback`（幂等+核验 fail-closed 框架；金额/订单号不符留痕——refundCallback 严格不翻状态，**payCallback 例外：钱已到账→置 paid 留 feeMismatch 痕供人工对账**，2026-06-13 深审收准）；bootstrap 须匹配部署密钥（堵 #15）。新函数用 kit 定义器才能成型，**不过闸无法写库成为结构事实**。
- **落点**：B1（kit）/ B2（28 函数全接入）。
- **绝迹证明**：闸还原（去掉 gate）对应反向测试必红；防伪闸自检（带 OPENID 调回调必拒）。

### 4. 金额浮点靠手工纪律
- **病史**：applyRefund 退款分摊靠「分运算防浮点」的手工纪律（K 批次 PR #73）；全库价格存元、运算时各自小心。
- **本质**：金额的安全性依赖每个写代码的人记得换算，类型系统不设防。
- **根治**：shared 包 `Fen` 品牌类型（branded type），金额全链「分」整数，元只在展示层出现；云函数边界收元转分一次完成。
- **落点**：B1（shared 类型）/ B2（钱链迁移时换轨）。
- **绝迹证明**：编译期——浮点金额赋给 Fen 直接编译失败。

### 5. 样板复制即漂移
- **病史**：28 份 `cloud.init` 样板；payCallback↔refundCallback 70% 重复；种子数据 data/ 与 cloudfunctions/seed 双份手工同步（「需与 xx 保持一致」镜像注释）；adminApi `str` 助手曾 4 处重复（优雅性 A-1）；admin 防抖三件套散三处、安全审修 flush 时逐个手补正是「散落必漏」实证（定向 B-2 → 优雅性 B-1 抽 useAutosave）。
- **本质**：部署边界（每函数独立打包）曾使共享「不划算」（优雅性 A-2 旧判定），于是复制成为默认——而复制必然漂移。**A-2 翻案依据**：共享面已从「省几行」变成 闸×3+transition+留痕+分页+确定性id 的整套内核，esbuild 构建期内联使共享与独立打包不再矛盾——旧取舍的前提消失。
- **根治**：cloud-kit 共享内核（esbuild 内联+摇树，部署形态与现状同构）；种子单一来源进 shared；镜像注释清零，「保持一致」从人工义务变机器保证。
- **落点**：B1（kit）/ B2（收编样板）/ B3（种子归一）。
- **绝迹证明**：净化指标 commit 报数（init/闸/双回调归零，cloud 层净减 ≥25%）；种子生成物 diff 空；全库 grep「保持一致」类注释为零。

### 6. 演示回退双路径泄漏
- **病史**：K-⑤（api/order.js 曾把云端拒单吞进本地回退=交易动作本地伪成功——暗病）；8 个 api 文件各写各的回退形状；优雅性 A-3（createOrder↔api/order.js 订单镜像靠注释维系）；债#8（pending-pay 演示孤页滞留到 P4 才清退）。
- **本质**：「无云回退演示」与「云端真路径」并存却无统一契约，错误归类（拒单≠无云）只能靠每处 catch 自己分辨；镜像一致性靠注释。
- **根治**：回退策略收口为统一形状——**无云(null)→演示回退；云异常/拒单→上抛**，8 个 api 一个模式；镜像对照测试（parity test：本地生成器 vs 云函数同输入同输出）把镜像从注释变机器锁。
- **落点**：B3.5（新增批：miniapp 回退收口）。
- **绝迹证明**：parity 测试锁订单号格式/金额一致；「拒单被吞」反向用例必红。

### 7. 固定 limit，规模即挤出
- **病史**：债#13（订单 100/后台订单 200/后台售后 200/评价 200 固定条数，真实增长后旧数据被挤出工作台；评价汇总只算最近 200 条非全量口径）；定向 A-1 提及 listDrafts 同通道响应体上限。
- **本质**：「先写个数字够用」没有退出机制，数据增长是必然而 limit 是静默炸弹。
- **根治**：kit `paging.js` 统一分页协议（cursor/limit 契约），后端先行且兼容无参调用（部署窗口内新旧前端都不坏），再接 admin 订单/售后页与小程序 order-list 翻页。【用户拍板：纳入重构】
- **落点**：B4（随 adminApi 拆分同批接入）。
- **绝迹证明**：分页契约测试；无参调用兼容用例。

### 8. 验证样本失真——「E2E 过了」≠「真的能用」
- **病史**：日志 F（1×1 测试图过、真实 200KB 照片必拒——最小样本只验逻辑不验边界）；日志 G（凭证只验「拿到」没验「用」，直传路径从未真走通；「双保险」只兜第一步等于没兜）。

- **落点**：B7（质量基线批）。
- **绝迹证明**：方法论清单进验收手册模板；fixture 进测试基建。
