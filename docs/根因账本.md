> **框架循环**（痛→本质→不变量→守卫→反向自检→归因）的 canonical 定义在 `docs/元模式.md` §A；本账本是其 §B 的「病根清单」实例（17 类病根），不复述循环本身。覆盖率由守卫注册表单源 `scripts/check-structure.mjs` 的 `guard-coverage` 机器核。
> 重构的脊柱文档（2026-06-13 立，用户拍板：**全覆盖**——代码架构 + 运维资产档案 + 验收方法论 + 文档体系规则）。
> 方法：通读生产仓全部问题史（调试日志 A–K、技术债 #1–#15、四份审核报告、工作日志教训），聚类出**反复咬人的本质问题**，每类配一个让它「从此绝迹」的系统性解法；批次跟着病根走，批次表见 `现状与路线.md`（第一次重构 v5 批次表归 `archive/`）。

## 一、17 类病根（病史 → 本质 → 根治 → 落点 → 绝迹证明）

### 1. 并发「先查后写」、一次性副作用不幂等
- **病史**：退款审批曾无原子抢占，并发调用可致同一笔订单被双倍退款；确定性 `_id` 修复前，login/updateProfile 等首次写入也曾在查后写窗口内重复建档。详见史料层。
- **本质**：把「检查」与「写入」当两步做，并发窗口必然存在；一次性副作用没有和「首次」原子绑定。
- **根治**：确定性 `_id` 通例化（kit `ids.js` 规范：`${OPENID}__${courseId}` 等，撞 id 即并发方已写，天然幂等）+ 写抢占一律条件更新（`transition()`）+ 一次性副作用绑定首次状态转移；存量数据惰性迁移（命中新 id 用之，未命中原子搬迁，任意时刻部署都安全）。
- **落点**：B1（kit ids/transition）/ B2（全函数接入 + 迁移脚本 dry-run）。
- **绝迹证明**：反向测试「并发双调用只产生一条文档」——恢复先查后建写法必红。

### 2. 状态转移散写、副作用反查状态集窄化
- **病史**：退货权失效的反查逻辑曾写死单一状态 `status:'paid'`，真实订单流转到 shipped/done 时静默断链、权益判断失真；订单状态机曾散在 6+ 函数各自手写条件更新。详见史料层。
- **本质**：状态机没有单一权威表达，每处转移各自背诵规则；「正向流转窄条件」被误用于「副作用反查」。
- **根治**：kit `transition(coll, id, fromStatuses, toStatus, patch)` 携带合法流转表，钱相关转移自动留痕（stateEvents lite，单据仍是唯一账本——账本拆分已否决勿捡回）；TS 状态联合类型让写错状态名=编译失败。
- **落点**：B1（kit + shared 类型）/ B2（钱链函数全部换用）。
- **绝迹证明**：流转表 from 改宽一档对应 BAD_STATUS 测试必红；编译期拒绝非法状态名（**限调用方用 `ORDER_STATUS` 常量**；`transition()` 边界收 raw string、不挡 typo——跨集合泛型的已知折中，2026-06-13 深审收准）。

### 3. 信任边界靠每个函数自觉
- **病史**：支付/退款回调曾可被客户端直接伪造成「支付成功」调用，是全库最险的一处信任缺口；seed/init 等『不像业务』的函数也曾裸奔无鉴权，任意登录用户可把生产商品表抹回种子数据。详见史料层。
- **本质**：安检不是缺省而是每次手写的自觉——忘一处就是钱/权限的洞；越「不像业务」的函数（seed/回调）越容易忘。
- **根治**：kit gate fail-closed 缺省——`withOpenId`（不带身份进不来）/`withAdminGate`/`assertServerCall`（回调防伪内置：见 OPENID 即拒）/`defineNotifyCallback`（幂等+核验 fail-closed 框架；金额/订单号不符留痕——refundCallback 严格不翻状态，**payCallback 例外：钱已到账→置 paid 留 feeMismatch 痕供人工对账**，2026-06-13 深审收准）；bootstrap 须匹配部署密钥（堵 #15）。新函数用 kit 定义器才能成型，**不过闸无法写库成为结构事实**。
- **落点**：B1（kit）/ B2（28 函数全接入）。
- **绝迹证明**：闸还原（去掉 gate）对应反向测试必红；防伪闸自检（带 OPENID 调回调必拒）；createOrder 缺支付配置必拒 PAY_CONFIG_MISSING（守卫 pay-config-fail-closed，仅 env ALLOW_MOCK_PAY=1 放行 mock）。

### 4. 金额浮点靠手工纪律
- **病史**：退款分摊运算曾靠「分运算防浮点」的手工纪律硬撑，全库价格以元存储、换算全凭人记。详见史料层。
- **本质**：金额的安全性依赖每个写代码的人记得换算，类型系统不设防。
- **根治**：shared 包 `Fen` 品牌类型（branded type），金额全链「分」整数，元只在展示层出现；云函数边界收元转分一次完成。
- **落点**：B1（shared 类型）/ B2（钱链迁移时换轨）。
- **绝迹证明**：编译期——浮点金额赋给 Fen 直接编译失败。

### 5. 样板复制即漂移
- **病史**：支付回调与退款回调两份实现体量重复约七成、各自演化易漂移；商品种子数据在 data/ 与 cloudfunctions/seed 两处手工同步，全靠注释提醒「保持一致」。详见史料层。
- **本质**：部署边界（每函数独立打包）曾使共享「不划算」（优雅性 A-2 旧判定），于是复制成为默认——而复制必然漂移。**A-2 翻案依据**：共享面已从「省几行」变成 闸×3+transition+留痕+分页+确定性id 的整套内核，esbuild 构建期内联使共享与独立打包不再矛盾——旧取舍的前提消失。
- **根治**：cloud-kit 共享内核（esbuild 内联+摇树，部署形态与现状同构）；种子单一来源进 shared；镜像注释清零，「保持一致」从人工义务变机器保证。
- **落点**：B1（kit）/ B2（收编样板）/ B3（种子归一）。
- **绝迹证明**：净化指标 commit 报数（init/闸/双回调归零，cloud 层净减 ≥25%）；种子生成物 diff 空；全库 grep「保持一致」类注释为零。

### 6. 演示回退双路径泄漏
- **病史**：本地演示回退路径曾把云端明确的拒单响应吞成本地「交易动作已成功」的假象，是最隐蔽的一类暗病。详见史料层。
- **本质**：「无云回退演示」与「云端真路径」并存却无统一契约，错误归类（拒单≠无云）只能靠每处 catch 自己分辨；镜像一致性靠注释。
- **根治**：回退策略收口为统一形状——**无云(null)→演示回退；云异常/拒单→上抛**，8 个 api 一个模式；镜像对照测试（parity test：本地生成器 vs 云函数同输入同输出）把镜像从注释变机器锁。
- **落点**：B3.5（新增批：miniapp 回退收口）。
- **绝迹证明**：parity 测试锁订单号格式/金额一致；「拒单被吞」反向用例必红。

### 7. 固定 limit，规模即挤出
- **病史**：订单/后台订单/后台售后/评价等列表长期固定 limit（如 200 条），真实数据增长后旧数据被静默挤出后台工作台。详见史料层。
- **本质**：「先写个数字够用」没有退出机制，数据增长是必然而 limit 是静默炸弹。
- **根治**：kit `paging.js` 统一分页协议（cursor/limit 契约），后端先行且兼容无参调用（部署窗口内新旧前端都不坏），再接 admin 订单/售后页与小程序 order-list 翻页。【用户拍板：纳入重构】
- **落点**：B4（随 adminApi 拆分同批接入）。
- **绝迹证明**：分页契约测试；无参调用兼容用例。

### 8. 验证样本失真——「E2E 过了」≠「真的能用」
- **病史**：最小尺寸测试图与内存桩长期能让 E2E 全绿，真实体积照片、真实签发链路一上线立刻失败；一条缺 items 字段的脏订单曾让「全部订单」整页白屏——测试样本永远带 items＝样本失真。详见史料层。
- **本质**：验证的置信度取决于样本与真实的距离；走到链路第一步就宣告通过。
- **根治**：验收方法论入册（acceptance-check skill 与 `docs/运维手册.md` §⑦）：① E2E 必须用真实尺寸样本（测试基建备真实大小 fixture）；② 凭证/签名类必须按签发约定的完整请求形状走完整链路；③ 兜底路径必须覆盖主路径全部失败点并被测试逼出。
- **落点**：B7（质量基线批）。
- **绝迹证明**：方法论清单进 acceptance-check skill 与 `docs/运维手册.md` §⑦；fixture 进测试基建；订单脏数据白屏**双层守**——store 入库归一（`normalizeOrder`）+ 渲染访问点安全访问器（`data/orders.js orderItems/orderQty` 单源，列表/详情用之），`tests/orders.test.js` 两层各有直测锁（单层只测入库＝复发暴露的样本盲点）。

### 9. git 外资产无正册
- **病史**：支付连接器/支付流/退款流等八件控制台资产此前只散记在工作日志，误删即支付瘫痪；15 个集合的库权限规则也从未入档，验收单 X 长期靠人肉核对。详见史料层。
- **本质**：系统的真实形态 = git 内代码 + git 外配置，后者没有版本管理、没有重建手册，灾难恢复靠记忆。
- **根治**：`console-assets/` 正册化——八件资产副本（密钥 REDACTED）+ 工作流节点与入参表达式 + script1 代码 + **库权限规则期望表**（验收单 X 固化）+ drift-checklist + 30 分钟重建手册；变更纪律「先 repo 后控制台」。
- **落点**：B6。
- **绝迹证明**：对照控制台人工核一遍；重建手册演练（纸面）。

### 10. 工具实例状态不可信
- **病史**：开发者工具作为有状态共享实例，build 产物与自动化调试互踩后曾多次出现握手成功但 RPC 不回包、真机会话被推坏白屏等假象。详见史料层。
- **本质**：开发者工具是有状态的共享实例，build/自动化/真机调试互踩后其表象不可信——在僵实例上排查业务代码是纯浪费。
- **根治**：守则机器化能做的已做（visual-check RPC 探活守卫）；余者成文入样板房 CLAUDE.md：跑 build 前停自动化；工具怪象先 `cli quit`+重启再排查。
- **落点**：B7（守则随质量基线入册）。
- **绝迹证明**：守则进 CLAUDE.md；脚本探活守卫保留。

### 11. 文档职责渗漏
- **病史**：现状与路线等活文档单条目曾膨胀到 60+ 行，CLAUDE.md 一度涨到 314 行，状态描述在多处漂移不一致。详见史料层。
- **本质**：文档没有「容量与归档」规则，膨胀是默认走向；规则不机器化就会回潮。
- **根治**：文档体系规则入样板房 CLAUDE.md：现状条目 ≤8 行、溢出沉工作日志；调试日志按季卷档；正册只放「仍会被查」的内容；约定/记录/参考三类边界明示。【用户拍板：纳入全覆盖】
- **落点**：B7（与验收方法论同批）；样板房新文档（重构日志/账本/总计划）开局即守此规则。
- **绝迹证明**：规则入 CLAUDE.md；每批收尾自查条目行数。

### 12. 平台规则外部风险
- **病史**：实物类小程序云调用支付曾被平台整条封死改走强制工作流；新线退款也曾把含中文字符的内部售后单号直接当微信 `out_refund_no` 传出，被 `PARAM_ERROR` 拒收（微信单号只认 ASCII 64 位以内）。详见史料层。
- **本质**：平台接缝有两类外部风险：① 平台规则**单方变化**（不可预防，靠接缝收到最少点位、改动面最小）；② 我方**传出参数须符平台契约**（字符集/长度/格式），把内部标识符（可含任意字符/超长）直接当外部平台单号用＝违约被拒——可预防，接缝处规整即可。
- **根治**：① `callFlow` 单点收口 cloudbase_module 调用；接缝清单（flowId/refundFlowId/notify_url）入 console-assets 正册与 runbook。② **接缝传出标识符经规整**：`kit.refundNoFor(_id)` 产出微信合规 `out_refund_no`（已合规原样·否则「订单号前缀 + cyrb53 短哈希」ASCII≤64·确定·唯一），售后单落 `outRefundNo` 供回调反查。
- **落点**：B1（callFlow）/ B6（接缝清单）/ 调试案 A 修复批（refundNoFor·2026-07-05）。
- **绝迹证明**：全库 cloudbase_module 调用点 grep=1 处（守卫 `flow-seam-single`）；退款单号合规由 `rw-money3-golden`（refundNoFor 恒配平台正则 + 回调按 outRefundNo 反查）+ `rw-admin3-golden`（approve/override 发出的 out_refund_no 恒合规·非原始中文 _id）焊死（反向自检 identity 篡改 4 红）。

### 13. 无频控，认证/写入端点可被爆破或滥用穿透
- **病史**：管理后台口令端点曾是公网可达、口令仅 6 位下限 + sha256 快哈希、无任何重试限制的裸接口，理论上可从任意地方暴力破解到完全接管后台；用户端写函数（trackEvent/createOrder/login）同样无频控，可刷爆库 QPS。详见史料层。
- **本质**：认证与写入端点一旦暴露公网，没有「砸门」限速则口令可枚举、资源可耗尽——门锁（认证/授权/防伪/钱）再硬也挡不住无限次尝试。
- **根治**：`kit.throttle`（滚动窗口失败计数 + 达阈锁定，按 key 隔离）；adminApi 口令校验路径先过 `throttleLocked` 闸、失败 `throttleFail`、成功 `throttleReset`。守卫 `admin-login-throttled` 焊「口令路径必经频控」。
- **落点**：fix-后台登录限流（P1）+ fix-用户端写函数频控（P2：withRateLimit + `user-writes-throttled` 守卫），均 2026-06-15。
- **绝迹证明**：守卫 `admin-login-throttled`（grep adminApi=在）+ `user-writes-throttled`（trackEvent/createOrder/login/updateProfile 须包 withRateLimit）；测试 adminThrottle（5 错→第 6 次正确口令亦 429 + IP 隔离）+ userRateLimit（超限即 RATE_LIMITED + openid 隔离）。**两段（P1 认证 + P2 用户写）齐，落点闭合。**
- **续防（2026-07-10 批6·离线面）**：频控只挡在线爆破，病史里点名的「sha256 快哈希」离线彩虹表面此前未收——口令凭证升 salt+scrypt KDF（`kdf`/`keyMatches` 单源·存量 sha 账号首登无感升级·createAgent 撞口令拒语义盐感知保留·扫描顶上限 `alertIfScanAtCap` 留痕），守卫 `rw-admin-key-kdf` 焊死；会话令牌（32B 随机高熵）保持 sha 不变。

### 14. 失败静默化——吞错 / 伪成功 / 告警不进人眼（2026-07-04 立·全日志回归分析）
- **病史**：客服机器人对话失败曾只写进无人盯的日志，可能数日静默罢工；坐席发送失败的接口甚至回「成功」骗过前端清空输入框，用户体验从「发不出」演成「无声消失」。详见史料层。
- **本质**：失败点的缺省行为是「静默」——catch 默认吞、发送失败默认装 ok、告警默认只进无人盯的日志；fail-soft（降级返回语义）与「失败不可观测」被混为一谈，故障发现只能靠下次真用撞见。旧线「伪成功」是它在演示回退域的特例（已归 #6），本条是通例。
- **根治**：**失败必可观测**——kit `observe.alert`（`[LD_ALERT]` 结构化留痕 + webhook 触达运营）为唯一告警出口；动作类失败（接缝外呼/发送/回调验签）分支必留痕，fail-soft 只许改返回语义、不许抹可观测性；读路径 `doc.get().catch(()=>null)`（缺席=null 惯用法）不在此列；刻意静默须行内注释声明为什么。
- **落点**：M2 批9（callFlow 两失败分支落 alert）；M3 批8 坐席「发送成功才清框」；旧线钱链告警接入。**靠人面**：外部接缝「配过一次≠一直通」的活体探活未排期，见 `待办与债` 待办。
- **绝迹证明**：守卫 `rw-flow-observable`（反向测试：失败分支不留痕即红）+ `moneychain-alert-wired`（payCallback/refundCallback 静默语义失败须经 alert）。

### 15. 加载链路冗余云调用——跨页共享的静态目录数据无会话缓存收口
- **病史**：detail/cart 等页面曾绕过已立的 `lib/catalog.ts` 缓存范式、直调云端重拉同一份商品列表；player/me/my-courses 各自重拉静态课程目录、同会话内无缓存收口；单次云调用往返实测 0.7-1.4s，是这条冗余的实际代价。详见史料层。
- **本质**：`lib/catalog.ts` 已示范「模块级 let cache + prime + miss 兜底重拉」的会话缓存范式（T3「云为唯一真相」精神在前端会话层的延伸），但范式没有被结构性收口——新页面各自决定要不要复用，于是各自重犯。跨页共享的只读静态目录数据，理应像 catalog 一样一次拉、全会话复用；页面各自重拉是「样板复制即漂移」（病根#5）在加载路径上的特例：不是复制了错代码，而是漏复制了缓存这一步。
- **根治**：商品目录复用既有 `lib/catalog.ts` 单源；课程目录比照同一范式新立 `lib/courses.ts`（模块级缓存 + prime + miss 兜底），detail/cart/player/me/my-courses 五页改从 lib 层取数、不再直调 api 层的 `getProducts`/`getCourses`；页面层与 lib/api 层的调用边界由守卫钉死，防止「改完当时对、下一页又直连」的复发。
- **落点**：批A 一批完成——守卫先行立红（`rw-mp-catalog-cache-single-source`）+ 同批改到绿：home/detail/cart/player/me/my-courses 六页接入缓存、`lib/courses.ts` 新立、`welcome.ts` 两个独立调用去串行改 `Promise.all`。批B 图片面：内容图 lazy-load 接线表守卫（`rw-mp-image-lazy-wired`）先立红，cloud 端 fileID 批量转临时址与 detail 图册窗口渲染另计。
- **绝迹证明**：守卫 `rw-mp-catalog-cache-single-source`（页面层 import `api/catalog` 的 `getProducts`、import `api/learning` 的 `getCourses`、或裸写 `callApp('getProducts')`/`callApp('getCourses')` 字面量，出现在 api/lib 层以外即红）；反向自检——任一页面改回直连 api 层必复红。

### 16. 守卫绑定漂移——扫描面/指针指着死目标，空样本＝绿（2026-07-18 立·盲区体检）
- **病史**：残留 worktree 目录曾被守卫当活代码扫，让本地 `npm run check` 假红十余处而 CI 洁净 checkout 却全绿；同日还并发出现两对守卫同 id 重复注册（其中一份正是已验尸的坏版本仍在跑 6 天）与部署脚本清单指着已冻结的旧线拓扑。详见史料层。
- **本质**：守卫的**逻辑**不漂移，但守卫与现实之间的**绑定**（inScope 路径、reverseTest 指针、manifest 镜像、扫描目录清单）会漂移，且漂移不发信号——「扫到 0 个活文件」与「扫遍全仓无违例」同形，「清单 diff 为空」与「无事可部署」同形（空样本＝绿）。元守卫 `guard-coverage` 只核「有守卫声明 roots 含此病根」（声明层），从不核「扫描面还够不够得着活代码」（现实层）——声称覆盖与实际覆盖永久脱钩。M5 切换是集中爆发点：搬走了生产，没有任何机制清点「指向生产的引用」（`oldline-frozen` 守住了旧线的字节，没守指向旧线的指针）。
- **根治**：① 扫描面统一排除收口（`SKIP_DIR`/`lsScan` 单源·递归 walker 一律经它·dot 目录/vendor 恒不入扫描面）+ **金丝雀夹具**（`rewrite/mp/.claude/worktrees/fixture-scan-surface/` 故意含违例 token 常驻——排除失效当场咬红）；② 注册表 id 唯一元守卫；③ fileRule 活线活性下限（每条须命中 ≥1 个 rewrite/ 文件，守冻结参照的须显式入白名单——「声称守活线」与「真够得着活线」由机器对齐）；④ 工具链禁旧线引用（deploy/verify 类脚本引用 `packages/` 即红·冻结工具白名单豁免）；⑤ 摆设检查（空样本恒绿）按「别摆设守卫」原则退役，不修不裱。
- **落点**（过程账见 archive 重构日志 fable 分册）：盲区体检修复战役批1（扫描面+金丝雀+id 唯一+摆设退役）、批2（rw-dep-direction 活线镜像+活性下限+reverseTest 指活线）、批3（deploy-fns/deploy-test 指针迁移+禁旧线引用守卫）、批4（三道闸补真实构建+活线产物验证）、批5（preflight+deploy-drift 钱链检测迁活线·白名单退役 2 项）——见 `重构日志` 2026-07-18 盲区体检各批。
- **绝迹证明**：金丝雀夹具常驻（扫描面回潮当场红）；`guard-id-unique`（重复注册即红）；`guard-scan-liveness`（fileRule 活线零命中且不在白名单即红）；`rw-toolchain-no-oldline`（部署/验证脚本引用旧线即红）。反向自检：临时复制任一守卫 id → guard-id-unique 红；临时把 rw-dep-direction 的 inScope 改指 packages → liveness 红。

### 17. 快照外无传感器——持续性义务无机器建模，恶化不发信号（2026-07-23 立·瘦身大作战）
- **病史**：package-lock 里揣着旧线 639 个死包无人察觉，安全高危项大半来自这些死依赖；重构日志曾膨胀到 664KB/1732 行才被人工下令减肥，全程没有一个机器信号提醒。详见史料层。
- **本质**：守卫体系强在「快照断言」（此刻代码违不违例），弱在「趋势义务」（这个量上周多少、现在多少、该不该管）——膨胀每一步都不违例，累计起来是灾难；且瘦身成果无棘轮，砍掉的重量会静默长回来。#16 是「绑定漂移不发信号」，#17 是「恶化趋势不发信号」——同为传感器缺失，作用维度不同（空间 vs 时间）。
- **根治**：给持续性义务立**基线棘轮守卫**：现值对基线，涨超预算红（膨胀须显式记账上调）、基线虚高也红（瘦身后必须下调基线锁成果·防回弹）；零容忍类（死导出/幽灵依赖/孤儿资产）直接快照断言化。
- **落点**：瘦身大作战哨兵批（2026-07-23·`scripts/lib/slim-scan.mjs` 五哨兵）；数据面（库存守恒/集合保留期/证书倒计时）归 inspect 升格批照旧待拍板（`待办与债` 盲区清单条）。
- **绝迹证明**：`rw-loc-budget`/`rw-lock-budget`（棘轮双向咬）+ `rw-dead-exports`/`rw-phantom-deps`/`rw-orphan-assets`（零容忍）。反向自检：删基线文件/植死导出金丝雀 → 各哨兵红。
