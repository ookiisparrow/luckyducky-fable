# 项目思路重构——上帝视角审视（以根本意图为基础，可颠覆）

> 2026-06-13 立。这是一份**产品思路**文档，海拔高于工程重构（cloud-kit / B1–B8）。
> **它不改任何代码、不改工程重构计划、不拦 v1.0 上线**——它做一件事：从根本意图出发，把「这个项目到底在解决谁的什么问题」想穿，并在发现现行思路与根本意图错位时，给出一个**可颠覆的重构思路**作为建议，最终方向由所有者拍板。
> 配套阅读：根本意图与产品方向探索见生产仓 `docs/项目现状.md` 末节；现行设计依据 `设计规格-课程电商系统.md`；决策来龙去脉 `关键决策记录.md`；逻辑链全景 `业务逻辑架构.md`。

---

## 一、根本意图（去掉一切实现，只剩人的需求）

**幸运小鸭存在的理由 = 让一个钩织新手，真的把作品做出来。**

不是「卖材料包」，不是「教钩织视频」——那些是手段。被验证过的真实新手痛点（团队自己的调研结论）：

| 痛点 | 本质 | 视频能解吗 |
|---|---|---|
| 看不懂针目 | 识别 | 解一半（看得到示范，但镜头一过就忘） |
| 数错针 | 计数 | **解不了**（视频不知道你数到第几针） |
| 记错针数 / 忘了做到哪 | 状态记忆 | **解不了**（视频不记得你停在哪一行） |

一句话定调（团队原话）：**这是「计数 / 状态」问题，视频只解决「看不懂」的一半。**

> 根本意图的检验标准：一个新手用完幸运小鸭，**作品做完了没有**？而不是「视频看完了没有」。

## 二、现行项目思路全景（上帝视角，它实际是什么）

现行思路一句话：**「课程电商系统」——卖实物材料包，用包内唯一二维码把视频课程锁给买家。**

十条业务链按「服务于人的哪个需求」归类：

| 链 | 在做什么 | 服务根本意图吗 |
|---|---|---|
| 1 登录身份 | 基建 | 中性（地基） |
| 2 商品上架 | 电商 | 否（卖货） |
| 3 课程内容（编排→发布→**观看视频**） | 内容分发 | **解「看不懂」的一半** |
| 4 下单 | 电商 | 否 |
| 5 履约（发货/收货） | 电商 | 否 |
| 6 二维码激活 | 内容 DRM（把视频锁给买家） | 否（防白嫖，不解痛点） |
| 7 学习进度（**视频段看没看完**） | 分析 | **几乎否**——记的是「视频段落位置」，不是「你钩到第几针第几行」 |
| 8 评价 | 电商 | 否 |
| 9 首页橱窗 | 电商营销 | 否 |
| 10 售后退款 | 电商 | 否 |

**上帝视角结论**：现行系统约 **8.5/10 是电商 + 内容 DRM 基建，真正触碰「计数/状态/记忆」核心痛点的链 = 0 条。** 整个 v0.9（28 云函数、15 集合、控制台、真实支付、退款双闭环）盖的是一座**收银台 + 内容版权闸**，学习体验最薄的一块——一个播放器 + 「视频段看没看完」的进度——恰恰是离根本意图最近的地方，却被做得最轻。

把它做到 v1.0 完美上线，用户拿到的是：**「一个带材料包的钩织视频网站」**。它的差异化只剩「材料包」，而「课程」与 B 站 / YouTube 上免费的钩织教程同质——**根本意图里最硬的痛点（数错/记错）一个都没解。**

## 三、来龙去脉（这个思路怎么长成现在这样）

不是有人决策错了，是**洞察到达的顺序**造成的：

1. **2026-06-07 第一次需求对话**：所有者用电商语言描述了完整业务——「实物购买 → 收货扫码 → 激活绑账户 → 确认即失退货权 → 只看自己买的课」。这是现行思路的**源头**，一个自洽的「卖课+防白嫖」框架，于是 P0–P5 全部照它盖。
2. **2026-06-10 根本需求复盘**：细化为三层课程 / 条目级退货 / 放弃手机号——**仍在电商框架内打磨**，没有跳出。
3. **某刻的更深洞察（产品方向探索）**：团队识别出真实痛点是计数/状态/记忆，并明确写下「视频只解决一半」「候选承重柱=交互式计数陪伴」「登月级=AI视觉点数」——**但当时决定「先把视频UI做完作为保底，以后推进，不锁死」。**

**关键的一岔**：最深的洞察来得比框架晚，且被**降级为「以后」**。于是一年的工程全部投在电商/DRM 基建上，那根「应该承重的柱子」始终停在探索区。现行思路 = **所有者最初的框架，惯性滑行至今**；真正的承重柱，团队自己早看见了，只是停在门口。

## 四、可颠覆的观察（诚实地两面看）

做颠覆前先替现行思路辩护，否则是不负责任的推倒：

**现行思路站得住的地方（不可轻贱）**：
- 生意要先能收钱。卖实物包是被验证的变现；计数陪伴是个功能，未必自己成生意。
- 「买包才有码、码才解锁内容」是一道真实的护城河与防白嫖设计。
- 已盖的基建（支付/履约/激活/控制台/账号）**在任何思路下都要用**——计数陪伴一样需要「用户、商品、购买、把内容绑给买家」。**电商脊柱是必要地基，不是浪费。**
- 「先有 kit X 被买、被绑给买家」才谈得上「为 kit X 做计数陪伴」——先盖电商脊柱，顺序上说得通。

**现行思路偏离根本意图的地方（要正视）**：
- 验证过的痛点是计数/记忆，系统里**没有任何东西碰它**。产品完整上线 ≠ 解决用户的真实问题。
- 最深的护城河是计数陪伴（交互状态追踪 + 针目特写 + 远期 AI 点数）——视频是大路货，材料包谁都能配。**差异化押在了最浅的地方。**
- 庞大的电商/DRM 投入**挤掉了**唯一差异化的投入：学习体验是全系统最薄的一块。

**颠覆点不在「砍掉电商」，在「重新认定承重柱」**：现行思路把承重柱**隐性**设成了「卖包+锁视频」；根本意图要求承重柱是「陪着把作品做完」。

## 五、以根本意图为基础的重构思路（建议的新思路）

**一句话新思路**：幸运小鸭不是「带材料包的钩织视频站」，而是**「钩织完成陪伴」——产品是『一个新手成功做完第一个作品』。材料包是入口，视频是参考资料，承重体验是动手过程中的计数/状态/记忆陪伴。**

这不是换功能，是**换重心**。重心一换，架构的引力中心随之反转：

### 5.1 学习层从「看视频」反转为「做东西时被陪着」

| 维度 | 现行（视频为主） | 重构（陪伴为主） |
|---|---|---|
| 主屏 | 播放器播视频段 | **制作陪伴台**：大计数器（点一下+1针）、当前行针法特写、「你现在应该有 N 针」校验点、上一步/下一步 |
| 视频地位 | 主体 | **降为参考面板**之一（与针目特写照片、图解并列，手忙时收起） |
| 进度模型 | `progress`：哪个视频段看完 / 暂停在几秒 | **`makingState`：钩到第几行、本行第几针、各步实际针数 vs 目标针数、停在哪、上次几点碰过** |
| 内容模型 | `courses`：章→课时→段(videoFileId) | **`project`：一串「行/圈/步骤」，每步带 {针法、目标针数、参考(视频段/特写图/图解)、校验点}** ——钩织作品的天然结构是「行 × 针数 × 针法」，不是「视频章节」 |
| 完成的定义 | 视频段看完 | **作品做完**（最后一步针数对上、收针） |

> 关键洞察：团队**已经为这个反转留过位**——规格 §五-5 把进度记忆做到了 segment 粒度，原话「为『计数陪伴』方向留位」。`progress` 折叠模型就是 `makingState` 的种子，不是从零造。

### 5.2 二维码激活的「目的」改写（机制几乎不动，语义升级）

- 现行：扫码 = 解锁视频 DRM。
- 重构：扫码 = **把这盒实物 kit 绑定到它专属的「带计数的引导制作项目」**（这盒就是这个作品、这套行数针数、这些校验点）。
- 代码改动极小：`activateCourse`/`confirmEnter` 链原样保留，绑的对象从 course 变 project；退货权失效节点逻辑不变。

### 5.3 什么原样存活 / 什么重新指向 / 什么是新建（让颠覆变得负责任）

| 类别 | 内容 |
|---|---|
| **原样存活**（电商脊柱，两种思路通用） | 登录/users、商品、订单、履约(发货/收货)、支付/退款、二维码激活链、控制台骨架、评价、橱窗 |
| **重新指向**（壳在、内容换） | 控制台第④步「课程视频编排」→「**项目/图谱编排**」（行/针数/针法 + 参考素材）；播放页 → **制作陪伴台**；`progress` → `makingState` |
| **新建**（真正的差异化，今天就能盖） | 计数/状态陪伴 UI + 数据模型；针目特写参考库；（远期可选）AI 视觉点数 spike |

> **承重版本是「手动交互式计数陪伴」**（app 替你数针/记进度 + 针目特写），**今天的技术就能盖**。AI 视觉点数是「可行性未知」的登月项，**不押在它身上**——它是锦上添花的未来增强，不是地基。

## 六、与工程重构（cloud-kit / B1–B8）的关系：互补，不冲突

这点对所有者很重要，先讲结论：**产品思路重构不推翻工程重构，反而让工程重构更值。**

- 工程重构在硬化的是**电商脊柱的代码结构**（统一安检 kit、状态机助手、金额分类型、确定性 id）——这套脊柱**在两种产品思路下都存在、都要用**。所以 B1–B8 不白做，它加固的是两种思路共享的地基。
- 产品思路重构**新增**的是一层「制作陪伴」内容/学习层——它**长在被 cloud-kit 硬化过的地基上**，地基越稳，盖这层新柱子越便宜、越安全。
- 所以正确的叠放是：**工程重构（硬化地基，与产品思路正交）→ 在硬地基上盖根本意图要求的新承重柱（制作陪伴）。** 两件事可以并行推进，互不阻塞。

工程重构的红线「功能等价」也不破：B1–B8 保持电商脊柱功能等价；制作陪伴是**地基之上的新建**，不在 B1–B8 的等价承诺范围内，属于另一条产品线。

## 七、风险与边界（我不替所有者拍的板）

1. **要不要换重心，是生意决策，不是技术决策**——属于所有者。本文是建议，不是已执行的转向；没有任何代码/计划被本文改动。
2. **不必停上线**。最稳的路径可能是：**电商脊柱按 v1.0 照常上线（开始收钱 + 验证需求），同时把「制作陪伴」作为 v2 差异化开盖**——先用真实用户验证「计数陪伴」确实是他们要的，再加码。节奏由所有者定。
3. **AI 点数不背刺**：重构的承重版本是手动计数陪伴，可行、可建；AI 视觉是未知数，单独 spike，绝不让地基依赖它。
4. **沉没成本不是负债**：电商脊柱在新思路里继续承重，不存在「白盖了」——只是它从「主角」回到「必要配角」。

## 八、给所有者的决策点（这份文档真正要你回答的）

**幸运小鸭的承重柱，认哪一根？**

- **A · 维持现思路**：承重柱=卖包+锁视频；先把电商课程站完整上线，计数陪伴继续停在探索区。（最快见收入，差异化最弱）
- **B · 扶正承重柱（本文建议）**：根本意图认「陪着做完作品」；电商脊柱照常上线收钱，**同时立项「制作陪伴」作为差异化主线**，学习层从「看视频」逐步反转为「做东西被陪着」。（差异化最硬，多一条产品线的投入）
- **C · 先验证再定**：上线前/上线后用一个最小的「计数陪伴」原型（哪怕只覆盖一个作品）做真实用户验证，用数据决定 A 还是 B。

> 我的建议是 **B 的渐进版 = 先 C 后 B**：不赌、不停摆——电商照常上线，并行做一个最小计数陪伴原型验证根本意图，验证成立就把它扶成主线。
