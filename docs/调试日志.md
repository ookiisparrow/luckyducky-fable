### 2026-06-16 · Q：「全部订单」整页白屏——一条缺 items 的脏订单让列表渲染崩（根因#8 真实数据形状）　✅ 已修
- **现象**（用户报·微信开发者工具）：「全部订单」页白屏；Console 红报 `TypeError: Cannot read property 'reduce' of undefined`（vue runtime-1）。
- **根因**：order-list `qtyOf(o)=o.items.reduce(...)` 与 order 详情 `cfgFromOrder` 的 `o.items.map(...)` **未防御 items 缺失**。库里存在缺 items 的脏订单（手工写库/历史 schema——`createOrder`/`confirmEnter` 两条写路径都不产此形状，故为外部数据），「全部」tab 渲染**所有**订单 → 撞上脏行即整页抛白屏；状态 tab 因过滤掉该行反而看着正常（症状非对称＝定位钥匙）。云函数侧 `getMyOrders`/`pageQuery` 原样返回文档、不投影，脏形状直穿到渲染。
- **修复**：单一收口在 store 入库边界——`store/orders.js` 加 `normalizeOrder`（`Array.isArray(o.items)? o : {...o, items:[]}`），在 load/loadMore/create/fetchById **四个入口**归一，保证 `this.list` 每条订单 items 恒为数组；脏行降级为「共 0 件」、绝不白屏整页。列表/详情都只经 store 取数 → 两处崩点一处治（不散 `||[]` 碎补）。
- **同类隐患（结构性·根因#8）**：列表/详情对「云返回的单条文档」做 `.reduce/.map/.length` 的渲染热点，都得当**不可信形状边界**防御——测试样本永远带 items 就是「样本失真」，把缺字段的失败点漏在测试外。已查同表 `.items` 引用：review/aftersales 早已 `||[]` 守、cart 是本地 state、order-list/order 两处此批补齐。云是不可信形状边界，回灌过 sanitize（同 CLAUDE.md §6 persist 纪律）。
- **是否结构性**：是（信任边界 + 验证样本失真）。守卫：`tests/orders.test.js` 4 条锁——① store 入库归一（load/fetchById 脏单 items 必为数组）；② **渲染访问点直喂裸脏单**（`orderItems`/`orderQty` 不抛、降级 0）。反向自检：去 normalize 或还原裸 `o.items` 即红。
- **状态**：✅ 已修。**2026-06-16 复发＝旧构建**（用户跑 096e560 前 dist，store 归一未进 build）；但暴露 store 单层防御之脆——上版守卫只测了入库不测访问点、`o.items.reduce/.map` 渲染点仍裸。**收紧**：`data/orders.js` 出 `orderItems/orderQty` 安全访问器（单源、与 review/aftersales 的 `||[]` 同约定），order-list/order 改用之 + 访问点直测锁。`npm run check` 全绿 341 测试、build mp-weixin DONE（含修复，用户重导 `dist/build/mp-weixin` 即消除）。残：控制台核查脏订单来源/补 items（数据清理，非代码缺陷）。

### 2026-06-16 · P：压测台真跑全假阴——内存桩按 wx-server-sdk 写、真跑 node-sdk 形状不同（根因#8 桩绿≠真证）　✅ 已修
- **现象**：压测台首跑通认证后，CAS 测试连报假结果——先 `v=undefined·全耗尽`、再 `v=0·全耗尽`，脚本判「❌ CAS 非原子」。看着像真库丢更新，其实是脚本读错了真库返回。
- **根因（三层 SDK 形状错位，全因「桩按云函数 wx-server-sdk 写、真跑用 node-sdk 直连」）**：① `update()` 返回 node-sdk 是 `{updated}`、脚本读 `r.stats.updated`（wx-server-sdk 形状）→ 永远 undefined → 每个 worker 都以为失败、重试耗尽 → 假「丢更新」；② `get().data` node-sdk 是**数组**、脚本按对象读 → `v=undefined`；③ `set/update/add` node-sdk 吃**裸对象**、脚本 `{data:{...}}` 包一层 → 存成嵌套 `{data:{v}}`、顶层无 `v` → `where({v})` 全不命中 → v 卡 0。
- **修复**：`readV`/`updatedCount` 两 helper 兼容两种形状（导出 + `tests/scripts/loadtest.test.js` 锁死，反向自检过）；去掉三处 `{data:}` 包层；附带 `createCollection` 兜底（CloudBase 不自动建集合）+ 缺密钥诊断 + 小批并发灌。改对后真证：CAS `v=20·0丢更新`、capacity `count=2000=gmv=2000·991ms`。
- **同类隐患（结构性·根因#8）**：**「内存桩绿」只证「按桩的假设写对了」，桩的假设若偏离真实 SDK，桩越绿越骗人**——本仓 cloud 测试用桩模 wx-server-sdk，压测台直连却是 node-sdk，两套返回形状不同。债#19 压测台的全部价值正在此：把「桩绿」推到「真库证」才暴露这层。教训：跨 SDK 复用调用代码前，先核两边返回形状（data 数组/对象、updated 顶层/嵌套、写入要不要 {data} 包）。
- **状态**：✅ 已修并真库验证通过（2026-06-16，测试环境 `luckyducky-test`）。真容量副得：环境对突发 DB 有 QPS 限流（200 并发即挡、20 安全）。

### 2026-06-15 · O：物流卡点不进——快递100 插件跳转写法错（plugin-private:// → plugin://别名）　✅ 已修
- **现象**（用户报）：admin 填发货信息后，小程序订单页出现物流信息条，但**点了进不去**快递100 轨迹页。
- **根因**：上一批（express-plugin-wired）写的跳转 URL `plugin-private://wx6885acbedba59c14/pages/result/result?nu=&com=` **基于公开资料推断、未真机验（根因#8）**，写法错——微信插件正确跳法是 **`plugin://<别名>/<导出页>`**（别名＝manifest plugins 的 key `kuaidi100`，导出页 `index`），不是 `plugin-private://APPID/内部页`。且参数应为 `num`（非 `nu`）+ `appName`（小程序名·必填）+ `com`（编码·**选填，不传插件自动识别单号**）。
- **修复**：`plugin://kuaidi100/index?num=<运单号>&appName=<店名>&com=<编码>`；因 com 选填，去掉「未命中编码就只复制」的门槛——**微信端总是开插件**（com 有则带、无则插件自认），打开失败/非微信端才回退复制。守卫 `express-plugin-wired` 同步改判 `plugin://kuaidi100/`（反向自检：改坏 scheme→红→还原）。
- **同类隐患（结构性·根因#8）**：**平台 SDK/插件的调用契约（scheme/页名/参数）必须真机验，不能照搬博客/推断**——「构建过 + 守卫绿」只证「按我以为的写法编进去了」，不证「写法对」。守卫只能锁「用了 plugin:// 别名跳转」，**跳转真能打开仍须真机**。
- **状态**：✅ **已修并验证通过**（2026-06-15 用户工具/真机验：插件能打开 + 物流信息正确）。残留「加载慢」＝快递100 插件实时查件的第三方延迟、咱不可控（传 com 省自识别一步略快）。首次跳转弹「安装插件」须点安装。

### 2026-06-14 · M：Tab 页「返回直接退出小程序」防误退　✅ 已解（分支 v0.9.1.1返回重构版）
- **现象**：自绘 TabBar 用 reLaunch 切页，Tab 页是栈底页 → 返回手势/键 = 直接退出小程序，用户常误退。
- **两个死路（真机实测排除）**：① `enableAlertBeforeUnload` 只拦「页面间 navigateBack」，**拦不住栈底 Tab 页退出**（真机无弹窗）；② `page-container` 能拦返回手势/键，但 **show=true 武装时锁死页面级滚动**（overlay=false 也改不了，真机页面不能滑）。
- **活路（PoC 验证成立）**：页面内容套 `<scroll-view>` → 滚动发生在 scroll-view 内部、page-container 锁的是「页面级滚动」锁不到 → 既能武装拦返回、内容又能滚。双击退出：第一次弹 toast「再按一次退出」+解除武装，2s 内再返回放行、2s 后重武装（`composables/useExitGuard.js`）。
- **迁移代价**：三 Tab 页内容进 scroll-view；首页页面级滚动 API 全迁——`onPageScroll`→`@scroll`、`pageScrollTo`→受控 `:scroll-top`（先同步当前位、下一拍再设目标，绕过「相同值不触发」）、购买滚到商品卡改 FeatureProducts emit 目标给父页滚。
- **同类隐患（结构性）**：mp-weixin **无 onBackPress、拦不住栈底页退出**；要拦返回手势只能 page-container，而它锁页面滚动 → **凡需拦返回的可滚页面，内容都得放 scroll-view**（新 Tab 页照此）。
- **状态**：✅ 三页真机过（能滚 + 双击退出 + 首页购买/回顶/探索滚动正常）。在分支，待商量合 main。

### H · 退货权失效只认 'paid' 订单——真实流程里几乎永远不触发（启发式审核 2026-06-11）　✅ 已修
- **现象/判定**(2026-06-11 启发式逻辑链审核发现,非线上报障):业务规则「确认开始观看 → 失去该商品退货权」。`confirmEnter` 里失退货权的反查写死 `orders.where({ _openid, status: 'paid' })`——只在订单**仍是「待发货」**时才翻 `items[].refundable=false`。
- **根因**:触发条件与真实顺序错位。真实流程是 下单(paid) → 商家发货(shipped) → 顾客收货(done) → 扫包装内码 → `confirmEnter` 确认进课。**等顾客确认进课时订单早已不是 paid**(是 shipped/done),`where({status:'paid'})` 扫不到 → 退货权失效**静默不执行**。顾客已进课,系统却仍标 `refundable:true`。与「确认收货不能跳过发货状态」同类:一个状态条件设窄,让本应发生的链路断掉。
- **同类隐患/启发式**:状态机里凡是「A 动作触发 B 副作用」,要核对 **B 反查的状态集是否覆盖 A 可能发生的所有时点**——退货权失效(B)能在订单任意状态(A 时点)发生,却被钉死在单一状态。`shipOrder`/`confirmReceive` 的状态门是「正向流转」(只允许特定前态),而本处是「副作用反查」,反查不该用正向流转的窄条件。
- **影响/紧迫度**:当前**潜在**——真实退款功能尚未实现(售后仍样例),没有代码读 `refundable` 去放行/拒绝退款,故暂无线上后果;但逻辑是错的,**P4 接真实退款时若不先修,会按错误标记放行退款**(已进课仍可退)。规格注明 `refundable` 为「展示性标记,裁决以退货验收扫码为准」→ 有兜底,故非 P1;但 `status:'paid'` 的窄化是疏漏而非设计意图。
- **修法**:✅ 已修(2026-06-11,本批)——订单状态过滤改 `status: _.in(['paid','shipped','done'])`,收货后(shipped/done)进课也能失退货权;补 `tests/cloud/confirmEnter.test.js` 用例锁定(shipped 订单进课失退货权),反向自检:还原成 'paid' 该用例变红。
- **是否结构性**:业务逻辑层,非工具/环境。原 P2,与 I 同批修复并补测试关账。**待 P4 真实退款上线时复验**(届时 refundable 才被真正读取)。
