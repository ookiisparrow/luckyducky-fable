## C（fable）· 2026-07-05 · recordAnomaly 真机不落库（部署 inspect 逼出·根因#8·已修+守卫·结案）
**现象**：巡检机 inspect 部署到生产、`tcb fn invoke` 回 `red:["stuck-order"]`，但 `anomalies` 集合恒空（`inspectRuns` 却有 2 条）。**根因（#8 构建过≠真机能用）**：`recordAnomaly` 用 `doc().update()`-first，真机 wx-server-sdk 对缺失文档/未建集合 update **抛错**（非返 updated:0），被外层 fail-soft `catch` 静默吞、再到不了 add fallback——`inspectRuns` 有 add+createCollection 兜底故正常，`anomalies` 无、恒空；**整个 bug 收集器半边（含 notifyAlert 桥）真机不落库**。**为何测试没抓**：桩 `doc().update()` 缺失文档返 updated:0 + `add` 自动建集合，双重掩过真机（桩≠真机经典）。**修**：`recordAnomaly` 改 add-first + createCollection 兜底（不依赖 update-on-missing）。**守卫（治本·让桩诚实）**：桩加 `control.markUncreated` opt-in 逼真机「未建集合 add/update 抛错」+ anomaly.test「集合尚未建也能落库」用例（旧写法必红）。**验证**：重部署 inspect + 重调 → anomalies 落库成功（stuck-order 带 9 单 samples）；check 1386→1387 全绿·反向自检禁 createCollection→用例红→还原绿。**结案**（详 `重构日志.md 2026-07-05 部署记账`）。

## A（fable）· 2026-07-04 立 / 2026-07-05 结案 · 真单联测退款触发失败 REFUND_TRIGGER_FAIL（真根因订正·已修+守卫）

- **现象**：M2 钱链真单联测——新线 app 下单支付成功（订单 `2026070413019480`·2 元·微信交易号 `4200003134202607040164483193` 在案），申请售后成功，admin 审批同意 → `REFUND_TRIGGER_FAIL`、售后单条件回滚回 `applied`（卡"待审核"）。
- **真根因（2026-07-05 订正·取到微信原文报错定案）**：退款经 `callFlow` → 微信，`out_refund_no` = 售后单 `_id` = `orderId__lineId`，而 `lineId = productId__spec`。**新线**订单行带 `lineId`，此单 SKU spec = `sku测试文案`（**含中文**），故 `_id` = `2026070413019480__pmq8yfwvp__sku测试文案`。微信退款单号仅允许 `^[0-9a-zA-Z_|*@-]{1,64}$`，中文必拒——`59cd5b4` 修好 callFlow 可观测性后，adminApiV2 日志 7/05 拿到微信原文：`PARAM_ERROR ... "/body/out_refund_no" ... 必须匹配 ^[0-9a-zA-Z_|*@-]{1,64}$`。**是代码把含中文的内部主键当外部平台单号用，不是平台漂移。**
- **推翻旧结论**：原判「退款工作流 `kbgzl_n8gojr3a` 平台漂移（被删/失效）」，依据「今日入参与 6/14 成功单逐字同构」——**该前提错了**：6/12–6/25 成功单是**旧线**建的（订单行**无 lineId**→ `_id=订单__productId`·纯 ASCII 退成功），7/04 是新线建的（`_id` 带中文 spec）。out_refund_no 形状 ASCII→含中文，正是新旧线差异。**工作流一直是好的**（它正确转发、微信才拒）——曾差点据错误结论去动控制台资产 #3，是可观测性缺陷（`.catch(()=>null)` 吞真错·#14）把排障引偏。
- **修复（本批·根因#12 接缝契约）**：kit `refundNoFor(_id)` 把内部 _id 规整为微信合规单号（已合规 ASCII 老单原样·含非 ASCII/超长者派生「订单号前缀 + cyrb53 短哈希」）；applyRefund/overrideRefund 建单落 `outRefundNo` 字段、approveRefund 取库存或现派生并落库、发微信改用它；refundCallback 按 `outRefundNo` 反查（回退 `.doc(_id)` 兼容纯 ASCII 老单）。守卫 `rw-money3-golden`/`rw-admin3-golden`（roots 加 #12·断言 out_refund_no 恒配平台正则 + 回调按字段反查命中·反向自检 identity 篡改 4 红）。
- **待办③ 收尾（✅ 已闭环·2026-07-05）**：部署 `adminApiV2` + `refundCallback` 到生产（`app` 未部署·approveRefund 对无 outRefundNo 的老单现场派生，故非必需·M5 同名替换时再上；adminApiV2 冒烟 `/adminv2` ping 200·HTTP 未坏）→ 用户后台重点该单"同意"→ **派生合规单号 `2026070413019480pmq8yfwvpsku-ho0hyd0n1g` 被微信受理**（不再 PARAM_ERROR）→ ~12s 后 refundCallback 按 outRefundNo 反查认领 → `status:refunded`、`refundTransactionId:4200003134202607040164483193`、`refundedAt` 在案。**2 元原路退回·端到端生产验证通过·无需新单**。
- **归因**：根因#12（平台接缝——非"平台单方漂移"，是**接缝未校验传出参数合规**：内部标识符直接当外部平台单号用，违反平台字符集/长度契约。#12 根治由"接缝收口单点"扩为"收口 + 传出合规"）+ **根因#14**（`.catch(()=>null)` 吞微信真错致排障引偏·扳机已由 `59cd5b4` 补 `[LD_ALERT]`）。结构性：是。
