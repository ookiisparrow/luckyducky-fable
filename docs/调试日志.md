## 2026-06-07 · 购物车列表显示空白行　✅ 已系统性处理

- **现象**:小程序里购物车商品行只剩复选框 +「¥」,图/名/价/数量步进器全空;合计仍显示 ¥1。
- **根因**:持久化的**历史脏数据**——早期测试存进 storage 的残缺条目(缺 name/price 等),
  冷启动被 `persist.js` 原样 `$patch` 回灌。精确产生时刻不可考(开发期跨版本/热重载),
  但**真正病根是持久化回灌零校验**:`$patch(JSON.parse(saved))` 只防 JSON 解析失败,
  不防"解析成功但内容残缺"。
- **同类隐患(启发式)**:
  - 三个 store 都持久化、都无回灌校验(cart / address / user)。
  - **数组类状态才是脆弱点**:`$patch` 对对象深合并(user.profile 有完整 default,相对安全),
    对数组整体替换(cart.items / address.list 元素无兜底,直接撑乱)。
  - **address.list 同病、且更危险**:残缺地址会污染结算页默认地址、影响下单。
  - 无 schema 版本/迁移 → 每次改数据结构都在埋雷,会反复发生。
- **是否结构性**:**是**。不是购物车一个页面的 bug,是持久化层 + 数据边界的结构性缺陷。
- **状态**:✅ 已处理。数据契约校验(`persist.sanitize` + `keepValid`,cart/address/user 各配契约,
  清洗时 `logger.warn` 留痕)+ 全局错误捕获 + 回归测试(`tests/persist.test.js`、`validate.test.js`
  锁定);残留的临时脏数据用微信工具「清缓存」清除。

## 2026-06-07 · 购物车契约漏校验 qty（外部审核 P2）　✅ 已修

- **现象**：`sanitizeCart` 只校验 id/name/price、漏了 qty。脏数据里 `{id,name,price}` 无 qty 会通过清洗 → `count` getter 累加成 NaN；连回归测试都把"无 qty"当成了有效。
- **根因**：契约只校验了**部分字段**，而非"模板 / getter 实际用到的所有字段"。是上一条质量体系刚落地时自埋的缺口——**契约要覆盖完整字段，测试数据要贴近真实结构**。
- **同类隐患（启发式）**：其它 store 契约同样要审视"是否覆盖所有被读取的字段"；selected 这类布尔字段也应归一化（脏值 1/0/字符串）。
- **是否结构性**：**是**（契约完整性原则）。
- **状态**：✅ 已修。`sanitizeCart` 加 qty 正数校验 + qty 取整 + selected 布尔归一化；提成导出函数供测试直接验；`persist.test.js` 加"无 qty 被丢弃 + 归一化"断言锁定。

## 2026-06-07 · 云函数 login 漏写 _openid（P0 登录联调）　✅ 已修
- **现象**:`login` 云函数 `callFunction:ok` 但返回 `user` 为空;新建的 `users` 记录没 `_openid`,后续按 `_openid` 查不回。
- **根因**:**云函数是管理端,`add` 不会自动写 `_openid`**(只有小程序端 add 才自动写)。没显式写 → 踩了 `微信云开发文档梳理.md` 第 4 节明确警告的坑。
- **同类隐患（启发式）**:所有"云函数 add 业务数据 + 之后按 `_openid` 查/鉴权"的地方都要**显式写 `_openid`**(将来 activations/orders 同理);返回数据别依赖"add 后立即查"(有一致性延迟),直接构造返回。
- **是否结构性**:**是**(云开发管理端 `_openid` 规则,贯穿所有云函数写库)。
- **状态**:✅ 已修。`login` 显式 `_openid: OPENID` + 直接返回构造的 user。教训:接云开发的云函数写库**必显式 _openid**。
