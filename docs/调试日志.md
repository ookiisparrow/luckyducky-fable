### 2026-06-07 · 购物车列表显示空白行　✅ 已系统性处理
- **现象**:storage 历史脏数据被 `persist.js` 原样 `$patch` 回灌,购物车商品行字段全空。
- **教训**:**持久化回灌必须过数据契约校验**(`$patch(JSON.parse(...))` 只防解析失败,不防内容残缺;数组类状态最脆弱)。
- **状态**:✅ 已处理——`persist.sanitize` + `keepValid` 契约(cart/address/user)+ 全局错误捕获 + `tests/persist.test.js` 锁定。

## 待修（已记录,排期单独处理）

### G · 控制台传视频失败「直传失败 HTTP 403」（直传请求方法与凭证不符）　✅ 已修待复验
- **现象**(2026-06-11 用户验收单 T 备料时发现):第④步上传 mp4 报「视频上传失败：直传失败 HTTP 403」,直接终止。
- **根因**:`getUploadMetadata`(manager-node 4.x 旧版接口)签发的凭证按微信云开发 HTTP API 约定**只认 POST 表单上传**(签名串含请求方法),前端却用 `PUT + Authorization 头`直传 → 云存储判签名不符,403 拒绝。直传路径此前从未被真实走通(此前 E2E 只覆盖到取凭证成功)。
- **修法**:① 直传改 POST 表单(`key`/`Signature`/`x-cos-security-token`/`x-cos-meta-fileid` + file,key 从 fileId 截取);② **补结构性缺口**——原回落只覆盖「取凭证失败」,现在上传本身失败也自动回落分片通道(慢但可达,用户无感),失败原因留浏览器控制台。
- **同类隐患/教训**:① 凭证类接口要按签发方约定的**完整请求形状**使用(方法/字段/顺序都参与签名,不能只对上 token);② 「双保险」要覆盖主路径的**全部失败点**,只兜第一步等于没兜;③ E2E 过了「拿到凭证」不等于过了「用凭证」(与 F 的"最小样本只验逻辑"同根)。
- **状态**:✅ 已修(fix/video-upload-403)并重发线上控制台;待用户第④步重传视频复验,过了即关账。

### F · 控制台传图失败 UPLOAD_FAIL（HTTP 服务请求体上限）　✅ 已修待复验
- **现象**(2026-06-11 用户验收发现):控制台上传产品图报 UPLOAD_FAIL;此前 AI 的 E2E 用 1×1 测试图通过。
- **根因**:HTTP 访问服务**请求体上限约 100KB**(实测 50KB 过/200KB 拒,`EXCEED_MAX_PAYLOAD_SIZE`);真实照片压缩后 base64 200–700KB 必超。且网关错误响应无 `ok` 字段 → 前端只能报笼统 UPLOAD_FAIL。
- **修法**:**分片上传**——前端按 ~80K 字符切片 → `uploadChunk` 暂存 `uploadChunks` 集合 → `uploadFinish` 拼回转存云存储(顺手清 1 小时孤儿分片);小图保留单发;适配层把网关错误转可读文案。
- **同类隐患/教训**:① **E2E 样本要贴近真实数据尺寸**——最小样本只验逻辑不验边界(本案教训);② 同通道响应体也有上限,listDrafts 商品多后需分页;③ 网关层错误与业务错误形状不同,适配层必须兜底。
- **状态**:✅ 已修已部署(2026-06-11);80K 分片过网关+路由验证 ✓(口令为用户私有,拼装链路由用户在控制台 UI 复验,过了即关账)。

### C · 订单无实体数据贯通(外部审核 P1)　✅ 已修(2026-06-10 订单上云,PR #25)

- **教训**:贯通型问题若已临近对应的云改造里程碑,直接在云改造里一并解决,避免先做一版前端临时方案再推翻。
