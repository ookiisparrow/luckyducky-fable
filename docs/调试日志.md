## S（fable）· 2026-07-24 · 开发者工具 Nightly 回归：checkout 空草稿自动返回死锁模拟器导航道（冒烟三连败·根因#10 新变种）

- **现象**：开发者工具接入批首跑 `npm run smoke:mp`，三次在同一位置死（home/detail/cart PASS 后 `timeout waiting for automator response`·exit 2），下一页恰是 checkout。
- **复现与隔离**（一次性诊断脚本 E1-E7b·新鲜实例逐项对照）：① 同序列纯导航（E1）与全开关行为复刻（E5·console 订阅+evaluate+waitFor）均全通 → 排除僵实例/第4次序数/cart 毒化/武装 page-container；② 单发 `callWxMethod('navigateTo', checkout)` 必挂（E6），此后 reLaunch 永挂而 systemInfo/pageStack/evaluate 秒活——**半僵态**（导航道瘫·信息道活）；③ 对照 navigateTo→about 正常（E7a）→ 非 API 整体坏；④ evaluate 发后即忘进 checkout（E7b）：checkout 786ms 才上栈（>600ms），自动返回被吞（2s 后栈顶仍 checkout）且导航道照瘫 → 元凶不是 RPC 形态，是 **checkout.ts backTimer 600ms 空草稿 `wx.navigateBack()` 打在前向转场未完成的窗口内**，新版（2.02.2607232·前日自动更新）导航状态机扛不住该竞态（≤07-19 时代旧 build 同场景 21/21 绿）。
- **根因**：平台回归 × 我方竞态窗口叠加——backTimer 在 `refresh()`（onShow 即达）起表，不等转场落地；冷池转场可迟至 ~800ms。
- **修复（工具层·同批落地）**：mp-smoke.cjs 三适配（checkout 末位 / evaluate 发后即忘+轮询两段式断言 / connect 导航道双探活自愈冷启）——2026-07-24 全量 23 页 PASS 实证（热池下 checkout 两段式断言原语义照过·断言零放水）。
- **同类隐患**：navback-timer 同族页（review/profile-edit/feedback）冒烟以 reLaunch 单层栈进入，navigateBack fail-fast 不触发转场、不撞该竞态（容忍清单本就豁免）；真机低端机同竞态理论可达（back 被吞≠死锁·退化为「空结算页多停留」），随 checkout 红区提案一并考虑。
- **结构性**：是（根因#10 新变种：半僵态使旧「systemInfo 探活」失明）——守卫收紧至导航道双探活；工具自动更新＝回归随时再来，冒烟层由此有自愈路径。
- **状态**：✅ **已全部闭环（同日晚·赵老板拍板同意红区提案后落地）**：checkout.ts backTimer 起表点改 onShow→onReady 后（见 U 案同批·600ms 语义不变），`npm run check` 全绿 + 亲自逐行核红区 diff + 真实开发者工具实例复验**23/23 PASS**（含 checkout 两段式断言原语义照过）。

## U（fable）· 2026-07-24 晚 · 开发者工具同日二次自动更新：Tool.getInfo 响应形状改变，miniprogram-automator 版本门崩溃（真发现·独立于 S 案）

- **触发**：落地 S 案 checkout.ts 修复后复验，工具在会话中途**又一次**静默自动更新（2.02.2607232→2.02.2607242，同一晚），`npm run smoke:mp` 换了个新报错：`TypeError: Cannot read properties of undefined (reading 'split') at licia/cmpVersion.js`。
- **隔离**：栈追到 `miniprogram-automator` 内部 `MiniProgram.checkVersion()`——`connect()`/`launch()` 两条路径都无条件调它，逻辑是 `t=(await this.send('Tool.getInfo')).SDKVersion` 再拿 `t` 跟 `'2.7.3'` 比大小。写一次性诊断脚本绕开 automator 的 `connect()`/`launch()` 包装、直接用 `Launcher.connectTool()` 拿原始连接手调 `Tool.getInfo`，实测新版返回 `{"version":"2.02.2607242"}`——**`SDKVersion` 字段已被拿掉，改成 `version`**，`t` 因此是 `undefined`，`undefined.split('.')` 崩。
- **根因**：devtools 自动化协议的 `Tool.getInfo` 响应形状在这次 Nightly 更新里变了，`miniprogram-automator`（第三方库，非我方代码）尚未跟进。这条版本门本身只是起步兼容性检查，不影响后续任何真实自动化能力（page/evaluate/pageStack 等一概不读 SDKVersion）。
- **修**：不碰 `node_modules`（第三方包改了会被下次 `npm install` 冲掉，且掩盖真相）——在 `scripts/mp-smoke.cjs` 自己的代码里，`require('miniprogram-automator')` 之后立即把 `MiniProgram.prototype.checkVersion` 猴补成 no-op，头注写清「不是我们模拟器版本太老，是协议形状变了」。
- **反向自检**：还原补丁后重跑，`npm run smoke:mp` 精确复现同一条 `cmpVersion.js` 崩溃 → 补丁存在时正常放行 → 有效性实锤。
- **重要更正（诚实记账）**：补丁后一度又炸出「全部 RPC（evaluate/pageStack/systemInfo/reLaunch）均在约 10s 处超时」，一度怀疑是同批 Nightly 引入的第二个更深回归——**排查后证伪，是我方自己的操作坑，不是产品 bug**：`rewrite/mp/project.config.json` 被 `.gitignore` 排除、只存在于主目录（`git ls-files` 确认零追踪），而 git worktree 只带受版本控制的文件——本次诊断切到 worktree（`.claude/worktrees/devtools-smoke-fix`）后跑冒烟，worktree 里的 `rewrite/mp` 压根没有这份配置，devtools 编译不出 `pages/home/home.js`，模拟器直接「启动失败」（截屏亲证：模拟器画面显示「模拟器启动失败 Error: app.json: 未找到 ["pages"][0] 对应的 pages/home/home.js 文件」），没有活的小程序实例应答任何自动化调用，于是所有 RPC 统一卡在同一个「等自动化响应」超时上，表象酷似协议级故障。补 `cp` 一份 `project.config.json` 进 worktree 后立刻 23/23 全绿——证实与 Nightly 本身无关。
- **同类隐患/记账**：往后任何在 git worktree 里跑 `npm run smoke:mp`（或手动连 devtools）的场合，须先把主目录 `rewrite/mp/project.config.json`（gitignored 本地文件）拷进 worktree 同路径——已在 `scripts/mp-smoke.cjs` 头注补一句前置提醒；不新增机器守卫（一次性环境搭建步骤，机器管不了「这个人有没有先 cp」，硬立守卫属过度工程）。
- **结构性**：checkVersion 崩溃是（第三方库跟不上平台协议变化，猴补属可复用应对模式）；RPC 全超时不是结构性 bug，是本次诊断的操作失误，如实记账防止未来会话误读成「工具又坏了」去重复排查。
- **状态**：✅ 已闭环。工具本身在该次更新后仍稳定可用（23/23 实证），S 案与本案均已解决。
