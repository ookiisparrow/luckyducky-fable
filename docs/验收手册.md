# 验收手册（非技术版）

> 2026-06-10 立。给不读代码的验收人用：**验收看现象，不看实现**。
> 你只需要回答「它的行为对不对」——这恰好只有懂业务的人能判断。
> 约定：每个里程碑收尾，AI 交付一份「验收单」（操作步骤 + 预期现象），照单操作，全过才算完成。

## 一、三个观察窗口（都不需要懂代码）

1. **云开发控制台**（后端的"货架"）
   - 入口：微信开发者工具顶部「云开发」按钮 → 控制台。
   - 「数据库」= 所有业务数据的表格视图。点开集合（如 `products`）就能看到每条记录的字段和值。
   - 「云函数」→ 某个函数 → 「日志」= 每次操作的流水记录（什么时候被调用、成功还是失败）。
   - 「统计」= 调用量、存储量等运行指标。
2. **小程序本体**（行为是最终事实）
   - 开发者工具模拟器或手机真机（工具栏「预览」扫码）。
   - 最硬的验证手法：**后台改一下 → 前台变没变 → 改回来**（见下方验收单示例）。
3. **项目文档**（中文，无代码）
   - `重构日志.md`：每天做了什么。`现状与路线.md`：到哪了、下一步。`关键决策记录.md`：为什么这样做。
   - 文档说的和你看到的现象应该一致；不一致就是问题，直接质问。

## 二、黄金路径（核心业务流程清单）

每个大里程碑收尾后，这条线要能从头走通（现阶段到 4 为止，后续逐步延长）：

1. 打开小程序 → 首页商品正常显示（价格、标签）。
2. 进商品详情 → 加购 → 购物车 → 结算 → 支付成功页。
3. 视频教程：欢迎页 → 课程目录（章节/课时/试看标记）→ 播放页（分段进度条、段末自动停、求助面板）。
4. 「我」页：地址簿增删改、资料页、订单列表（角标与订单状态对得上）。
5. （P3 后）扫激活码 → 课程欢迎页 → 确认 → 我的课程出现该课。
6. （P4 后）真实支付 → 订单状态流转 → 条目级退货。

## 四、不懂代码时怎么判断质量（四个代理信号）

1. **验收单全过**：行为对，比代码"看起来对"重要得多。
2. **测试数量只增不减**：`重构日志` 里每批改动会写"test N ✓"（当前 27）。数字变小要问为什么。
3. **文档与现实一致**：`现状与路线.md` 说完成的，验收单能验证；说没完成的，小程序里确实没有。
4. **问题有账本**：出过的问题在 `调试日志.md` 里有现象/根因/教训，而不是"修好了就忘"。

## 五、何时该打回 / 该追问

- 验收单有一条不过 → 打回，不接受"代码层面是对的"。
- 演示只在模拟器跑、没在真机验证过涉及真机的项 → 追问。
- 一个改动说不清"用户会看到什么变化" → 追问（说不清现象的改动要警惕）。
- 文档没更新就说完成 → 打回（按 CLAUDE.md 记录约定，重构日志 + 现状与路线是收尾的一部分）。

## 六、验收方法论（根因#8：「E2E 过了」≠「真的能用」）

原项目两次踩坑：1×1 测试图过、真实 200KB 照片必拒（日志 F）；凭证只验「拿到」没验「用」、直传从未真走通（日志 G）。验证的置信度 = 样本与真实的距离。新功能验收前对照三条：

1. **真实尺寸样本，不用玩具数据**：E2E/单测用接近线上的样本——图片走真实 200KB（不是 1×1）、列表走真实条数、金额走真实分。测试基建已备真实尺寸 fixture（见 `tests/cloud/adminMisc.test.js` 真实 200KB 图分片重组用例，断言落库字节 = 原图大小、未截断）。
2. **凭证/签名按完整请求形状走完整链路**：不止「拿到凭证」，要「用凭证」把签发约定的完整请求走到最后一步；「双保险」若只兜第一步 = 没兜。
3. **兜底路径覆盖主路径全部失败点并被测试逼出**：每个会失败的点都要有兜底，且兜底被一条用例主动触发，不留「没人走过」的暗路径。

收尾自查：① 本批样本够真实吗？② 凭证类真「用」过吗？③ 兜底被测试逼出过吗？三问有「否」即补，不放过「最小样本过了就算过」。

## 七、部署后 smoke（钱链/交易云函数上生产必走·根因#8 续）

云函数改了钱链/交易逻辑后，「部署成功」「测试全绿」都 ≠ 生产真能用——**改了哪条路径就真机走哪条，重点验新逻辑没误伤正常流程**。模板（2026-06-25 部署 P0 防超卖 / P1 停售拒单后首用·全绿）：

0. **先确认部署同步**（技术侧）：`.deploy-manifest.json` 各函数 hash 与当前产物比对 N/N 一致（或 `node scripts/preflight.mjs` 钱链漂移绿）——同步了再上真机，否则验的是旧版。
1. 🟢 **只读先行**：打开有真实数据的页（评价汇总 / 后台订单详情抽屉），看数字/列表/抽屉正常不崩、不空不 NaN。
2. 🔴 **正常路径不被误伤**（最关键）：用真实在售商品走主流程，确认**没被新加的校验拦住**——改了下单校验就验正常单能建、改了支付回调就验正常支付翻单。新加的拒绝逻辑最易误伤生产里字段不规整的老数据。
3. 🔴 **真钱小额**：交易类必走一笔真实小额（下单 → 付款 → 转已付 → 可退款），不用 mock；顺带验退款链还活着。
4. **诚实边界**：并发竞态（超卖 / 重复回调 / 抢占）真机几乎复现不出，靠 `*.test.js` 反向自检锁——真机只验正常路径，竞态属「逻辑验过·生产未实测」，知道这缺口即可，别假装真机覆盖了它。

任一 🔴 红 → **先停后续上线动作**回查（多为生产数据字段或回调配置），别带病发布。

**为什么靠人**：smoke 是真机看现象，「生产真能用」机器（守卫/测试）证不出（根因#8 本质，同第六节）——此模板是方法论、不配守卫。

## 八、对账差错处置（上线后·看仪表盘现象处理·根因#8 续）

后台看板「交易异常」(`getTxAlerts`·与财务对账 S16 同源) 亮起任一即照此处置。**机制字段/动作在 `系统事实.md` 登记（feeMismatch/refundMismatch/clearFeeMismatch/txAlerts/refund_required 行）·此处不复述**——本节只给运营「看到什么现象 → 做什么动作」。每次处置在 `调试日志.md` 记一条（日期/异常类型/订单或售后号/金额/根因/动作）。

四类异常（前两类多为生产老数据字段不规整或回调配置，后两类是涉钱必须当天处理）：

1. 🔴 **feeMismatch（订单·实付≠应付）**：现象＝看板 feeMismatch 列有单、且该单**发货被 `FEE_MISMATCH_HOLD` 挡住**。处置＝① 商户后台核实实收金额 vs 订单应付 → ② 判因（客户端改价 / 税费算法）→ ③ 真有差额→商户后台补退差额；核实无误→后台「金额异常解除」(`clearFeeMismatch`·写 `feeMismatchClearedAt`) 清标、发货恢复。**没核实前别清标硬发货。**
2. 🔴 **refundMismatch（售后·退款回调单号/金额与售后单不符·已拒置已退款）**：现象＝看板 refundMismatch 列。处置＝商户后台核实该退款真实状态+金额，售后单人工核对。**注意：代码暂无 `clearRefundMismatch` 解除动作（只 feeMismatch 有）**——只能人工核实后记账，并把该笔交后端按债处理（见 `待办与债`）。
3. **stuckRefunds（售后·approved 超 1h 未到成功回调）**：现象＝看板 stuckRefunds 列。处置＝商户后台查该退款状态——已成功则等回调/人工核对；失败则重新发起退款工作流。
4. 🔴 **refund_required（订单·已付但售罄无法履约·`[LD_ALERT] PAID_BUT_OOS` 告警）**：现象＝前端/admin「待退款」chip + 告警。处置＝商户后台凭 `transactionId` **手动全额退款** → 记账（系统内一等退款闭环尚未做·见 `待办与债` 5b）。

**为什么靠人**：对账处置靠看真实流水 + 商户后台动作，「钱真退对了 / 真该清标」机器证不出（根因#8 本质·同第六、七节）——本节是方法论、不配守卫。
