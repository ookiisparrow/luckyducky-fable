# 验收手册（非技术版）

> 2026-06-10 立。给不读代码的验收人用：**验收看现象，不看实现**。
> 你只需要回答「它的行为对不对」——这恰好只有懂业务的人能判断。
> 约定：每个里程碑收尾，AI 交付一份「验收单」（操作步骤 + 预期现象），照单操作，全过才算完成。

## 一、三个观察窗口（都不需要懂代码）

1. **云开发控制台**（后端的"货架"）
   - 入口：微信开发者工具顶部「云开发」按钮 → 控制台。
   - 「数据库」= 所有业务数据的表格视图。点开集合（如 `products`）就能看到每条记录的字段和值。
   - 「云函数」→ 某个函数 → 「日志」= 每次操作的流水记录（什么时候被调用、成功还是失败）。
   - 「统计」= 调用量、存储量等运行指标。
2. **小程序本体**（行为是最终事实）
   - 开发者工具模拟器或手机真机（工具栏「预览」扫码）。
   - 最硬的验证手法：**后台改一下 → 前台变没变 → 改回来**（见下方验收单示例）。
3. **项目文档**（中文，无代码）
   - `工作日志.md`：每天做了什么。`项目现状.md`：到哪了、下一步。`关键决策记录.md`：为什么这样做。
   - 文档说的和你看到的现象应该一致；不一致就是问题，直接质问。

## 二、黄金路径（核心业务流程清单）

每个大里程碑收尾后，这条线要能从头走通（现阶段到 4 为止，后续逐步延长）：

1. 打开小程序 → 首页商品正常显示（价格、标签）。
2. 进商品详情 → 加购 → 购物车 → 结算 → 支付成功页。
3. 视频教程：欢迎页 → 课程目录（章节/课时/试看标记）→ 播放页（分段进度条、段末自动停、求助面板）。
4. 「我」页：地址簿增删改、资料页。
5. （P3 后）扫激活码 → 课程欢迎页 → 确认 → 我的课程出现该课。
6. （P4 后）真实支付 → 订单状态流转 → 条目级退货。

## 三、已完成部分的验收单（现在就可以做）

**A. 商品数据在云端**
1. 开发者工具 → 云开发 → 数据库 → `products`：应有 5 条记录，价格与小程序首页一致。
2. （可选实验）把某商品 `price` 改个数 → 小程序杀掉重开 → 首页该商品价格变了 → **改回来**。

**B. 课程数据在云端（三层结构）**
1. 数据库 → `courses`：应有 1 条 `course-duck`，点开能看到 chapters（4 章）→ lessons（每章 4 课）→ segments（每课 5 段）。
2. （可选实验）把课程 `title` 改一个字 → 小程序重开进「课程目录」→ 标题变了 → **改回来**。

**C. 静默登录在工作**
1. 数据库 → `users`：应有你真机登录产生的记录（一串 o 开头的 openid）。

**D. 云函数在被调用**
1. 云开发 → 云函数 → `getCourses` → 日志：每次打开课程目录页应新增一条成功记录。

## 四、不懂代码时怎么判断质量（四个代理信号）

1. **验收单全过**：行为对，比代码"看起来对"重要得多。
2. **测试数量只增不减**：`工作日志` 里每批改动会写"test N ✓"（当前 27）。数字变小要问为什么。
3. **文档与现实一致**：`项目现状.md` 说完成的，验收单能验证；说没完成的，小程序里确实没有。
4. **问题有账本**：出过的问题在 `调试日志.md` 里有现象/根因/教训，而不是"修好了就忘"。

## 五、何时该打回 / 该追问

- 验收单有一条不过 → 打回，不接受"代码层面是对的"。
- 演示只在模拟器跑、没在真机验证过涉及真机的项 → 追问。
- 一个改动说不清"用户会看到什么变化" → 追问（说不清现象的改动要警惕）。
- 文档没更新就说完成 → 打回（按 CLAUDE.md 记录约定，工作日志 + 项目现状是收尾的一部分）。
