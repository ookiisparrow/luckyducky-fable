# 后台360工作站 · 架构规范（顶层设计单源）

> **这是什么**：后台360工作站(自建客服+360+私域沉淀体系)的**顶层设计思维与架构铁律**，所有施工批次(见 `施工蓝图.md`)必须遵守。**核心思维＝组件化 / 模块化**：系统 = 一组可独立开关、可组件化接入、彼此解耦的**功能板块(模块)**。
> **provenance**：用户 2026-06-30 确定自建后，给施工蓝图追加的架构规范。调研/决策史见 `../archive/客服选型研究.md`。

## 0. 一句话顶层设计

> **后台360工作站 = 一组自包含的「功能板块(模块)」**。每个板块管好自己的数据/逻辑/UI，对外**只暴露一个声明式接口**；板块之间**不直接耦合**，要协作就经**统一接口/注册表**；每个板块**可单独开关**；新板块**组件化接入、核心零改**。

## 1. 五条架构铁律

### 铁律一：模块化为单位（不是散落改动）
- 系统按**功能板块**切分：每板块 = 自己的 cloud actions + admin 组件/页 + 自己的集合(如有)。
- **每个施工批次(B*.*)产出一个自包含模块**，而非横跨多处的散落 patch。模块边界 = 批次边界。

### 铁律二：板块间解耦（禁直接互 import）
- 板块之间**禁止直接 `import` 对方内部**。A 板块要用 B 的能力，只能经**统一接口/注册表**，不得伸手进 B 的实现。
- 沿用项目既有「**平台接缝单点**」思想(根因#12)：耦合点收口到一处、不散落。

### 铁律三：耦合处收束为通用接口（provider 模式）
- 必要的协作**收束为声明式通用接口**。典型＝**360 聚合用 provider 接口**：
  - `getCustomer360` = **orchestrator(编排器)**，本身不认识任何具体板块。
  - 每个 360 板块实现统一接口 **`CustomerPanelProvider { key, label, enabled, order, fetch(openid) → panelData }`**，注册到 **provider registry**。
  - 编排器只遍历 registry 里 `enabled` 的 provider、调 `fetch`、组装结果。**加/删/改一个 360 面板 ⇒ 只动那个 provider，不碰编排器。**
- admin 前端同构：板块 = 组件，经**板块注册表**挂载；页面壳只渲染注册表里的板块，不硬编码板块列表。

### 铁律四：功能板块可单独开关（feature toggle）
- 一个**模块注册表 + feature flag**(config 驱动，如 `config` 集合 `cs.modules.<key>.enabled`)。
- 每板块声明 `enabled`；关掉即**不挂载前端 / 不进 360 聚合 / 不暴露入口**。
- 价值：**灰度上线**(先开只读360、后开节点诊断)、**出问题单独关**(不波及其它板块)。

### 铁律五：新板块组件化接入（开放-封闭）
- 新增板块 = ① 实现接口(provider / admin 组件) ② 注册到注册表 ③ 配 feature flag。
- **核心(编排器 / 页面壳 / 路由)零改动**——对扩展开放、对修改封闭。

## 2. 机器守卫候选（元模式·每条规范配守卫·B0 立）

| 守卫候选 | 焊住的不变量 | 对应铁律 |
|---|---|---|
| `cs-module-registered` | 每个客服板块必在模块注册表声明(防散落/绕过) | 一/五 |
| `cs-no-cross-module-import` | 板块间无直接 import（只经接口/registry） | 二 |
| `cs-panel-via-provider` | 360 面板经 provider 接口聚合（编排器不硬编码板块） | 三 |
| `cs-module-toggleable` | 每板块有 enable flag（可开关） | 四 |

> 守卫在 **B0 模块化地基** 批次先立(红)、再让框架满足(绿)，此后每加一个板块自动受这几条守卫约束（违反即 `npm run check` 红）。

## 3. 与现有架构的衔接（不另起炉灶）

- **复用现有 admin 壳 + adminApi action 注册表**(`adminApi/index.ts` 的 `ACTIONS` 表本就是「注册式接入」雏形)——provider/板块注册表是同一范式的延伸。
- **复用 kit 收口纪律**(T2「cloud 原语只在 kit」)：板块的微信/云原语经 kit，不裸调——这也服务 S3「将来可迁自有实例」。
- **复用 `getCustomer360` 编排器**作为 360 板块的统一汇流点(B1.1 即按 provider 模式建，不要写成大 if-else)。

## 4. 反过度工程边界（防把"模块化"做成摆设）

- 模块化是为**解耦 + 可开关 + 可扩展**，**不是为每个小功能都套一层抽象**。板块粒度 = 一个有意义的功能单元(只读360/节点诊断/知识库/承面C…)，不是把一个函数也包成"模块"。
- provider 接口**只在真有多个同类板块要聚合时**用(360 面板就是)；单一、不会扩展的逻辑**内联具体**即可(CLAUDE §7·Rule of Three)。
- feature flag **只给会独立开关的板块**，不给每个内部函数加开关。
