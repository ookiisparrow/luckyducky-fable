## 2. 技术方案细化

### 2.1 推送线（企业微信应用消息）
- **kit 接缝**：`kit/wecom.ts` 加 `sendAppMessage(token, { agentid, touser: string[], textcard })`——调企微 `message/send`，token 复用 `getCachedKfToken`（同自建应用密钥·app 消息与微信客服共用应用 token）。**单接缝**：全库应用消息只此一处出（守卫 `app-message-single-seam`·类比 `bot-push-single-seam`）。
- **挂钩点**：`kfCallback` 转人工分支 `enqueueSession`（现 upsert csSession pending）后，查 `agentState` status=online 的坐席 → 取其 `wecomUserId`（缺则跳）→ 推 textcard「新会话待接｜顾客摘要｜[点击接待]」，卡片 url = `www.luckyducky.cn/agent/?session=<sessionId>`。
- **fail-soft 铁律**：推送失败**绝不反噬入队**（try/catch 吞·同 `archive` 铁律）——推送挂了不能致转人工丢单。
- **escalate**：`escalateToMerchant`（active→escalated）时推商户超管（其 wecomUserId）。

### 2.2 免登线（企业微信 OAuth）
- **agentApi 新 action**：`loginByWecomCode`——收 OAuth `code` → 企微 `auth/getuserinfo` 换 `userid` → 在 adminConfig 查 `wecomUserId===userid` 且 `disabled!==true` 的坐席账号 → 命中即按其 role/caps 签发 session（同现口令登录出口）。**fail-closed**：无 code / 换不到 userid / 无匹配账号 / 账号停用 → 一律拒、不签发（守卫 `wecom-login-gated`）。
- **前端 `/agent` 登录三态**（`pages/Login.vue` + 入口判定）：① 在企业微信内打开 → 走 OAuth 静默授权（snsapi_base）→ `loginByWecomCode` 免登；② 在微信内打开（备胎壳·M⑧ 占位）→ 提示或 wx.login 分支；③ 其他浏览器 → 保留现有口令登录（超管仍可口令进）。
- **超管侧**：超管也可填自己的 wecomUserId → 企微内免登进 `/agent`（复用同 action，按超管 caps 签发）。

### 2.3 移动端适配（前端·工作量最大）
- `/agent` 桌面三栏 → 手机窄屏：队列/会话/360 三区改**单栏切换或抽屉**（会话为主视图·队列抽屉进出·360 侧滑）。
- 消费**会话直达路由** `?session=<id>`：推送卡片点入直接打开该会话（若未 claim 先走 claim 流）。
- QuickReplies / Customer360Panel / AgentStatusToggle 移动布局校准。
- 纯前端、对 mock 可全程独立开发（`VITE_AGENT_MOCK=1`）。

### 2.4 靠人配置清单（用户·企业微信管理后台 work.weixin.qq.com）
1. 建部门「外包客服」→ 拉外包成员（填手机号邀请·外包装企业微信 App 接受）；
2. 通讯录可见范围限制（外包只见本部门·看不到全公司）；
3. 自建应用 Lucky Ducky「可见范围」= 外包客服 + 内部客服部门；
4. 应用主页 URL = `https://www.luckyducky.cn/agent/`；OAuth 可信域名 = `luckyducky.cn`（已备案·无需新购/新备案）；
5. **微信客服「接待人员」不加外包**（他们只经 /agent 接待·不碰原生接待台·免 95018 互斥）；
6. 超管在 `/admin` 坐席账号里填每个坐席的企微 `wecomUserId`；
7. 坐席日常把自己在 `/agent` 置「在线」（agentState online）才进推送名单。

## 4. 阶段0 地基（master 串行·扇出前必完成）
> 目的：把三车道共同遵的接口点立齐，车道才能各干各的不打架。
1. **坐席账号加 `wecomUserId` 字段**：`agents.ts` create/update 支持写、list 白名单不回敏感——字段位置定死（车道 A 推送取它、车道 B 免登查它）；
2. **`loginByWecomCode` 契约 + 空实现登记**：ACTIONS 注册 + 系统事实接口册 + agentApi.js 网关加导出（I/O 形状定死·车道 B 填实现）；
3. **`sendAppMessage` 签名 + 桩**：kit/wecom.ts 定签名（车道 A 填实现·车道 A 也可整个自己做，地基只需保证签名不变）；
4. **会话直达路由约定**：`/agent/?session=<id>`（车道 A 推送写它、车道 C 前端消费它）；
5. 建三条守卫的**空壳**（roots + reverseTest 占位·`guard-provenance` 过）——车道各自补断言到红→绿。

## 8. 验收（真机·根因#8）+ 剩项
**真机验收单**：
- 推送：测试顾客发消息→点「找人工」→值班坐席手机企业微信收到「新会话待接」应用消息→点卡片→免登直达该会话→认领回复→顾客收到。
- 免登：企业微信内打开应用→OAuth 静默→不输口令进 `/agent`→身份正确（超管见全队列·外包只见自己 claim）。
- 移动：手机窄屏能看队列/认领/回复/看 360/升级/结束。
- 隔离：外包账号在企业微信只见 Lucky Ducky 一个应用·看不到通讯录全员·碰不到原生接待台。

**剩项（不并本批）**：① 富媒体（顾客发图坐席看图/回图·拉+传素材）；② M⑧ 微信小程序备胎（外包拒装企微才做）；③ 承面 A 内部 VIP（企微侧栏·M③·独立线）。
