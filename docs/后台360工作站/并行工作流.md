# 后台360工作站 · 多 CC 并行工作流（执行机制·非板块内容）

> **这是什么**：用多个 Claude Code 会话并行建客服板块的**协作机制**（谁做 / 怎么隔离 / 怎么自动中继 / 怎么整合）。板块「做什么」见 `施工蓝图.md`、板块清单/状态见 `README.md`——**本文只讲怎么并行执行、不重抄板块内容**（遵单源）。
> **四拍板（2026-06-30·用户）**：① 地基串行先行（B0 框架+信任边界不能并行）② master 唯一上 main（并行 CC 不各自直推）③ 一 CC 一组相关板块 ④ **中继自动化·人不当中继**（人转达状态会漂·该机器化）。
> **provenance**：吸收 2026-06-16 多 CC manifest 事故（共享可变面并发写）+ 2026-06-18 回单 CC（重型派单/游标/探针被否决）+ 2026-06-24 取并集整合经验。**本次轻量之因**：客服后台360工作站是模块化设计（自包含板块·禁互 import·经 registry/provider 协作）→ **切分边界现成（板块清单）·中继靠 git 信号不靠人·不重建被否决的派单/游标**。

## 0. 一句话

模块化让**板块 = 车道**：master 先**串行**筑地基(B0 框架 + 信任边界)定义所有人遵的框架/守卫 → 再按「一组相关板块」开 2-3 个 CC 各占一棵 worktree **端到端**做一组 → 车道 CC 打 **git 就绪 tag** → **master 用 `/loop` 自动轮询 tag、自动整合上 main**（人不传递）。**冲突热点(5 单源登记面)串行化、独立板块码并行化。**

## 4. 自动中继机制（核心·人不当中继·靠 git 信号 + master loop）

**媒介 = git**：worktree 共享同一 `.git`，是 CC 间唯一可靠的共享通道。信号走 **git tag**（最轻·原生·跨 worktree 天然可见·不污染分支内容）。

**① 车道 CC → master（就绪信号）**
车道一批做完·`npm run check` 全绿·commit+push `feat/cs-<lane>` 后：
```
git tag cs/<lane>/ready-<seq>   &&   git push origin --tags
```
= 「我这批好了、待整合」。`<seq>` = 该车道第几批（每批 +1）。

**② master 协调 loop（自动轮询·不靠人传递）**
master 地基做完后跑 `/loop`，每周期（ScheduleWakeup ~20–30min·git 状态变化慢、不必频繁）：
```
git fetch --all --tags --prune
for 每个就绪 tag cs/<lane>/ready-<n> 且无 cs/<lane>/merged-<n>:
    git merge feat/cs-<lane>            # 整合·冲突取并集
    npm run check                       # 全局闸
    若全绿 → 记账 + push main + git tag cs/<lane>/merged-<n> + push tags
    若红/冲突解不了 → git tag cs/<lane>/blocked-<n> + 写因 + 通知用户（不硬解·不默默卡）
无就绪信号 → 继续 sleep
```

**③ master → 车道 CC（反向信号）**
车道 CC 开下一批前 `git fetch --tags`：
- 见 `cs/<lane>/merged-<n>` → 上批已进 main → `git rebase origin/main` 继续下一批。
- 见 `cs/<lane>/blocked-<n>` → 读因 → 修了重交（重打 ready）。

**fail-safe（诚实标局限）**：① master loop 须挂着（master 终端 + loop 运行）——会话关了中继停，但这是「基础设施在线」非「靠人记」，启动一次即可；② 自动整合遇取并集解不了的冲突 → master **打 blocked + 报警转人工**，绝不硬解/默默卡；③ ready tag 靠车道 CC 守约定打——写进工单(§6 第4步) + `/refactor-batch` 收尾，是机制的一部分。

## 5. 整合纪律（吸收 06-16 事故·恒守）

- **master = 唯一 git owner 上 main**：并行 CC 只在 `feat/cs-*` commit、**不推 main**；整合由 master loop 自动做。
- **git 铁律**：push 前 `pull --rebase`·绝不 `--force`·有未提交绝不 `reset --hard`/`checkout .`·改完即提交·整合冲突**取并集**。
- **5 单源登记面只 master 串行改**：`kit/collections.ts`(集合) · `adminApi` ACTIONS + `系统事实.md`(action) · `check-structure.mjs`(守卫) · `README.md` 进度表 · provider registry。并行 CC **报**（在 ready tag 附带 commit 说明或信号文件里列「需登记 X 集合/Y action/Z 守卫」）、master 整合时统一登记或取并集。
- **禁 `git add -A` / `commit -a`**：各 CC 显式路径 add 自己车道文件（防扫到别人半成品）。

## 6. CC 工单（每个并行 CC 的冷启动入口·自包含）

每个并行 CC 是独立会话、互不可见、冷启动。领一个车道即能自包含开干：
1. **读** `docs/后台360工作站/`(本文 + `施工蓝图.md` + `架构规范.md` + `README.md` 板块清单) → 进入状态。
2. **守** 架构规范五铁律 + §1.5 信任边界 + CLAUDE 既有 92 守卫 + 走 `/refactor-batch`(先立守卫红→改绿→反向自检)。
3. **产出** 一组板块的 cloud action(s) + 集合 + admin/miniapp UI + **该组守卫 + 测试 + 记账**·`npm run check` 全绿。
4. **发就绪信号**（不交人）：push `feat/cs-<lane>` → `git tag cs/<lane>/ready-<seq>` + push tags → 完事（master loop 会自动整合）。
5. **登记面** 不自己改 5 登记面、在 ready 信号里列清要登记什么、master 整合时办。
6. **下一批前** `git fetch --tags` 看 merged/blocked → rebase 续 或 修。

## 7. 你的角色（一次性启动·非中继·之后旁观）

- **一次性启动**：master 铺好 worktree 后，你 `cd <worktree> && claude` 各开一个 CC，一句话交代车道(「你是车道 A·先读 `docs/后台360工作站/并行工作流.md`」)；并在 master 终端跑一次 `/loop` 启动协调循环。
- **之后不参与传递**：「车道好了→整合」全走 git tag + master loop·自动。你只在 master `blocked` 报警时介入裁决。
- **节奏**：你决定开几个车道、何时进下一波。

> **进阶选项（更彻底免人·待你定）**：若想连「开终端」都免，master 可改用 **background Agent（worktree 隔离）** 自己 spawn 各车道、完成自动回传整合——全程零人工启动。代价：那是 agent fan-out 模型（非独立 CC 会话·冷启动重 derive 上下文·按量计费），且按惯例 spawn 需你明确授权。当前主方案＝独立 CC + 本文自动中继；要不要升级到 agent 自驱动，你说。
