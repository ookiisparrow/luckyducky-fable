# 自动化验证系统——整个重构的可靠地基

> 2026-06-13 立，用户拍板：四条主张一起扶正，且**用程序自动化的方式扶正**——整个重构流程建立在一个可靠的自动验证 + 自动 hook 系统上。
> 本文 = 这套系统的设计与契约。它先于一切改造存在（批次 B1 就是建它本身），之后每一批改造都骑在它上面。

## 一、核心原则：每条主张 → 一条不变量 → 一个永久守卫

重构不靠「这批小心点」，靠**机器**。任何一条架构主张，落地三件套：

1. **不变量**（一句话能判真假的架构事实，如「没有任何运行时代码引 `data/`」）。
2. **守卫**（机器把关：`check-structure` 规则 / 反向测试 / TS 类型——三选一或叠加）。
3. **判据**：主张「做完」= 守卫存在且全绿；主张「不回退」= 守卫永久咬住，回退即红。

> 这是设计宪章第三原则「让错误无处藏身」推到尽头：架构决策从「写在文档里靠人记」变成「编进闸里靠机器守」。靠人记的，迟早漂移（病根 5/11 的教训）；靠机器守的，不会。

## 二、五层自动验证（一次编辑到一次切换，层层有网）

| 层 | 触发时机 | 工具 | 守什么 |
|---|---|---|---|
| ① 编辑 hook | 每次 Edit/Write 当场 | PostToolUse → `check-conventions --hook` + `check-structure --hook` | 单文件样式 + 结构违例即时反馈（违例 exit 2 回灌 Claude 立改）|
| ② 提交闸 | 每次 `git commit` | pre-commit → `npm run check` | 全套红灯拦提交 |
| ③ CI | 每次 push / PR | `.github/workflows/ci.yml` → `npm run check` | 云端绿勾/红叉 |
| ④ 反向自检 | 每批收尾 | 临时篡改不变量 → 守卫必红 → 还原 | 证明守卫真在咬（不是摆设）|
| ⑤ 类型系统 | 每次 typecheck | `tsc -b`（shared/cloud）| 非法状态编译期不可表达（状态联合类型 / `Fen` 金额）|

**三道闸单一定义（①②③ 不漂移）**：全部以 `npm run check` 为唯一定义——
`check = check:conventions && check:structure && typecheck && lint && test`。
pre-commit 与 CI 都只调 `npm run check`，不各列命令；这一点本身由 `check-structure` 的
**gate-single-source** 不变量守护：谁把三闸改散了，check 立红。**闸自己守着闸。**

## 三、check-structure 引擎（结构不变量的载体）

`scripts/check-structure.mjs`，与 `check-conventions` 分工：conventions 管单文件样式 / 多端写法，structure 管**跨文件 / 仓级架构不变量**。两类规则：

- **repoChecks**（仓级断言，跑一次）：扫 package.json / 配置 / 目录形状。
- **fileRules**（逐文件，支持 `--hook` 即时反馈）：B2 起按主张追加；刻意例外行内加 `structure-ok`。

**B1 已就位的 3 条仓级不变量**（都已绿，且反向自检证明会咬）：

| id | 不变量 | 守的根因/铁律 |
|---|---|---|
| `stub-only-sdk` | 任何包对 `wx-server-sdk` 的依赖必须 `file:` 内存桩 | 测试地基（引真 sdk 测试会触网/触真云，170 回归网失效）|
| `gate-single-source` | pre-commit 与 CI 都只调 `npm run check` | 三道闸不漂移 |
| `deploy-deny-all` | guard-deploy 对一切 tcb 部署/发布 deny | 样板房禁部署铁律 |

## 四、四条主张 → 各自的守卫（随批次长出）

| 主张 | 落地批 | 守卫（机器不变量）|
|---|---|---|
| **T1 砍多端** | B3 | fileRule：api 层禁 `#ifdef MP-WEIXIN` 回退分支、禁 `localXxx` 镜像实现；删完即守卫永拦其重生 |
| **T2 域分组函数** | B2/B4 | fileRule：禁 kit 外裸 `cloud.init()`/`getWXContext()`；repoCheck：部署单元数 ≤ 阈值（28→个位数后钉住）|
| **T3 云唯一真相** | B3 | fileRule：运行时代码（api/store/cloud）禁 import `data/`（仅 seed/构建可引）；repoCheck：种子生成物 diff 空 |
| **T4 按链内聚** | B6 | repoCheck：依赖方向（pages→store/api→kit/shared，无逆向、无循环）|
| 钱/权限闸（账本 1/2/3/4）| B2/B4 | 反向测试：删闸/改宽流转表/并发双调用 → 对应用例必红；TS：写错状态名、浮点金额 → 编译失败 |

> 纪律：**先立守卫（红），再改造到绿**（意图先行）。守卫缺失的主张不算做完；守卫不会咬的主张（反向自检不变红）不算可靠。

## 五、为什么这让重构「可靠」

- 单人 + 非技术所有者，靠纪律守不住几十个架构不变量——机器能。
- 每条主张做完就「焊死」，后续任何批次（或将来的功能开发）想破坏它，三道闸当场红——**重构成果不会在后续被悄悄侵蚀**。
- 切换上线时（B9），「与生产功能等价 + 全部不变量绿」是机器可证的，不是人拍胸脯。
