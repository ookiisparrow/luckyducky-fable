# 微信云开发文档梳理（项目要点版）

整理日期：2026-06-07；2026-06-10 精简为项目要点版（删去官方 API 教程复述：数据库增删改查样例、存储 API 样例、HTTP 云函数、云调用细节），完整版见 git 历史，API 细节查文末官方链接。

## 1. 云开发是什么 / 适合什么

面向微信生态应用的后端服务：云函数 + 云数据库（JSON 文档型）+ 云存储 + 云调用 + CMS。免自建服务器，天然拿到微信登录态（openid）。

不适合纯前端直连云资源的场景：订单、支付、余额、权限、审核等敏感业务——这些必须走云函数。

## 2. 初始化要点

- 建议 **test / release 双环境**，各自独立数据库 / 存储 / 云函数，测试数据和生产数据不混。
- 小程序端启动时初始化，**必须显式传 `env`**，不要依赖默认环境：

```js
wx.cloud.init({ env: '环境ID', traceUser: true })
```

- `wx.cloud.init` 多次调用只有第一次生效；小程序端的 `env` 不会自动影响云函数内部。
- 云函数里单独初始化，用 `DYNAMIC_CURRENT_ENV`（数据库 / 存储 / 调其他函数默认跟随当前环境）：

```js
const cloud = require('wx-server-sdk')
cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })
```

## 3. 数据库关键规则

概念对应：collection = 表，record/doc = 行，field = 列。每条记录有 `_id`；`_openid` 标识创建者。

**`_openid` 管理端规则（本项目踩过的坑，见调试日志）**：

- 只有**小程序端** `add` 才自动写 `_openid`。
- **云函数和控制台属于管理端，`add` 不会自动写 `_openid`** —— 云函数写业务数据后还要按 `_openid` 查 / 鉴权的，必须显式写入（取 `cloud.getWXContext().OPENID`）。
- 不要假设所有数据都有 `_openid`。

批量更新 / 删除放云函数里做，并结合权限校验。

## 4. 安全规则（数据库 + 存储 + 云函数）

核心原则：

- 前端直连数据库只用于**非敏感数据**；用户只能读写自己有权限的数据。
- 敏感写操作（订单、支付、积分、审核、激活）**全走云函数**，服务端校验。
- **不信任前端传入的 `openid`、角色、金额、状态**；云函数里用 `cloud.getWXContext()` 拿可信 `OPENID`。
- 云函数属于管理端、权限高——安全规则放行 ≠ 业务安全，函数内部仍要校验"这条数据是不是你的"。

常见设计：

| 场景 | 推荐方式 |
| --- | --- |
| 用户只能读写自己的记录 | 规则中比对 `auth.openid` 和记录归属字段 |
| 公共内容所有人可读，管理员可写 | 前端只读，写操作走云函数 |
| 订单、支付、积分、审核 | 不开放前端写权限，全部走云函数 |

云函数默认安全规则（不允许未登录调用），可按函数单独配置：

```json
{ "*": { "invoke": "auth != null" } }
```

存储要点：上传后拿 `fileID` 存数据库；临时链接（`getTempFileURL`）有有效期，只用于展示 / 下载，不当永久地址存。存储安全规则只限制客户端，发布前检查：是否误设公开写入、用户是否只能访问自己的私有文件、管理员写入是否走云函数。

## 5. 云函数要点

- `project.config.json` 配 `cloudfunctionRoot: "cloudfunctions/"`；每个函数独立目录 + `package.json`。
- 基础模板：`cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })` + `exports.main = async (event, context) => {}`，函数内 `cloud.getWXContext()` 取 `OPENID / APPID`（`UNIONID` 需满足开放平台条件）。
- 小程序端调用：`wx.cloud.callFunction({ name, data })`（本项目收口在 `utils/cloud.js` 的 `callCloud`）。
- 云调用（免 access_token 调微信开放接口）：函数目录 `config.json` 声明 `permissions.openapi`，用 `cloud.openapi.xxx` 调；权限配置有缓存延迟。细节见官方链接。
- 需要标准 HTTP 路由 / 对外 Web API 时另有「HTTP 云函数」形态（管理后台可能用到），见官方链接。

## 6. 微信支付接入路径

| 路径 | 适用场景 |
| --- | --- |
| 云调用 | 快速接入支付接口，代码量较少 |
| 工作流 | 订单、退款、通知、对账等复杂流程编排 |
| 微信支付管理系统 | 订单查询、退款、运营后台管理 |

建议：简单付款先看云调用；涉及退款 / 状态流转 / 通知重试优先工作流；下单、验签、状态变更等敏感逻辑必须在服务端。需要商户号 + 小程序支付资质。

## 8. 常见坑

1. 不传 `env`，导致请求打到第一个创建的环境。
2. 小程序端初始化了环境，但云函数里没有初始化环境。
3. 把前端传入的 `openid` 当作可信身份。
4. 前端直接写订单金额、支付状态、审核状态。
5. 数据库基础权限配置过宽，导致越权读写。
6. 存储规则设成公开写入。
7. 临时文件 URL 当作永久 URL 保存。
8. 云调用权限改了马上测试，忽略缓存延迟。
9. 云函数依赖没有安装或没有随函数上传。
10. 测试环境和生产环境共用同一套数据。
11. （本项目实踩）云函数 `add` 不自动写 `_openid`，后续按 `_openid` 查不回 —— 管理端写库必须显式写。

## 9. 建议分层

| 类型 | 放哪里 |
| --- | --- |
| 公开展示数据读取 | 小程序端直连数据库，配安全规则 |
| 用户个人数据写入 | 小程序端可写，但必须配安全规则 |
| 订单、支付、积分、审核 | 云函数 |
| 管理员操作 | 云函数或云后台 |
| 文件上传 | 小程序端上传，云函数校验业务归属 |

## 10. tcb CLI 实操（本项目已验证）

> 2026-06-07 验证。用 CloudBase CLI（`tcb`）从本机命令行部署云函数、读写云数据库，不必每次开微信开发者工具 GUI 右键上传。

### 配置

项目根放 `cloudbaserc.json`（`tcb fn deploy` 自动读取 envId / functionRoot / functions）：

```json
{
  "envId": "环境ID",
  "functionRoot": "cloudfunctions",
  "functions": [{ "name": "getProducts", "timeout": 10, "handler": "index.main" }]
}
```

### 常用命令

| 命令 | 用途 |
| --- | --- |
| `tcb env list` | 列环境（验证登录态） |
| `tcb fn deploy --all` | 批量部署配置里所有云函数（首次即创建；自动云端装依赖） |
| `tcb fn deploy <name> --force` | 部署 / 覆盖单个已存在函数 |
| `tcb fn invoke <name>` | 调用云函数（看返回与日志） |
| `tcb db nosql execute --command '<MgoCommands JSON>'` | 直接读写文档库（Mongo 风格 QUERY / UPDATE / INSERT / DELETE） |

### 注意

- `tcb fn invoke` **没有 openid 上下文**：依赖 `cloud.getWXContext().OPENID` 的函数（如 login）CLI 调不出用户态，需小程序端调；不依赖 openid 的（如 getProducts / seedProducts）可 CLI 直接调，适合做数据验证。带 openid 的端到端验证可走 miniprogram-automator 在工具里 `evaluate` 调 `wx.cloud.callFunction`（样板：`scripts/p3-verify.cjs`）。
- 部署不带 `--force` 时若函数已存在可能报错；更新已有函数用 `--force`。
- **`--force` 更新不会自动在云端装依赖**（新建函数才会）→ 已在 `cloudbaserc.json` 每个函数配 `installDependency: true`；如 invoke 报 `Cannot find module 'wx-server-sdk'` 即此坑（2026-06-10 踩过）。
- **tcb login 过期重登**：浏览器授权页**选「微信公众号」扫码**（小程序管理员微信），不要用腾讯云账号密码——微信云开发环境挂在微信侧账号体系下（2026-06-10 验证）。
- 直接改 live 库（`db nosql execute` 写操作）属敏感操作，按环境权限策略可能需要显式授权。
