## 2. 云开发适合解决什么问题

微信云开发是面向小程序、小游戏、公众号网页等微信生态应用的后端服务。它的核心价值是让开发者不必自行搭建传统服务器，就能使用数据库、文件存储、云函数、云调用、静态网站、CMS、工作流和支付相关能力。

典型适用场景：

- 小程序需要快速上线，后端复杂度不高。
- 需要直接使用微信登录态、openid、云调用、订阅消息、内容安全等微信能力。
- 需要数据库、文件上传、云函数、管理后台，但不想维护服务器。
- 需要一套后端环境同时支持小程序、公众号网页、Web 等多端。

不适合完全依赖前端直连云资源的场景：

- 涉及订单、支付、余额、权限、审核等敏感业务。
- 需要复杂事务、跨系统集成或强审计能力。
- 需要自定义网络、安全、容器或服务治理能力，此类场景可考虑云托管或独立后端。

## 3. 小程序接入路线

### 3.1 准备工作

1. 注册微信小程序，获取真实 `AppID`。
2. 安装并登录微信开发者工具。
3. 创建小程序项目，后端服务选择「微信云开发」。
4. 在开发者工具左侧点击「云开发」，开通云开发并创建环境。

建议至少创建两套环境：

| 环境 | 用途 |
| --- | --- |
| `test` | 日常开发、联调、测试 |
| `release` | 生产环境 |

每个环境拥有独立的数据库、存储空间、云函数配置等资源。不要把测试数据和生产数据混在同一个环境。

### 3.2 初始化云开发

小程序端使用云能力前，需要在应用启动时初始化：

```js
wx.cloud.init({
  env: '你的环境ID',
  traceUser: true
})
```

注意点：

- `wx.cloud.init` 通常只调用一次，多次调用只有第一次生效。
- 必须显式传入 `env`，不要依赖默认环境。
- 小程序端的 `env` 只影响小程序端 API，不会自动影响云函数内部 API。

云函数中也需要单独初始化：

```js
const cloud = require('wx-server-sdk')

cloud.init({
  env: cloud.DYNAMIC_CURRENT_ENV
})
```

建议在云函数里使用 `cloud.DYNAMIC_CURRENT_ENV`，这样数据库、存储、调用其他云函数时默认使用当前云函数所在环境。

## 4. 数据库

云开发数据库是 JSON 文档型数据库。概念对应关系：

| 传统关系型数据库 | 云开发文档型数据库 |
| --- | --- |
| database | database |
| table | collection |
| row | record / doc |
| column | field |

每条记录通常包含：

- `_id`：记录唯一标识。
- `_openid`：小程序端创建记录时由系统写入，用于标识创建者。

云函数和控制台属于管理端，创建记录时不一定带 `_openid`。不要假设所有数据都有 `_openid`。

### 4.1 获取数据库和集合

```js
const db = wx.cloud.database()
const todos = db.collection('todos')
```

指定其他环境：

```js
const testDB = wx.cloud.database({
  env: 'test'
})
```

### 4.2 新增数据

```js
await db.collection('todos').add({
  data: {
    description: 'learn cloud database',
    done: false,
    createdAt: new Date()
  }
})
```

### 4.3 查询数据

查询单条记录：

```js
const res = await db.collection('todos').doc('record-id').get()
console.log(res.data)
```

按条件查询：

```js
const res = await db.collection('todos').where({
  done: false
}).get()
```

使用查询指令：

```js
const _ = db.command

const res = await db.collection('todos').where({
  progress: _.gt(30)
}).get()
```

常用查询指令：

| 指令 | 含义 |
| --- | --- |
| `eq` | 等于 |
| `neq` | 不等于 |
| `lt` / `lte` | 小于 / 小于等于 |
| `gt` / `gte` | 大于 / 大于等于 |
| `in` / `nin` | 在数组中 / 不在数组中 |
| `and` / `or` | 与 / 或 |

### 4.4 更新数据

局部更新：

```js
await db.collection('todos').doc('record-id').update({
  data: {
    done: true
  }
})
```

原子自增：

```js
const _ = db.command

await db.collection('todos').doc('record-id').update({
  data: {
    progress: _.inc(10)
  }
})
```

常用更新指令：

| 指令 | 含义 |
| --- | --- |
| `set` | 设置字段 |
| `remove` | 删除字段 |
| `inc` | 原子自增 |
| `mul` | 原子自乘 |
| `push` / `pop` | 数组尾部追加 / 删除 |
| `shift` / `unshift` | 数组头部删除 / 追加 |

### 4.5 删除数据

删除单条记录：

```js
await db.collection('todos').doc('record-id').remove()
```

删除多条记录建议放在云函数中执行，并结合权限校验：

```js
const cloud = require('wx-server-sdk')
cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })

const db = cloud.database()

exports.main = async () => {
  return db.collection('todos').where({
    done: true
  }).remove()
}
```

## 5. 数据库安全规则

数据库安全规则用于控制客户端发起的数据库请求。它比基础四种权限配置更灵活，适合正式项目使用。

核心原则：

- 前端直连数据库只适合非敏感数据。
- 用户只能读写自己有权限的数据。
- 敏感写操作放到云函数中，由服务端校验。
- 不要只依赖前端传入的 `openid`、角色、金额、状态等字段。

常见设计：

| 场景 | 推荐方式 |
| --- | --- |
| 用户只能读写自己的记录 | 规则中比对 `auth.openid` 和记录归属字段 |
| 公共内容所有人可读，管理员可写 | 前端只读，写操作走云函数 |
| 订单、支付、积分、审核 | 不开放前端写权限，全部走云函数 |

## 6. 存储

云存储用于上传、下载、删除文件。上传成功后拿到的是 `fileID`，后续操作应基于 `fileID`，不要把临时链接当作长期地址。

### 6.1 上传文件

```js
const res = await wx.cloud.uploadFile({
  cloudPath: 'uploads/example.png',
  filePath: tempFilePath
})

console.log(res.fileID)
```

### 6.2 下载文件

```js
const res = await wx.cloud.downloadFile({
  fileID: 'cloud://xxx'
})

console.log(res.tempFilePath)
```

### 6.3 换取临时链接

```js
const res = await wx.cloud.getTempFileURL({
  fileList: ['cloud://xxx.png']
})

console.log(res.fileList)
```

临时链接有有效期，不适合长期存储在业务数据中作为永久访问地址。

### 6.4 存储安全规则

存储安全规则只限制客户端请求。云后台和服务端仍然有完整读写权限。

发布前必须检查：

- 是否误设成公开写入。
- 用户是否只能访问自己的私有文件。
- 管理员写入是否通过云函数或后台完成。
- 文件路径是否避免泄露敏感业务含义。

## 7. 云函数

云函数是在云端运行的后端代码，适合放置敏感逻辑和跨资源操作。

### 7.1 项目配置

在 `project.config.json` 中配置云函数目录：

```json
{
  "cloudfunctionRoot": "cloudfunctions/"
}
```

云函数目录结构通常是：

```text
cloudfunctions/
  add/
    index.js
    package.json
```

### 7.2 基础模板

```js
const cloud = require('wx-server-sdk')

cloud.init({
  env: cloud.DYNAMIC_CURRENT_ENV
})

exports.main = async (event, context) => {
  const wxContext = cloud.getWXContext()

  return {
    openid: wxContext.OPENID,
    appid: wxContext.APPID,
    data: event
  }
}
```

### 7.3 小程序端调用云函数

```js
const res = await wx.cloud.callFunction({
  name: 'add',
  data: {
    a: 1,
    b: 2
  }
})

console.log(res.result)
```

### 7.4 获取用户身份

云函数中通过 `cloud.getWXContext()` 获取可信的 `OPENID`、`APPID`，不需要让前端传 `openid`。

```js
const { OPENID, APPID, UNIONID } = cloud.getWXContext()
```

注意：

- `OPENID` 是可信上下文。
- `UNIONID` 只有满足微信开放平台相关条件时才会返回。
- 云函数属于管理端，拥有更高权限；必须自己做业务权限校验。

### 7.5 在云函数中访问数据库和存储

```js
const db = cloud.database()

exports.main = async () => {
  return db.collection('todos').get()
}
```

上传文件：

```js
const fs = require('fs')
const path = require('path')

exports.main = async () => {
  const fileContent = fs.createReadStream(path.join(__dirname, 'demo.jpg'))

  return cloud.uploadFile({
    cloudPath: 'demo.jpg',
    fileContent
  })
}
```

## 8. 云函数安全规则

云函数安全规则用于限制 C 端用户访问云函数。默认规则通常是不允许未登录用户调用：

```json
{
  "*": {
    "invoke": "auth != null"
  }
}
```

可为特定函数单独配置规则。即使规则允许调用，也不代表业务权限已经安全，云函数内部仍需校验用户是否有权执行该操作。

## 9. HTTP 云函数

HTTP 云函数适合 Web 服务场景，支持原生 HTTP、实时通信和多函数路由。它更接近传统 Web Server 开发方式。

特点：

- 可以使用 Node.js 的 Express、Koa、NestJS 等框架。
- 可以使用 Python 的 Flask、Django 等框架。
- 需要 `scf_bootstrap` 作为启动入口。
- 默认监听端口为 `9000`。

最小结构：

```text
nodejs-http-function/
  scf_bootstrap
  package.json
  index.js
  node_modules/
```

`scf_bootstrap` 示例：

```bash
#!/bin/bash
node index.js
```

适合场景：

- 已有 Web 服务需要迁移。
- 需要标准 HTTP 路由。
- 需要对外提供 Web API。

## 10. 云调用

云调用让云函数免维护 `access_token`、签名和证书，直接调用支持云调用的微信开放接口。

接入步骤：

1. 确认目标接口支持云调用。
2. 查看该接口的云调用文档。
3. 在云函数目录的 `config.json` 声明权限。
4. 在云函数中通过 `cloud.openapi` 调用。

`config.json` 示例：

```json
{
  "permissions": {
    "openapi": [
      "subscribeMessage.send"
    ]
  }
}
```

调用示意：

```js
const cloud = require('wx-server-sdk')

cloud.init({
  env: cloud.DYNAMIC_CURRENT_ENV
})

exports.main = async () => {
  return cloud.openapi.subscribeMessage.send({
    touser: 'OPENID',
    page: 'index',
    lang: 'zh_CN',
    data: {}
  })
}
```

注意：

- 云调用权限配置存在缓存，更新后可能需要等待一段时间。
- 每个云函数需要声明自己使用的开放接口。
- 建议 `wx-server-sdk` 保持较新版本。

## 11. 微信支付

云开发提供三种与微信支付相关的接入路径：

| 路径 | 适用场景 |
| --- | --- |
| 云调用 | 快速接入支付接口，代码量较少 |
| 工作流 | 订单、退款、通知、对账等复杂流程编排 |
| 微信支付管理系统 | 订单查询、退款、运营后台管理 |

建议：

- 简单付款可先看云调用方案。
- 涉及退款、订单状态流转、通知重试时优先考虑工作流。
- 生产项目必须把下单、验签、状态变更等敏感逻辑放到服务端。

## 12. 项目落地检查清单

### 初始化

- [ ] 已注册小程序并使用真实 `AppID`。
- [ ] 已开通云开发。
- [ ] 已创建测试环境和生产环境。
- [ ] 小程序端 `wx.cloud.init` 明确传入 `env`。
- [ ] 云函数端 `cloud.init` 使用 `cloud.DYNAMIC_CURRENT_ENV` 或明确环境 ID。

### 数据库

- [ ] 集合命名清晰，避免把所有业务塞进一个集合。
- [ ] 敏感集合不开启前端任意写。
- [ ] 使用安全规则替代粗粒度基础权限。
- [ ] 批量更新/删除放在云函数中。
- [ ] 关键业务字段由服务端写入，不信任前端传值。

### 存储

- [ ] 上传路径按业务和用户分组。
- [ ] 文件 `fileID` 存数据库，临时链接只用于展示或下载。
- [ ] 私有文件配置用户级访问规则。
- [ ] 管理员上传、删除走云函数或后台。

### 云函数

- [ ] `project.config.json` 配置了 `cloudfunctionRoot`。
- [ ] 每个云函数有独立目录和 `package.json`。
- [ ] 使用 `cloud.getWXContext()` 获取用户身份。
- [ ] 不信任前端传入的 `openid`、角色、价格、订单状态。
- [ ] 本地调试通过后再部署。
- [ ] 线上检查日志、监控和错误码。

### 云调用和支付

- [ ] 云函数 `config.json` 声明了所需 `permissions.openapi`。
- [ ] 云调用失败时检查权限缓存和 SDK 版本。
- [ ] 支付相关状态变更只由服务端完成。
- [ ] 订单、退款、通知具备幂等处理。

## 13. 常见坑

1. 不传 `env`，导致请求打到第一个创建的环境。
2. 小程序端初始化了环境，但云函数里没有初始化环境。
3. 把前端传入的 `openid` 当作可信身份。
4. 前端直接写订单金额、支付状态、审核状态。
5. 数据库基础权限配置过宽，导致越权读写。
6. 存储规则设成公开写入。
7. 临时文件 URL 当作永久 URL 保存。
8. 云调用权限改了马上测试，忽略缓存延迟。
9. 云函数依赖没有安装或没有随函数上传。
10. 测试环境和生产环境共用同一套数据。

## 14. 建议的最小后端结构

```text
miniprogram/
  app.js
  pages/

cloudfunctions/
  login/
    index.js
    package.json
  createOrder/
    index.js
    package.json
  updateProfile/
    index.js
    package.json
  adminUpdate/
    index.js
    package.json
```

建议分层：

| 类型 | 放哪里 |
| --- | --- |
| 公开展示数据读取 | 小程序端直连数据库，配安全规则 |
| 用户个人数据写入 | 小程序端可写，但必须配安全规则 |
| 订单、支付、积分、审核 | 云函数 |
| 管理员操作 | 云函数或云后台 |
| 文件上传 | 小程序端上传，云函数校验业务归属 |
