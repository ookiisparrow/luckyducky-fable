## 0. 环境与铁律

- **唯一生产环境**：`cloudbase-d4gcssqbv06865479`（腾讯云开发 tcb）。小程序 **AppID `wxcbd2fb68b81bcfb1`**。
- **本仓 = 生产源**：`rewrite/` 重写线是该环境唯一在线版本（16 云函数 + `/admin` `/agent` 后台 + 根路径内容站）。对该环境的部署/建表/删表**真实生效，不是沙盒**。`packages/` 是字节级冻结的旧线参照（守卫 `oldline-frozen`），**部署不碰它**。
- **部署与 git 解耦**：`git push` 只进代码库，**不改动线上**。上线是**独立的人工授权动作**——每次部署都要有用户明确指令（「部署」），且授权范围仅覆盖用户点名的对象。git 落地 ≠ 已生效；「待部署」的改动前端应诚实显空、不显假数（记账见 `重构日志.md`/`待办与债.md`）。
- **guard-deploy 二次确认闸**：`scripts/guard-deploy.mjs` 是 Claude Code 的 PreToolUse hook。当命令段首真是 `tcb ...` 且为写部署（deploy/publish/delete/create/framework）并命中**敏感函数**（钱/权限/状态：createOrder/pay*/refund*/adminApi*/app/kf*/login/seed*/initDb… 全表见脚本 `SENSITIVE_FNS`），或为批量/认不出的非敏感函数 → 弹 `permissionDecision:'ask'` 要人确认。读类 `tcb`（invoke/log/list/detail）和单个纯读函数（getProducts 等）放行。**提交信息/echo 里提到 "tcb"/"deploy" 字样不会误拦**。人在自己终端跑的命令不经此 hook。
- **前置**：`tcb` CLI 已登录且默认环境为上述 envId（`tcb login` / `tcb env list` 核）。命令里显式带 `-e cloudbase-d4gcssqbv06865479` 最稳。

## ① 部署

### 1.1 云函数（16 个·重写线）

**构建**（产物落 `rewrite/cloud/dist/<函数名>/`，含 `index.js` + `package.json` + 按需 `config.json`）：
```bash
npm run build:rw-cloud      # esbuild 单文件打包 17 个函数；无需先 build shared
```
17 个函数（分组／源码名）：
`app/app` · `adminApi/adminApi` · `callbacks/payCallback` · `callbacks/refundCallback` · `cs/kfCallback` · `cs/kfSend` · `cs/kfMedia` · `timers/closeExpiredOrders` · `timers/cleanupEvents` · `timers/recallScan` · `timers/kfHealthProbe` · `timers/inspect` · `timers/billReconcile` · `ops/genQrcodes` · `ops/seedProducts` · `ops/seedCourses` · `ops/initDb`

**逐函数部署**（一次一个·敏感函数会过 guard-deploy 二次确认）：
```bash
tcb fn deploy <云端函数名> --dir rewrite/cloud/dist/<源码名> --force -e cloudbase-d4gcssqbv06865479
```

> **⚠️ 源码名 ≠ 云端名（唯一的坑，务必记牢）**：源码目录叫 `adminApi`，但**云端函数名是 `adminApiV2`**（M5 定名·HTTP 路径 `/adminv2`·admin/agent 前端已指向）。所以管理端部署恒为：
> ```bash
> tcb fn deploy adminApiV2 --dir rewrite/cloud/dist/adminApi --force -e cloudbase-d4gcssqbv06865479
> ```
> 云端旧名 `adminApi`（无路由绑定的孤立函数）待清理，**不要往它部署**。其余 16 个函数源码名＝云端名，`--dir` 与函数名同字。

> **⚠️ 配置确认弹窗（2026-07-17 tcb 3.5.6-beta 实测）**：**带 `config.json` 的函数**（`app`/`adminApiV2`——产物声明了 openapi 权限）部署时会弹 `? 是否使用默认配置继续部署？(y/N)`。无 config.json 的函数（定时器/多数）不弹、直接成功。**非交互式（脚本/Claude）部署须 `echo y | tcb fn deploy …` 管道回答**（不回答默认 N、部署中断且云端仍旧码——会「假部署」）。**回 y 后必核**：`tcb fn detail <fn> | grep -E '内存|超时'` 确认内存/超时未被默认配置改坏（本仓 app/adminApiV2 均应为 **256MB/15s**）——实测回 y 不改这两项、也不动 openapi 权限，但每次部署后核一眼最稳。**部署真生效验证**（非假绿·根因#8）：`tcb fn detail <fn> | grep <本批新增特征串>` 确认 ≥1（旧码为 0）。

**顺序纪律**（险级从低到高·每组 smoke 过了再下一组，承 M5 runbook Phase 1）：
1. **ops 组**（`genQrcodes`/`seedProducts`/`seedCourses`/`initDb`）——部署成功即可，勿在部署环节主动调用（`genQrcodes` 为持续运营函数：实体卡出新批次时按需人工调用生成激活码，非一次性种子；其余三个为一次性种子/建表）。
2. **定时器组**（`cleanupEvents`→`kfHealthProbe`→`recallScan`→`closeExpiredOrders`→`inspect`→`billReconcile`）——部署后逐个核**触发器仍挂**（见 ⑤·`billReconcile` 净新增须**新挂**触发器），都幂等可 `tcb fn invoke` 手动跑一次看日志无错。
3. **客服组**（`kfSend`→`kfCallback`→`kfMedia`）——`kfMedia` 净新增、只供 `adminApiV2` 跨函数调用无独立触发路径。冒烟见 ④「客服不回」反向：真发一条客服消息走通问答；HTTP 访问服务 GET echostr 仍绑；顾客发图坐席台可看。
4. **钱链最后**（`payCallback`→`refundCallback`）——最高险。冒烟：真单最低额支付一笔看 pending→paid（新函数日志接到回调）；退款按结案姿势真验一笔。
5. 用户端网关 `app` 单列——**部署前先看 1.4 的 stale 陷阱**。

**每类冒烟惯例**：`adminApiV2` 部署后打真端点错口令看 `401 {ok:false,error:BAD_KEY}`（函数活+auth 闸在+没碰真口令）；`ping` action 免鉴权探活应 200。钱链/客服冒烟如上。冒烟命令模板：
```bash
tcb fn detail adminApiV2 -e cloudbase-d4gcssqbv06865479          # 看更新时间/内存确认新码上了
tcb fn invoke inspect -e cloudbase-d4gcssqbv06865479            # 巡检机应回 {green,red,...}
tcb fn log <函数名> -e cloudbase-d4gcssqbv06865479 | grep LD_ALERT   # 部署后盯告警行
```

### 1.2 admin 静态托管（`/admin`）

```bash
npm run build:rw-admin                                          # @ldrw/admin·vite build → rewrite/admin/dist
tcb hosting delete admin --dir --yes -e cloudbase-d4gcssqbv06865479          # 先清空旧文件（--dir --dry-run 先预览命中范围）
tcb hosting deploy rewrite/admin/dist admin -e cloudbase-d4gcssqbv06865479   # 再传新构建
```
**先 delete --dir 再 deploy** 是硬纪律（记忆 `admin2-deploy-clean-overwrite`）：不清空会残留 stale chunk 与新构建混。`--dry-run` 先确认只命中 `/admin/*`、根 `/` 官网与其他路径零覆盖。

**入口哈希核验（真生效非假绿·根因#8）**：部署后带 `Cache-Control:no-cache` 绕 CDN，核线上 `index.html` 引用的入口 chunk 哈希（或整文件 sha256）**== 本地新构建**，且根 `/` 标题仍是官网「小棉鸭 Lucky Ducky…」未被覆盖：
```bash
curl -s -H 'Cache-Control:no-cache' https://<托管域名>/admin/ | grep -o 'index-[A-Za-z0-9_-]*\.js'
# 与 rewrite/admin/dist/index.html 里的入口名逐字比对；再抽验 curl 该 chunk 200 可达
```
> 托管域名 = tcb 静态托管默认域名（`tcb hosting detail` 查）。**TODO**：填入实际域名。

## ② 回滚

**核心心智：没有数据迁移，就没有数据回滚。** 新旧线**共用同一套数据库集合**（零迁移设计·确定性 `_id` + 幂等写：撞 id 即并发方已写）。所以回滚只涉及**代码层**（函数产物 / 静态包），数据层零动作。

- **单函数回滚**：`git checkout <好的commit> -- rewrite/cloud/src/functions/<组>/<fn>` → `npm run build:rw-cloud` → `tcb fn deploy <云端名> --dir rewrite/cloud/dist/<源码名> --force`（同 ①·adminApi 记得部署为 adminApiV2）。回调类回滚期间未 ACK 的微信通知会重推，**幂等设计不怕重**。
- **静态托管回滚**：重新构建目标 commit 的 admin/agent/site → 走 ①的 delete→deploy→哈希核验。
- **整条旧线重建**（灾难级）：`next` 仓（`/Users/sparrow/luckyducky-next`，封存回滚基线，HEAD 附近旧线全在）`npm run build:cloud` 产旧 26 函数包 → 逐函数 `tcb fn deploy`，**分钟级**可把旧线拉回。旧 mp v0.9.4 在微信后台「版本回退」一键回。
- **小程序回滚**：微信后台「版本回退」回上一发布版（人工动作·微信后台）。

**诚实标注的回滚 gaps（别以为有、其实没有）**：
- **重写线部署 manifest 已就位（2026-07-18 批3 迁活线）**：`scripts/deploy-fns.mjs` 现指 `rewrite/cloud/dist`，`--dry-run` 可查「哪些变了」（按产物内容 hash 算，非 git sha）；`preflight.mjs` 的钱链子集检测（批5）同源 hash 算法。当前 `.deploy-manifest.json` 在工作树中不存在——云端实际部署版本未知，是批3故意留的诚实未知态（首次授权部署后重建真清单），不是「机制没做」。
- **云端跑的是哪个 commit 无机器记录**：函数产物里不打 git sha，无法从云端反查对应源码版本。靠 `重构日志.md` 的部署记账（含入口哈希/时间）人工追溯。
- **next 仓回滚通道未演练**：M5 时旧静态包备份因 EISDIR 失败被跳过，回滚基线＝next 仓重建但**未实际跑通一次**。真需要时按上面步骤走、留出验证时间，别假设一次成。

### 3.3 轮换 Token / EncodingAESKey（`wxkf.token` / `wxkf.aesKey`）

## ⑤ 例行巡检

三头可观测（一处配失效另两头兜·别只依赖 webhook）：
1. **`[LD_ALERT]` 日志**：所有动作类失败的结构化单行（`kit/observe.ts`·`MARK='[LD_ALERT]'`）。控制台可按关键字配日志告警。
2. **`anomalies` 集合**：inspect 逮到的红项 + notifyAlert 桥接（`recordAnomaly`·去重指纹·带 samples 单号）。admin `/anomalies` 页四路筛选 + 标记已处理（`resolveAnomaly`）。
3. **企微群机器人**：`settings.alertWebhook` 配了才推（owner 手机实时收·`kit/botpush.ts`）。webhook URL 是凭证不进日志。

自动巡检机：
- **inspect 巡检机**（`timers/inspect`）：定时触发器 `inspectTimer` **每 2 小时**（`0 0 */2 * * * *`）跑 A 基建存活（db-reachable）+ B 业务不变量（money-conserved 钱守恒 / stuck-order 卡单+死信）→ 写 `inspectRuns` 体检报告，红项落 `recordAnomaly`，高危推告警。只读、不改业务集合、有界扫描 cap 500。**手动触发**：admin `/inspect` 页「立即巡检」按钮（action `runInspect`），或 `tcb fn invoke inspect`。回 `{green,red,red:[...]}`。
- **kfHealthProbe 活体探针**（`timers/kfHealthProbe`）：定时器 `kfProbeTimer` **每 30 分**（`0 */30 * * * * *`）探令牌/可信IP 静默故障→推企微告警。静默＝客服链健康。注意：`FORGED` 那类纯自动测不了（需合成真消息），靠「动配置后必发测试消息」兜。
- **billReconcile 每日对账**（`timers/billReconcile`·批B2）：触发器名建议 `billReconcileTimer`、cron 建议 `0 0 11 * * * *`（每日 11 点·微信前一日账单约次日 10 点后可下）。拉前一日账单落 `wxBills` → 比对「微信有款我方无单」（wxOnly）与金额不符 → `anomalies` 高危入账+告警；无账单日/未配凭证 fail-soft skipped。**凭证注意**：走 secureConfig 库（admin `/config-checklist` 填 wxpay 商户序列号+私钥）；若仍靠 env 层，须在**本函数**同配（env 不跨函数）——否则每日静默 skipped（`anomalies` 有 `BILL_RECON_NOT_CONFIGURED` low 面包屑可查）。

## ⑦ 部署后 smoke（钱链/交易云函数上生产必走·根因#8）

云函数改了钱链/交易逻辑后，「部署成功」「测试全绿」都 ≠ 生产真能用——**改了哪条路径就真机走哪条，重点验新逻辑没误伤正常流程**：

0. **先确认部署同步**：`.deploy-manifest.json` 各函数 hash 与当前产物比对一致（或 `node scripts/preflight.mjs` 钱链漂移绿）——同步了再上真机，否则验的是旧版。
1. 🟢 **只读先行**：打开有真实数据的页（评价汇总/后台订单详情抽屉），看数字/列表正常不崩、不空不 NaN。
2. 🔴 **正常路径不被误伤**（最关键）：用真实在售商品走主流程，确认没被新加的校验拦住——改了下单校验就验正常单能建、改了支付回调就验正常支付翻单。新加的拒绝逻辑最易误伤字段不规整的老数据。
3. 🔴 **真钱小额**：交易类必走一笔真实小额（下单→付款→转已付→可退款），不用 mock；顺带验退款链活着。
4. **诚实边界**：并发竞态（超卖/重复回调/抢占）真机复现不出，靠测试反向自检锁——真机只验正常路径，竞态属「逻辑验过·生产未实测」，知道缺口即可。

任一 🔴 红 → **先停后续上线动作**回查（多为生产数据字段或回调配置），别带病发布。（方法论·根因#8「真能用」机器证不出，不配守卫。）

## ⑧ 对账差错处置（上线后·看仪表盘现象处理·根因#8）

后台看板「交易异常」(`getTxAlerts`) 亮起任一即照此处置；机制字段/动作在 `docs/系统事实.md` 登记、此处不复述。每次处置在 `docs/调试日志.md` 记一条（日期/异常类型/单号/金额/根因/动作）：

1. 🔴 **feeMismatch（订单·实付≠应付）**：发货被 `FEE_MISMATCH_HOLD` 挡住。→ 商户后台核实实收 vs 应付 → 判因 → 真有差额补退差额；核实无误后台「金额异常解除」(`clearFeeMismatch`) 清标恢复发货。**没核实前别清标硬发货。**
2. 🔴 **refundMismatch（售后·退款回调单号/金额不符）**：商户后台核实真实状态+金额、人工核对售后单；暂无解除动作（只 feeMismatch 有），核实后记账、按债处理（见 `docs/待办与债.md`）。
3. **stuckRefunds（approved 超 1h 未到成功回调）**：商户后台查退款状态——已成功则等回调/人工核对；失败则重新发起退款工作流。
4. 🔴 **refund_required（已付但售罄无法履约·`[LD_ALERT] PAID_BUT_OOS`）**：商户后台凭 `transactionId` **手动全额退款** → 记账（一等退款闭环未做·见 `docs/待办与债.md`）。
