## A. 云函数安全面

### A-1（P1，✅ 已修）seed / init 云函数无鉴权，任意登录客户端可覆盖生产数据
- **问题**：`seedProducts` / `seedCourses` / `initDb` 的 `exports.main` 无任何身份校验。微信云开发里**任意登录小程序用户**可 `wx.cloud.callFunction({ name: 'seedProducts' })` 直接触发——`seedProducts` 会把 `products` 整表 upsert 回种子默认值（**抹掉控制台发布的真实封面 / SKU / 价格**），`seedCourses` 会把 `course-duck` 覆盖回样例（**抹掉刚上传的视频 fileID**）。虽需知道内部函数名、且可重新发布恢复，但这是「未受信客户端可破坏生产数据」的授权缺口，与项目「敏感写一律设闸」的纪律一致地应归 P1。
- **修法**：套用 `genQrcodes` 既有模式——`if (OPENID) { 须 users.isAdmin===true，否则 ADMIN_ONLY }`；CLI / 控制台 invoke 无 openid 仍放行（运维通道不变）。三个函数同改。
- **验证**：部署后客户端调用应得 `ADMIN_ONLY`；CLI `tcb fn invoke seedProducts` 仍正常（运维不受影响）。

### A-2（正向确认）核心闸门实现正确
- `createOrder`：openid 取 `getWXContext`；价格/金额按云端 `products` 现算，`ADDONS` 表硬编码在云端；SKU 必须在 `products.skus` 命中否则 `UNKNOWN_SKU` 整单拒；地址白名单。✓
- `activateCourse`：一码一用用 `where(status:'unused').update` 条件抢占（updated===1 即赢），并发安全，无需事务。✓
- `confirmEnter`：openid 闸 + `enteredAt` 只写一次（幂等）+ 退货权启发失效逻辑正确（送礼无单场景由验收扫码兜底，设计如此）。✓
- `getMyCourses` / `getMyOrders` / `getMyProgress`：均 openid 限定本人记录。✓
- `genQrcodes`：客户端须 isAdmin；adminApi 服务端互调无 openid 走管理通道，信任链以 adminApi 的 `checkKey` 为前置闸。✓
- `confirmReceive` / `submitReview`：openid + 订单归属 + 状态门（仅 shipped→done / 仅 done 可评 + 商品在单内）。✓
- `updateProfile`：白名单 `{nickname,bio,avatar}` + 截断 + avatar 仅收 `cloud://`。✓
- `login`：可信 openid upsert，显式写 `_openid`。✓

### A-3（P3，→ 技术债）trackEvent 写入放大、无频率限制
- **问题**：任意登录用户每次调用写 1 条 `events` 流水（+ 1 次 `progress` upsert），无频控、无 TTL。恶意刷调用可灌大 `events` 集合（配额 / 费用，非数据正确性）。`meta` 已限 1KB、`type` 已截断，单条有界；P5 看板聚合限 ≤1000 条读取，功能不受影响。
- **建议**：定期清理 / TTL（如保留 90 天）或单用户频率上限；优先级低（需登录 + 仅配额影响）。

## B. adminApi 路由 + 控制台前端

### B-1（正向确认）adminApi 鉴权一致
- 26 个 action：除 `ping` / `login` 外全部在 `checkKey(key,false)` 之后执行；`login` 仅首次（无 `adminConfig.auth`）bootstrap 设口令，之后须匹配 sha256。写路由（saveDraft/publishProduct/saveCourseDraft/publishCourse/saveCard/createBatch/saveShowcase/saveHomeContent/saveSettings）均经闸。✓
- `createBatch` 服务端互调 `genQrcodes`（管理通道）信任链正确——前置 `checkKey` 已挡。✓

### B-2（P2，✅ 已修）控制台防抖保存——快速离开页面丢最后一次编辑
- **问题**：`StepVideos` / `StepCards` 的 `onBeforeUnmount(() => clearTimeout(timer))` **只取消不补存**；防抖窗口（700–800ms）内编辑后立刻切步骤/退出，最后一次改动丢失。`Showcase`（排序/上下架 + 首页文案）原本**没有** `onBeforeUnmount`，路由切走同样丢。有「saving/saved」状态提示作缓解，但易被忽略。
- **修法**：三处 `onBeforeUnmount` 改为「pending 时立即补发一次保存」（保存幂等，多发无副作用）。
- **验证**：编辑后立即切走再回来，改动应已落库（控制台手动复验）。

### B-3（记录性，无需改动）adminApi v1 鉴权边界
- 口令明文在请求 body、`Access-Control-Allow-Origin: *`、口令存浏览器 localStorage——均为**已定 v1 设计**（`设计规格-管理控制台.md` §三：单管理员口令，HTTPS 传输，Web SDK 账号体系留作升级）。HTTPS 保护传输，单口令=店铺钥匙。记录在案，账号体系升级时一并处理。

## C. 数据库 / 云存储权限（代码外的闸，须用户核对）
- 「敏感写一律走云函数」的前提是**客户端不能直接写库**。15 个集合的客户端权限规则 CLI 读不到，须用户在控制台核对——见**验收单 X**。
- 期望：所有集合客户端权限为「仅创建者可读写」或更严（`products`/`courses`/`content` 等公开读集合可「所有人可读、仅管理端可写」）。云存储 products/videos/avatars 同理。
- 风险点：若某集合误设「所有人可读写」，则云函数闸门可被绕过（客户端直接改 `orders.status` / `qrcodes.status` 等）。这是上线前必须人工确认的一道闸。
