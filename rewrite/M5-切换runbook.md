## 三、切换日四阶段（每步带验证点↩回滚点）

### Phase 0 冻结与备份（T-1 日）
1. 定低峰时段；通知外包坐席切换窗口。
2. 备份：下载 `/admin` `/agent` 静态旧包；截图 4 个定时触发器配置;抓旧函数清单快照。
3. next 仓确认可部署（回滚通道热身）。

### Phase 1 后端同名替换（按险级从低到高·每组 smoke 过了才下一组）
1. **ops 组**：部署 `genQrcodes` `seedProducts` `seedCourses` `initDb`。验证=部署成功即可（只部署勿调用）。↩ 单函数从 next 仓重部署（分钟级）。
2. **定时器组**：部署 `cleanupEvents` → `kfHealthProbe` → `recallScan` → `closeExpiredOrders`。验证=控制台逐个核**触发器仍挂**+手动 invoke 一次日志无错（都幂等·safe）。↩ 同上。
3. **客服组**：部署 `kfSend` → `kfCallback`。验证=真发一条客服消息走通问答；HTTP 访问服务 GET echostr 核验仍绑。↩ 同上。
4. **回调组（最后·钱链）**：部署 `payCallback` → `refundCallback`。验证=**真单最低额支付一笔**看 pending→paid（新函数日志接到回调）；退款链按闸 5 结案后的姿势真验一笔。↩ 从 next 仓重部署旧回调（分钟级·期间未 ACK 的通知微信会重推·幂等设计不怕重）。

### Phase 2 提审（审核期 1–7 天·**期间后端零改动**）
1. 开发者工具上传 `rewrite/mp`（同 AppID·版本 v1.0.0·备注写清整体重写）。
2. 体验版真机全链黄金路径**真单**：登录→浏览→下单→支付→激活→播放→售后→客服。
3. 提审。审核期新旧并存安全（见一、论证）。↩ 审核被拒=修了再提，线上不受影响。

### Phase 3 发布与 48h 守望
1. 发布（用分阶段发布：先 25% → 观察数小时 → 全量）。
2. 同日切静态托管：新 admin/agent build 上 `/admin` `/agent`；通知坐席刷新；核三路径可达+登录可用。↩ 回传备份旧包（分钟级）。
3. **48h 守望指标**：LD_ALERT 企微告警零 P0；支付成功率对比前 7 日基线；`first_frame` 首帧埋点无劣化；kfHealthProbe 静默=客服链健康；adminApiV2 审计日志无异常写；订单/售后状态机无卡单。
4. **P0 回滚（发布后）**：微信后台「版本回退」一键回 v0.9.4 → 视根因决定是否回滚函数（next 仓重部署）→ 静态托管回旧包。数据层零动作（共库幂等设计）。

### Phase 4 清退与转正（T+7 观察期后·单独一批）
1. 下线旧 23 用户函数 + 旧 `adminApi` + 旧 `/adminapi` 路径（先停用 48h 再删）。
2. next 仓封存为回滚基线（README 标注）；本仓转正为生产源。
3. docs 改写：CLAUDE 身份段/系统事实/现状与路线；`packages/` 冻结产物处置；`console-assets/` 正册更新函数清单与路径。
4. 转正批完成 = M5 关账。
